## v0.7.0 Requirements (Final Status)

Code review mode: inspect, navigate, and annotate agent-generated diffs before a PR, then return structured feedback to the agent.

### Diff Viewer

- [x] **DIFF-01**: User can view a full branch diff (all changed files combined, vs main) — **satisfied** (Phases 24+25)
- [x] **DIFF-02**: User can expand collapsed context lines within a diff hunk — **satisfied** (Phase 25)
- [x] **DIFF-03**: User can toggle between unified and side-by-side layout — **satisfied** (Phase 25)
- [x] **DIFF-04**: User can navigate directly to any changed file via a file list — **satisfied** (Phase 25)
- [x] **DIFF-05**: User can select which commits to include in the current diff view — **satisfied** (Phase 26)

### Commit Navigation

- [x] **COMMIT-01**: User can view a list of all commits in the current branch — **satisfied** (Phases 24+26)
- [x] **COMMIT-02**: User can click a commit to view its individual diff — **satisfied** (Phases 24+26)
- [x] **COMMIT-03**: User can switch between per-commit view and full branch diff mode — **satisfied** (Phase 26)
- [x] **COMMIT-04**: User can navigate between commits with keyboard (prev/next) — **satisfied** (Phase 26)

### Inline Comments

- [x] **COMMENT-01**: User can add a comment anchored to any diff hunk — **satisfied** (Phase 27)
- [x] **COMMENT-02**: User can add a comment at the whole-file level — **satisfied** (Phase 27)
- [x] **COMMENT-03**: User can edit or delete their own comments — **satisfied** (Phase 27)
- [x] **COMMENT-04**: File list shows a comment count badge per file — **satisfied** (Phase 27)

### Review Submission

- [x] **SUBMIT-01**: User can approve the review when no comments exist — **satisfied** (Phase 28, design pivot: single "Send Review" button)
- [x] **SUBMIT-02**: User can include an optional global instruction when approving — **satisfied** (Phase 28)
- [x] **SUBMIT-03**: User can submit with comments; structured feedback JSON returned to agent — **satisfied** (Phase 28 + Phase 29.1 schema fix)
- [x] **SUBMIT-04**: Submitting "request changes" requires at least one comment — **satisfied** (Phase 28, design pivot: gate enforced by `canSend` logic)

### Integration

- [x] **INTEG-01**: User can invoke code review via a slash command — **satisfied** (Phase 29)
- [x] **INTEG-02**: Agent can trigger code review automatically via a pre-PR hook — **satisfied** (Phase 29)
- [x] **INTEG-03**: `plan-reviewer install` wires up slash command + hook; `uninstall` removes them — **satisfied** (Phase 29)

### Architecture

- [x] **ARCH-01**: Code review viewer replaces the existing (unused) diff tab — prior diff code removed — **satisfied** (Phase 25)

## Notes

- **SUBMIT-01 / SUBMIT-04 design pivot**: The Approve/Request-Changes split was replaced with a single "Send Review" button per user direction during Phase 28 human verification. The `canSend` gate (requires comments OR non-empty message) enforces the intent without binary gating.
- **SUBMIT-03 schema fix**: Phase 28 frontend payload `{message?,comments?}` was incompatible with Phase 24 backend `Decision{behavior:String}`. Phase 29.1 fixed the handler to accept `serde_json::Value` with key-presence dispatch.
- **COMMENT-01..04 / SUBMIT-01..04 checkboxes**: These were unchecked in the live REQUIREMENTS.md file due to tracker drift — the audit confirmed all as satisfied.
