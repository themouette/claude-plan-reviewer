# Phase 07.1: Add `review <file>` Subcommand — Research

**Researched:** 2026-04-10
**Domain:** Rust CLI subcommand dispatch, output format design, integration with existing hook flow
**Confidence:** HIGH

## Summary

Phase 07.1 adds a `review <file>` subcommand to `plan-reviewer`. Unlike the existing hook flows (which read from stdin JSON or `--plan-file`), this subcommand takes a plain markdown file path as a positional argument and renders it in the browser review UI. The key design question is the output format: the phase description says "neutral `{behavior}` decision for use in scripts and agent workflows."

The codebase already has three output formats: Claude Code's nested `hookSpecificOutput`, Gemini's flat `{decision, reason, systemMessage}`, and opencode's flat `{behavior, message?}`. The opencode format (`{behavior: "allow"|"deny", message?: "..."}`) is the cleanest, most generic format — it maps directly to what scripts and agent workflows would parse. "Neutral" here means not Claude-specific, not hook-protocol-specific: just `{behavior}`.

The implementation is straightforward: add a `Review` variant to the existing `Commands` enum, dispatch it in `main()` similarly to how `run_opencode_flow` works (file read + diff extraction + `async_main` + stdout write). No new dependencies are needed. The existing `server.rs`, `async_main`, and `build_opencode_output` functions are all reusable as-is.

**Primary recommendation:** Add `Commands::Review { file: String }` to the clap CLI, implement `run_review_flow(no_browser, port, file)` that mirrors `run_opencode_flow` but uses `build_opencode_output` for the output format, and add integration tests using the existing `assert_cmd` + `ureq` pattern.

### Pattern to Follow: `run_opencode_flow` (from `src/main.rs`)

The opencode flow is the closest analog to the review flow — both read from a file path rather than stdin JSON:

```rust
// Source: src/main.rs — verified by reading the file
fn run_opencode_flow(no_browser: bool, port: u16, plan_file: &str) {
    let plan_md = match std::fs::read_to_string(plan_file) {
        Ok(content) => content,
        Err(e) => {
            eprintln!("Failed to read plan file at {}: {}", plan_file, e);
            std::process::exit(1);
        }
    };
    let cwd = std::env::current_dir()
        .map(|p| p.to_string_lossy().into_owned())
        .unwrap_or_default();
    let diff_content = extract_diff(&cwd);
    let rt = tokio::runtime::Builder::new_current_thread()
        .enable_all()
        .build()
        .unwrap();
    let decision = rt.block_on(async_main(no_browser, port, plan_md, diff_content));
    let output = build_opencode_output(&decision);
    serde_json::to_writer(std::io::stdout(), &output).expect("failed to write hook output");
}
```

### Output Format: "Neutral `{behavior}`"

The opencode output builder already produces the neutral format:

```rust
// Source: src/main.rs — build_opencode_output
fn build_opencode_output(decision: &Decision) -> serde_json::Value {
    match decision.behavior.as_str() {
        "allow" => serde_json::json!({ "behavior": "allow" }),
        _ => {
            let mut obj = serde_json::json!({ "behavior": "deny" });
            if let Some(ref msg) = decision.message {
                obj["message"] = serde_json::Value::String(msg.clone());
            }
            obj
        }
    }
}
```

Output examples:
- Approve: `{"behavior":"allow"}`
- Deny with message: `{"behavior":"deny","message":"Missing tests"}`
- Deny without message: `{"behavior":"deny"}`

This is exactly what "neutral `{behavior}` decision for scripts and agent workflows" means — no hook-specific wrapper, no protocol-specific keys.

### Anti-Patterns to Avoid

- **Reading stdin in the `review` subcommand:** The hook flow reads all of stdin before anything else. The `review` subcommand must NOT read stdin — the file path comes from CLI args. The existing `run_opencode_flow` correctly avoids stdin; the `review` subcommand should follow the same pattern.
- **Designing a new output format:** The neutral `{behavior}` shape already exists as `build_opencode_output`. Introducing a new `build_review_output` function with a different shape would fragment the format landscape unnecessarily.
- **Making `--no-browser` and `--port` sub-command-local:** These flags must remain on the parent `Cli` struct so they compose with `review` the same way they compose with the existing hook flows. Integration tests rely on `--no-browser --port <N>` working uniformly.

## Don't Hand-Roll

| Problem | Don't Build | Use Instead | Why |
|---------|-------------|-------------|-----|
| Output JSON serialization | Custom serializer | `build_opencode_output` + `serde_json::to_writer` | Already tested; correct shape confirmed by unit tests in `main.rs` |
| File reading with error handling | Custom retry/buffered reader | `std::fs::read_to_string` + `eprintln!` + `process::exit(1)` | Established pattern in both `run_opencode_flow` and `extract_plan_content` |
| Browser launch | Custom OS command | `async_main` (already handles `webbrowser::open`) | Server start, browser open, timeout, and decision channel are all wired inside `async_main` |
| Integration test binary spawning | Custom `std::process::Command` wrapper | `assert_cmd::cargo::cargo_bin` | Already established in `install_uninstall.rs` and `server_cycle.rs` |

## Common Pitfalls

### Pitfall 1: Subcommand vs. Flag Conflict
**What goes wrong:** The existing `--plan-file` flag on `Cli` and a new `review <file>` subcommand serve overlapping purposes. If both are present and a user calls `plan-reviewer --plan-file foo.md review bar.md`, behavior is ambiguous.
**Why it happens:** `--plan-file` was added for opencode plugin invocation (programmatic use), and `review` is for human use. They are semantically different entry points.
**How to avoid:** Document in clap `about` text that `--plan-file` is for opencode plugin use only. The `review` subcommand is the human-facing interface. In code: when `Commands::Review` is dispatched, ignore `cli.plan_file` entirely.
**Warning signs:** Integration tests should assert that `plan-reviewer review <file>` ignores any `--plan-file` flag if both appear.

### Pitfall 2: Stdin Still Open During `review`
**What goes wrong:** If the test or calling script has stdin connected and doesn't close it, and if `review` accidentally falls through to `run_hook_flow`, the process will hang on `read_to_string(stdin)`.
**Why it happens:** The dispatch tree in `main()` is currently `Some(subcommand) | None`. Adding `Review` correctly requires it be in the `Some()` arm, not in `None`.
**How to avoid:** Add `Commands::Review` to the `Some()` match arm. Verify with an integration test that calls `plan-reviewer review <file>` with stdin closed (no piped input).
**Warning signs:** Test hangs indefinitely instead of timing out cleanly.

### Pitfall 3: File Path Resolution
**What goes wrong:** User passes a relative path like `./plans/foo.md`. If the binary changes directory internally (it does not currently, but worth verifying), the relative path breaks.
**Why it happens:** `std::fs::read_to_string` resolves relative to the process's working directory at call time. The binary does not change directory.
**How to avoid:** No special handling needed — `std::fs::read_to_string` with a relative path works correctly as long as cwd is not changed. Document in tests that both absolute and relative paths are tested.
**Warning signs:** File not found errors in CI when test fixtures use relative paths.

### Pitfall 4: Missing Frontend Assets Check (debug builds)
**What goes wrong:** In debug builds, `rust-embed` reads assets from the filesystem. If `ui/dist/` is missing, the server starts but serves nothing. The existing `run_hook_flow` has a `#[cfg(debug_assertions)]` guard that checks for `index.html` before starting.
**Why it happens:** `run_opencode_flow` (the template for `run_review_flow`) does NOT have this guard — it was added only in `run_hook_flow`. This is a gap in the current codebase.
**How to avoid:** Either add the same `#[cfg(debug_assertions)]` check to `run_review_flow`, or extract it into a shared helper that all flow entry points call.
**Warning signs:** Server starts on correct port, browser opens, but page is blank with no assets served in debug mode.

### Pitfall 5: Diff Extraction Cwd
**What goes wrong:** The `review` subcommand uses `std::env::current_dir()` for diff extraction (same as opencode flow). If the user is in a non-git directory, `extract_diff` returns empty string — which is correct behavior, not an error.
**Why it happens:** `extract_diff` already handles non-git paths gracefully (returns `""`). No action needed, but test coverage should confirm the empty diff case doesn't crash.
**How to avoid:** Already handled. Verify in unit test or integration test that reviewing a file while cwd is not a git repo produces `{"behavior":"allow"}` output (not a crash).

## Open Questions

1. **Should `review` extract a git diff from cwd?**
   - What we know: The hook flows extract the diff to show alongside the plan. The opencode flow (`run_opencode_flow`) also extracts the diff from cwd.
   - What's unclear: For a generic markdown review (not necessarily a plan), the diff may be irrelevant or confusing.
   - Recommendation: Default to diff extraction (same as opencode) for consistency. The diff panel in the UI is optional — if no diff content, the panel is empty and the user sees only the markdown. Verify this behavior is already handled in the UI.

2. **Should `review` have a `--no-diff` flag?**
   - What we know: No such flag exists anywhere in the current CLI.
   - What's unclear: Whether users reviewing non-plan documents (PRDs, READMEs) want the diff panel.
   - Recommendation: Defer to v0.4.0. Ship without `--no-diff` for now — the diff panel is non-disruptive when empty.

3. **Should `review` accept `-` (stdin) as the file argument?**
   - What we know: Standard Unix convention is `-` for stdin.
   - What's unclear: Whether this is needed for script/agent workflows.
   - Recommendation: Out of scope for this phase. The stated use case is "any markdown file" — stdin streaming is a separate feature.

## Security Domain

This phase adds a subcommand that reads a file from the local filesystem using a path supplied by the user (via CLI argument). Security considerations:

- **Path traversal:** `std::fs::read_to_string` follows symlinks and resolves `..` — this is intentional and correct for a local tool that the user invokes from their own terminal. No sandboxing is needed.
- **XSS via markdown:** The markdown is rendered server-side by `comrak` and sanitized by `ammonia` (already in the pipeline from the existing flows). No new XSS surface is introduced.
- **No network exposure:** The review server binds to `127.0.0.1` only (confirmed in `server.rs`). No change needed.

No new ASVS categories apply beyond those already addressed by the existing review flow.
