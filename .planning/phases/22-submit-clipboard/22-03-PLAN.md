       ```
       async function handleApprove() {
         if (shouldUseClipboard(connectivity)) {
           const json = buildClipboardPayload('allow', '', '', annotations)
           navigator.clipboard.writeText(json)
             .then(() => setSubmitState('clipboard_confirmed'))
             .catch(() => { setClipboardJson(json); setSubmitState('clipboard_error') })
           return
         }
         try {
           const res = await fetch('/api/decide', {
             method: 'POST',
             headers: { 'Content-Type': 'application/json' },
             body: JSON.stringify({ behavior: 'allow' }),
           })
           if (res.ok || res.status === 409) setSubmitState('confirmed_allow')
           else setSubmitState('clipboard_error') // fall-through to clipboard fallback view? NO — use a distinct error state per UI-SPEC §"Approve Flow (online)" — set a generic 'error' branch is not modeled in the state union, so map fetch failure to clipboard_error with the JSON we WOULD have written, mirroring App.tsx pattern.
         } catch {
           const json = buildClipboardPayload('allow', '', '', annotations)
           setClipboardJson(json)
           setSubmitState('clipboard_error')
         }
       }
       ```
       Implementation note: the state union in UI-SPEC §"SubmitControls internal state machine" has exactly six states and no plain `'error'`. When the online POST fails (network error or non-ok/non-409 response), fall back to populating `clipboardJson` with the payload and setting state to `clipboard_error` so the user can still hand off the decision via copy/paste.        async function handleAskForChanges(overallMessage: string) {
       ```
         if (shouldUseClipboard(connectivity)) {
           const json = buildClipboardPayload('deny', overallMessage, '', annotations)
           navigator.clipboard.writeText(json)
             .then(() => setSubmitState('clipboard_confirmed'))
             .catch(() => { setClipboardJson(json); setSubmitState('clipboard_error') })
           return

         const message = serializeAnnotations(overallMessage, '', annotations)
         try {

             body: JSON.stringify({ behavior: 'deny', message }),

           if (res.ok || res.status === 409) setSubmitState('confirmed_deny')
           else {
             const json = buildClipboardPayload('deny', overallMessage, '', annotations)
             setClipboardJson(json)
             setSubmitState('clipboard_error')

         } catch {

           setClipboardJson(json)
           setSubmitState('clipboard_error')
         }
       ```

## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Browser DOM → component state | User clicks Approve / Send Feedback / popover button — events drive state transitions and submission. |
| Component state → POST /api/decide | The component sends a JSON body to the local server with `behavior` and (deny only) `message` fields. |
| Component state → clipboard | The component writes a JSON string to the OS clipboard. |
| User-typed overallMessage → buildClipboardPayload / serializeAnnotations | Untrusted text flows into both the JSON body and the clipboard payload. |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-22-07 | Spoofing | POST /api/decide | accept | The local server binds to 127.0.0.1 (existing constraint, established in Phase 12). The PermissionRequest hook is a single-user local tool — no remote attacker can reach the endpoint. |
| T-22-08 | Tampering | overallMessage → JSON body | mitigate | The message string is embedded via `JSON.stringify({ behavior, message })` — control characters and quotes are JSON-escaped. The server receives valid JSON and treats `message` as opaque text. `serializeAnnotations` further prefixes every line with `> ` so embedded markdown cannot break the surrounding fences. |
| T-22-09 | Tampering | overallMessage → clipboard payload | mitigate | Same `JSON.stringify` escaping as the network path. The clipboard recipient is the user's own Claude conversation; the user is the source of the text. No injection vector exists. |
| T-22-10 | Repudiation | Approve / Ask for Changes click | accept | Single-user local tool; no audit log requirement. The 409 "already decided" handling is informational, not a security control. |
| T-22-11 | Information Disclosure | clipboard_error read-only textarea | accept | The textarea shows the JSON payload the user would have sent. The data is the user's own input + their own annotations — no system secrets, tokens, or paths are exposed. |
| T-22-12 | Denial of Service | Auto-close `setTimeout` | accept | `try { window.close() } catch {}` swallows any browser-policy refusal; cleanup via `clearTimeout` in the `useEffect` return prevents handler leaks. |
| T-22-13 | Elevation of Privilege | navigator.clipboard.writeText | accept | The Clipboard API requires user activation (a recent click), which is satisfied by the Approve or Send Feedback button click. Pitfall 4's transient-activation guard is enforced by the no-`await`-before-writeText pattern, asserted in the source-contract test. |
| T-22-SC | Tampering | npm/pip/cargo installs | n/a | No packages installed in this plan. |
