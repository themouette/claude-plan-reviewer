## Interaction Contract

### Gate Contract (SUBMIT-01)

**Approve button enabled condition:** `annotations.length === 0`
- When 0 comments: Approve is enabled; Ask for Changes is disabled (with helper text)
- When ≥1 comments: Approve is disabled (with helper text); Ask for Changes is enabled

**Initial state on page load:** Both buttons in their default enable/disable state determined by annotation count (0 on first load → Approve enabled, Ask for Changes disabled).

**Real-time updates:** Gate state recalculates on every render where `annotations` prop changes. No debounce. No explicit "re-check" mechanism needed.

### Approve Flow (online)

1. User has 0 annotations. Approve button is enabled.
2. User clicks "Approve".
3. `shouldUseClipboard(connectivity)` returns `false` → online path.
4. `POST /api/decide` with `{ behavior: 'allow' }`.
5. Response `ok` or `409` → `submitState` transitions to `confirmed_allow`.
6. Submit bar shows inline confirmation band: "Approve — done" in green.
7. Auto-close attempt: `setTimeout(() => { try { window.close() } catch {} }, 500)` — matches App.tsx pattern.

### Approve Flow (offline)

1. User has 0 annotations. Approve button is enabled.
2. User clicks "Approve".
3. `shouldUseClipboard(connectivity)` returns `true` (connectivity is `'offline'`) → offline path.
4. `buildClipboardPayload('allow', '', '', annotations)` called synchronously inside the click handler (no `await` before it — transient activation safety).
5. `navigator.clipboard.writeText(json)` called:
   - `.then()`: `submitState` → `clipboard_confirmed`
   - `.catch()`: store raw JSON in `clipboardJson` state, `submitState` → `clipboard_error`

### Ask for Changes Flow (online)

1. User has ≥1 annotations. Ask for Changes button is enabled.
2. User clicks "Ask for Changes" → `submitState` → `ask_form`. Overall-message textarea appears with `autoFocus`.
3. User types optional message (may remain empty).
4. User clicks "Submit" (or `Cmd+Enter` / `Ctrl+Enter` in textarea).
5. `shouldUseClipboard(connectivity)` returns `false` → online path.
6. `serializeAnnotations(overallMessage, '', annotations)` produces the `message` string.
7. `POST /api/decide` with `{ behavior: 'deny', message }`.
8. Response `ok` or `409` → `submitState` → `confirmed_deny`.
9. Submit bar shows inline confirmation band: "Ask for changes — done" in red.

### Ask for Changes Flow (offline)

1. As above steps 1–4.
2. `shouldUseClipboard(connectivity)` returns `true` → offline path.
3. `buildClipboardPayload('deny', overallMessage, '', annotations)` called synchronously.
4. `navigator.clipboard.writeText(json)`:
   - `.then()`: `submitState` → `clipboard_confirmed`
   - `.catch()`: store JSON in `clipboardJson` state, `submitState` → `clipboard_error`

### Cancel Ask-for-Changes Form

1. User clicked Ask for Changes, `ask_form` state is active.
2. User clicks "Cancel" or presses `Escape`.
3. `submitState` → `idle`. Overall-message textarea clears. Focus returns to Ask for Changes button.

### Keyboard Contract (Submit Bar)

| Key | Context | Action |
|-----|---------|--------|
| `Escape` | `ask_form` state (textarea focused) | Cancel → return to `idle` |
| `Cmd+Enter` / `Ctrl+Enter` | `ask_form` state (textarea focused) | Submit ask-for-changes |
| `Tab` | Any submit bar state | Natural focus order within bar |
| No "Enter to approve" shortcut | n/a | The v2 shell has many interactive elements — an Enter-to-approve global shortcut is too dangerous in this context. Approve requires explicit button click only. |

## Accessibility Contract

- Approve button: `type="button"`, `disabled={!canApprove}`, `aria-disabled` not needed (native `disabled` attribute is sufficient for ARIA)
- Ask for Changes button: `type="button"`, `disabled={!canAskChange}`
- Submit bar `<footer>` element: `role="contentinfo"` is not appropriate — use a plain `<div>` or `<section aria-label="Submit review">` to avoid semantic collision with page-level `<footer>`
- Offline banner: `role="status"` (matches App.tsx pattern; non-assertive live region — status, not alert)
- Overall-message textarea: `aria-label="Overall message (optional)"` (no visible label in compact bar layout)
- Helper text: `aria-live="polite"` on the helper text container so screen readers announce gate changes when annotation state changes
- Confirmation band: `role="status"` for the confirmation message container; `aria-live="polite"`
- Clipboard error payload textarea: `readOnly`, `aria-label="JSON payload — copy and paste into Claude"`, `onClick` selects all text
- Focus ring: `outline: 2px solid var(--color-focus); outline-offset: 2px` on `:focus-visible` for ALL interactive elements — `.submit-bar-btn:focus-visible` in CSS, or inline `onFocus`/`onBlur` handlers matching established project pattern
