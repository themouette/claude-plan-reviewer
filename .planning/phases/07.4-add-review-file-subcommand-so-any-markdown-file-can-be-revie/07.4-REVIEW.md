# Phase 07.4: Code Review Report

**Reviewed:** 2026-04-11
**Depth:** standard
**Files Reviewed:** 3
**Status:** issues_found

## Summary

This phase adds the `review` subcommand that lets users pass any markdown file directly to the browser UI without constructing hook-specific JSON. The implementation is clean and follows existing patterns well. All new CLI variants are covered by unit tests, and four integration tests exercise the full binary lifecycle.

Three warnings were found: a TOCTOU port-race in the test helpers (existing pattern inherited and now expanded), a `tokio::runtime` builder panic path with no user-friendly fallback, and significant code duplication between `run_review_flow` and the pre-existing `run_opencode_flow`. Two info items cover naming coupling and a deliberate test-fixture memory leak.

No security issues or data-loss bugs were found.

## Warnings

### WR-01: TOCTOU race between `find_free_port()` and process spawn

**File:** `tests/integration/review_subcommand.rs:6-8`
**Issue:** `find_free_port()` binds a `TcpListener` to port 0, reads the OS-assigned port, then immediately drops the listener. The port number is then passed via `--port` to the spawned child process. Between the drop and the child's `bind()`, another process on the system could claim that port, causing `wait_for_port` to time out (flaky test) or the server to fail silently with a different port.

This is an inherited pattern from the existing `server_cycle` integration test, but is now replicated into two additional tests (`review_approve`, `review_deny_with_message`, `review_serves_plan_content`) that follow the same shape.

**Fix:** Pass `--port 0` and let the server pick its own port. After the server starts, read the actual bound port from stderr (the line `"Review UI: http://127.0.0.1:<port>"` is already emitted). Alternatively, keep a `TcpListener` alive until after the child has bound — which requires the child to coordinate. The stderr-scraping approach is the lowest-friction fix:

```rust
fn spawn_review_flow_port_zero(file_path: &str) -> (std::process::Child, u16) {
    // Let OS assign a free port; read it back from stderr output
    let home = Box::leak(Box::new(tempfile::TempDir::new().unwrap()));
    let mut child = Command::new(binary_path())
        .env("HOME", home.path())
        .args(["--no-browser", "--port", "0", "review", file_path])
        .stdin(Stdio::null())
        .stdout(Stdio::piped())
        .stderr(Stdio::piped())
        .spawn()
        .expect("failed to spawn plan-reviewer review");
    // Read stderr lines until we find the "Review UI:" line
    // (requires switching stderr to BufReader — see server_cycle tests for precedent)
    // ...
    (child, port)
}
```

### WR-02: `tokio::runtime::Builder::build()` called with `.unwrap()` — silent panic on OS resource exhaustion

**File:** `src/main.rs:541-544`
**Issue:** Building the tokio runtime panics the process if the OS denies thread creation (e.g., `RLIMIT_NPROC` reached, out of memory). The pre-existing `run_opencode_flow` (line 492-495) has the same issue. The new `run_review_flow` copies the same pattern. A panic produces a Rust backtrace on stderr rather than a clean error message, which is confusing for end users.

**Fix:** Replace `.unwrap()` with explicit error handling so the process exits with a human-readable message:

```rust
let rt = tokio::runtime::Builder::new_current_thread()
    .enable_all()
    .build()
    .unwrap_or_else(|e| {
        eprintln!("Failed to create async runtime: {}", e);
        std::process::exit(1);
    });
```

### WR-03: `run_review_flow` and `run_opencode_flow` are near-identical — duplication risks divergence

**File:** `src/main.rs:469-551`
**Issue:** `run_review_flow` (lines 510-551) and `run_opencode_flow` (lines 469-502) share all of the following identical steps: read file into `plan_md`, log byte count, call `extract_diff` on cwd, log byte count, build tokio runtime, call `async_main`, call `build_opencode_output`, write JSON to stdout. The only behavioural difference is the `#[cfg(debug_assertions)]` asset check that was added to `run_review_flow` but is absent from `run_opencode_flow`. This means any future bug fix (e.g., WR-02 above) must be applied twice. The opencode path already lacks the asset check, suggesting the paths have already started to diverge.

**Fix:** Extract a shared `run_file_flow` function parameterised by file path, then call it from both:

```rust
fn run_file_flow(no_browser: bool, port: u16, file: &str) {
    let plan_md = match std::fs::read_to_string(file) {
        Ok(content) => content,
        Err(e) => {
            eprintln!("Failed to read file at {}: {}", file, e);
            std::process::exit(1);
        }
    };
    eprintln!("Plan loaded from file ({} bytes): {}", plan_md.len(), file);

    #[cfg(debug_assertions)]
    {
        if server::Assets::get("index.html").is_none() {
            eprintln!("ERROR: Frontend assets not found at ui/dist/index.html");
            eprintln!("Run 'cd ui && npm run build' first.");
            std::process::exit(1);
        }
    }

    let cwd = std::env::current_dir()
        .map(|p| p.to_string_lossy().into_owned())
        .unwrap_or_default();
    let diff_content = extract_diff(&cwd);
    eprintln!("Diff extracted ({} bytes)", diff_content.len());

    let rt = tokio::runtime::Builder::new_current_thread()
        .enable_all()
        .build()
        .unwrap_or_else(|e| { eprintln!("Failed to create runtime: {}", e); std::process::exit(1) });

    let decision = rt.block_on(async_main(no_browser, port, plan_md, diff_content));
    let output = build_opencode_output(&decision);
    serde_json::to_writer(std::io::stdout(), &output).expect("failed to write hook output");
}

fn run_opencode_flow(no_browser: bool, port: u16, plan_file: &str) {
    run_file_flow(no_browser, port, plan_file);
}

fn run_review_flow(no_browser: bool, port: u16, file: &str) {
    run_file_flow(no_browser, port, file);
}
```

## Info

### IN-01: `run_review_flow` uses `build_opencode_output` — naming coupling

**File:** `src/main.rs:549`
**Issue:** The `review` subcommand is documented as producing "neutral `{behavior}` JSON", but its implementation calls `build_opencode_output`. If the opencode output format ever changes (e.g., gains opencode-specific fields), `review` will silently inherit those changes. The intent is to have a stable, integration-agnostic format, but the implementation is tied to an integration-specific builder.

**Fix:** Either rename `build_opencode_output` to `build_neutral_output` (since both `review` and opencode now share it), or create a thin wrapper that makes the intent explicit:

```rust
/// Build the neutral {behavior} output format used by `review` subcommand
/// and the opencode integration.
fn build_neutral_output(decision: &Decision) -> serde_json::Value {
    build_opencode_output(decision)  // same format for now; rename if they diverge
}
```

### IN-02: `Box::leak` in test helper accumulates TempDirs for the lifetime of the test process

**File:** `tests/integration/review_subcommand.rs:37-39`
**Issue:** The comment acknowledges the leak: `Box::leak(Box::new(home))` intentionally prevents the `TempDir` from being dropped while the child process runs. However, each test invocation that calls `spawn_review_flow` leaks one `TempDir` permanently. With four integration tests each calling `spawn_review_flow` once, four temp directories are leaked per test binary invocation. This is benign in CI (process exits, OS reclaims) but can accumulate on developer machines if tests are run frequently without cleanup.

**Fix:** Pass the `TempDir` back to the caller so the caller can drop it after `wait_with_output()`:

```rust
fn spawn_review_flow(file_path: &str, port: u16) -> (std::process::Child, tempfile::TempDir) {
    let home = tempfile::TempDir::new().unwrap();
    let child = Command::new(binary_path())
        .env("HOME", home.path())
        .args(["--no-browser", "--port", &port.to_string(), "review", file_path])
        .stdin(Stdio::null())
        .stdout(Stdio::piped())
        .stderr(Stdio::piped())
        .spawn()
        .expect("failed to spawn plan-reviewer review");
    (child, home)  // caller drops `home` after child exits
}
```
