### Bug A: Paragraph hover background not visible

**Symptom:** Moving the mouse over paragraphs in `/v2` shows no visible background color change.

**What was tried:**
- Changed inline style from `rgba(255,255,255,0.04)` → `rgba(255,255,255,0.08)` in `PlanContent.tsx`
- Moved `onMouseMove`/`onMouseLeave` from `.plan-prose` to outer wrapper (to avoid GutterIcon flicker loop)
- Added early-return in `handleMouseMove` when target is the GutterIcon button

**Not yet investigated:**
- Is `hoveredParagraph` state actually being set? (Add a `console.log` in `handleMouseMove` to confirm)
- Is the `useEffect([hoveredParagraph])` running? Could there be a closure/stale-ref issue?
- Is another CSS rule overriding the inline `background` style? (Check DevTools "Computed" tab for `background-color` on a hovered `<p>`)
- Does the outer wrapper `onMouseMove` even receive events over the `.plan-prose` content? (The `dangerouslySetInnerHTML` div might swallow events)

**Key file:** `ui/src/reviewer-v2/PlanContent.tsx`

**Hypothesis to verify first:** Open DevTools → Elements → hover a paragraph → check if `style="background: rgba(255,255,255,0.08); ..."` is being applied to the `<p>` element at all. If not, the `useEffect` isn't running or `hoveredParagraph` isn't being set.

### Bug B: Text selection always starts from top of document

**Symptom:** Click-drag to select text — the selection visually anchors at position 0 (top of document) rather than at the click point.

**What was tried:**
- Removed `selection.removeAllRanges()` from the `mouseup` capture handler (was suspected to reset selection anchor in WebKit)
- Added `selectionchange` listener with mouse-drag guard (mouseDown flag)

**Not yet investigated:**
- Is the CSS Custom Highlight API (`applyHighlight` / `CSS.highlights.set('selection-lock', ...)`) interfering with native selection? Try temporarily removing the `applyHighlight(freshRange)` call to see if selection behavior normalizes.
- The `useLayoutEffect` (no deps array) runs after EVERY render and calls `applyHighlight(range)` from `storedOffsets.current`. During hover mousemove (frequent renders), if `storedOffsets.current` is non-null, this re-applies a CSS highlight on every render. Could this be interfering with in-progress native selection?
- Try disabling the CSS Custom Highlight entirely (comment out `applyHighlight` calls) and check if native selection works correctly. If it does, the `CSS.highlights` API is the culprit.

**Key file:** `ui/src/reviewer-v2/hooks/useTextSelection.ts`

**Hypothesis to verify first:** Comment out the `applyHighlight(freshRange)` call in `processSelection()` and the `applyHighlight(range)` in `useLayoutEffect`. Does native selection start from the correct position? If yes, the CSS Custom Highlight API is interfering with native selection.
