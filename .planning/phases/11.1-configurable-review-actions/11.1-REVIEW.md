# Phase 11.1: Code Review Report

**Reviewed:** 2026-04-12
**Depth:** standard
**Files Reviewed:** 5

## Summary

This phase adds configurable Approve/Deny button labels to the `review` subcommand via `--approve-label` and `--deny-label` CLI flags. The labels are propagated through `async_main` → `start_server` → `AppState`, exposed via a new `/api/config` endpoint, and consumed by the React frontend. The `annotate.md` command template in the Claude plugin is updated to pass the new flags with "No issues" / "Leave feedback" labels.

The implementation is clean and consistent. No security issues were found. Two warnings relate to a TOCTOU (time-of-check-time-of-use) race in the integration test port allocation and hardcoded "Plan approved/denied" text in `ConfirmationView` that does not reflect custom labels. Three informational items are also noted.

## Warnings

### WR-01: TOCTOU race in `find_free_port` — port may be taken before the server binds

**File:** `tests/integration/review_subcommand.rs:6-9`
**Issue:** `find_free_port` binds a `TcpListener` to port 0, records the OS-assigned port, then **drops the listener** before spawning the child process. Between the `drop` and the child binding to the same port number, another process (or a parallel test) can claim that port. The resulting failure is a flaky test, not a compile-time error, and it is more likely when the full test suite runs in parallel with `cargo test`.

```rust
fn find_free_port() -> u16 {
    let listener = std::net::TcpListener::bind("127.0.0.1:0").unwrap();
    listener.local_addr().unwrap().port()
    // listener is dropped here — port is free for the OS to reassign
}
```

**Fix:** Pass `port: 0` directly to the binary and let the server bind to `0`. Then read the actual bound port from the server's stderr output (which already prints `"Review UI: http://127.0.0.1:<port>"`), or keep the listener alive by leaking/returning it alongside the port so no gap exists:

```rust
fn find_free_port() -> (u16, std::net::TcpListener) {
    let listener = std::net::TcpListener::bind("127.0.0.1:0").unwrap();
    let port = listener.local_addr().unwrap().port();
    (port, listener)  // caller drops it only after the child has had time to bind
}
```

Alternatively — and simpler — pass `--port 0` to the binary, parse the port from stderr, then `wait_for_port` is unnecessary. This is the safer long-term approach.

### WR-02: `ConfirmationView` shows hardcoded "Plan approved/denied" regardless of custom labels

**File:** `ui/src/App.tsx:455`
**Issue:** When custom `--approve-label` / `--deny-label` values are set (e.g., "No issues" / "Leave feedback"), the post-decision confirmation screen still reads "Plan approved" or "Plan denied". This is inconsistent with the user-facing button labels and confusing when the `review` subcommand is used in a non-plan-review context (e.g., document annotation via `/plan-reviewer:annotate`).

```tsx
{approved ? 'Plan approved' : 'Plan denied'}
```

**Fix:** Thread `approveLabel`/`denyLabel` (or a derived past-tense string) into `ConfirmationView`. The simplest approach: add optional `approveLabel`/`denyLabel` props to `ConfirmationView` and use them in the heading:

```tsx
function ConfirmationView({
  decision,
  approveLabel = 'Approve',
  denyLabel = 'Deny',
}: {
  decision: Decision
  approveLabel?: string
  denyLabel?: string
}) {
  const approved = decision === 'allow'
  return (
    ...
    <h2>
      {approved ? `${approveLabel} — done` : `${denyLabel} — done`}
    </h2>
    ...
  )
}
```

Then at the call site:

```tsx
{appState === 'confirmed' && decision && (
  <ConfirmationView
    decision={decision}
    approveLabel={approveLabel}
    denyLabel={denyLabel}
  />
)}
```

## Info

### IN-01: `Box::leak` in integration tests causes intentional memory leak per test run

**File:** `tests/integration/review_subcommand.rs:39` and `67`
**Issue:** Both `spawn_review_flow` and `spawn_review_flow_with_labels` call `Box::leak(Box::new(tempdir))` to keep the temp directory alive for the child process. This is a known pattern but leaks memory permanently for each call. In a test binary that runs to completion this is harmless, but if test helpers are ever reused in a long-running harness the leak accumulates.

**Fix:** No immediate action required. If this becomes a problem, return the `TempDir` from the spawn helper and let the caller hold it for the child's lifetime:

```rust
fn spawn_review_flow(file_path: &str, port: u16) -> (std::process::Child, tempfile::TempDir) {
    let home = tempfile::TempDir::new().unwrap();
    let child = Command::new(binary_path())
        .env("HOME", home.path())
        ...
        .spawn()
        .expect("...");
    (child, home)  // caller keeps TempDir alive; it drops when the test function returns
}
```

### IN-02: `/api/config` is unauthenticated and exposes button labels — acceptable for local-only tool, worth documenting

**File:** `src/server.rs:50-55`
**Issue:** The `/api/config` endpoint returns `approve_label` and `deny_label` with no authentication. Because the server is bound to `127.0.0.1` only and the tool is documented as "local-only for v1", this is by design. The concern is theoretical (SSRF, local network attack), but the labels are not sensitive data.

**Fix:** No change needed. Consider adding a comment in `start_server` noting that all API routes are intentionally unauthenticated since the server only accepts loopback connections:

```rust
// All routes are intentionally unauthenticated — the server is bound to 127.0.0.1
// only and serves a single local user session.
```

### IN-03: Dead constant slice: `inlineChips` is always empty

**File:** `ui/src/App.tsx:168`
**Issue:** `inlineChips` is assigned `QUICK_ACTIONS.slice(0, 0)`, which is always an empty slice. The `inlineChips.map(...)` render block at line 230 therefore never renders anything. This is dead code — either the split was intended to be a non-zero index or the inline chip feature was removed but the code was not cleaned up.

```tsx
const inlineChips = QUICK_ACTIONS.slice(0, 0)   // always empty
const overflowChips = QUICK_ACTIONS.slice(0)     // all items
```

**Fix:** Remove the `inlineChips` constant and its `map` render block (lines 230-255), since all chips already appear under the `overflowChips` `<details>` dropdown. Or, if inline chips are intentionally planned for a future threshold, add a comment explaining the `0` upper bound.
