# Phase 11.1: Configurable Review Actions - Research

**Researched:** 2026-04-11
**Domain:** Rust CLI flags (clap), axum HTTP server state, React/TypeScript frontend data flow
**Confidence:** HIGH

## Summary

Phase 11.1 adds two optional CLI flags (`--approve-label`, `--deny-label`) to `plan-reviewer review`. The labels must flow from the CLI into the running server's state, then be delivered to the React frontend via a new API endpoint (or by extending an existing one), so the Approve and Deny buttons render custom text. When the flags are omitted, the existing "Approve" / "Deny" defaults are used without any behavior change.

The existing codebase provides all the required primitives. The change touches four layers:
1. `src/main.rs` — extend `Commands::Review` struct and thread labels into `run_review_flow`
2. `src/server.rs` — add `approve_label`/`deny_label` to `AppState`, expose via `/api/config`
3. `ui/src/App.tsx` — fetch `/api/config` on mount; bind button text to fetched labels
4. `src/integrations/claude.rs` — update the `annotate_content` string literal to pass `--approve-label "No issues" --deny-label "Leave feedback"`

The `annotate.md` file is written from a Rust `concat!()` string literal inside `claude.rs`. Updating it requires changing that literal and updating the corresponding unit test assertions.

**Primary recommendation:** Add `/api/config` as a new GET endpoint returning `{ approve_label, deny_label }`. The frontend fetches it on mount alongside `/api/plan`. Default values are "Approve" and "Deny" in the server when flags are absent.

## Architecture Patterns

### Pattern 1: Extend clap subcommand struct with named fields

The existing `Commands::Review` is a unit-like variant with a single positional `file: String`.
Add two optional named flags using the derive macro. [VERIFIED: src/main.rs lines 329-333]

**Current state:**
```rust
Review {
    /// Path to the markdown file to review
    file: String,
},
```

**After change:**
```rust
Review {
    /// Path to the markdown file to review
    file: String,
    /// Label for the Approve button in the browser UI
    #[arg(long, default_value = "Approve")]
    approve_label: String,
    /// Label for the Deny button in the browser UI
    #[arg(long, default_value = "Deny")]
    deny_label: String,
},
```

Using `default_value` rather than `Option<String>` keeps the downstream call site simple (no `.unwrap_or`). The clap derive macro automatically generates help text. [VERIFIED: clap 4.x derive macro, CLAUDE.md recommended stack]

**Impact on `main`:** The `Commands::Review` destructure in `main()` must be updated to capture the new fields and pass them to `run_review_flow`:

```rust
Some(Commands::Review { file, approve_label, deny_label }) => {
    run_review_flow(cli.no_browser, cli.port, file, &approve_label, &deny_label);
}
```

### Pattern 2: Extend AppState with label fields

`AppState` in `server.rs` currently holds `plan_md`, `diff_content`, and `decision_tx`. Add two `String` fields. [VERIFIED: src/server.rs lines 30-36]

```rust
pub struct AppState {
    pub plan_md: String,
    pub diff_content: String,
    pub approve_label: String,
    pub deny_label: String,
    pub decision_tx: Mutex<Option<oneshot::Sender<Decision>>>,
}
```

Extend `start_server` signature to accept the two new parameters:

```rust
pub async fn start_server(
    plan_md: String,
    diff_content: String,
    approve_label: String,
    deny_label: String,
    port: u16,
) -> Result<(u16, oneshot::Receiver<Decision>), Box<dyn std::error::Error + Send + Sync>>
```

### Pattern 3: New GET /api/config endpoint

Add a lightweight handler in `server.rs`:

```rust
async fn get_config(State(state): State<Arc<AppState>>) -> impl IntoResponse {
    Json(serde_json::json!({
        "approve_label": state.approve_label,
        "deny_label": state.deny_label,
    }))
}
```

Register it in the router:
```rust
.route("/api/config", get(get_config))
```

This is the minimal change that avoids coupling label data to the plan endpoint response format. The frontend fetches `/api/config` independently.

### Pattern 4: Frontend — fetch /api/config on mount

The frontend currently fetches `/api/plan` and `/api/diff` in separate `useEffect` hooks on mount. [VERIFIED: App.tsx lines 578-643]

Add a third `useEffect` for `/api/config`:

```typescript
const [approveLabel, setApproveLabel] = useState('Approve')
const [denyLabel, setDenyLabel] = useState('Deny')

useEffect(() => {
  fetch('/api/config')
    .then((res) => res.ok ? res.json() : null)
    .then((data: { approve_label: string; deny_label: string } | null) => {
      if (data) {
        setApproveLabel(data.approve_label)
        setDenyLabel(data.deny_label)
      }
    })
    .catch(() => { /* non-critical — defaults remain */ })
}, [])
```

The Approve and Deny button text in the JSX changes from the string literals `'Approve'` and `'Deny'` to `{approveLabel}` and `{denyLabel}`. [VERIFIED: App.tsx lines 1267, 1305 — hardcoded strings today]

**Deny form submit button** (line 1388): the inner button labeled `'Submit Denial'` does NOT need a custom label. It is the confirmation step after the deny toggle, which the user explicitly clicks. Only the initial toggle button ("Deny") gets the custom label.

The `annotate_content` `concat!()` string literal at `claude.rs` line 182-184 currently reads:

It must become:

The existing unit test at `install_creates_annotate_md_with_expected_content` checks for the string `"plan-reviewer review"` — it will still pass. However, it also checks text in Step 2 ("Use **Deny** to leave feedback with comments, **Approve** if you're satisfied") which must be updated to match the new framing ("No issues" / "Leave feedback"). The Step 2 user-facing message should also be updated for consistency:
```

```

### Anti-Patterns to Avoid

- **Extending `/api/plan` response** with label fields: This couples concerns. The plan endpoint is about markdown content; labels are UI configuration. Keep them separate.
- **Passing labels through CLI global flags** (`--approve-label` at the top-level `Cli` struct): The flags are specific to the `review` subcommand. Putting them at the top level would allow them to silently be accepted (and ignored) for `install`, `uninstall`, etc. Keep them scoped to `Commands::Review`.
- **Using `Option<String>` with `unwrap_or`**: Using `default_value = "Approve"` in the clap attribute is cleaner. The function signature takes `&str`, not `Option<&str>`, keeping all callers simple.
- **Making `/api/config` a blocking requirement**: If it fails, the UI must degrade gracefully to defaults rather than entering error state. Labels are cosmetic.

## Don't Hand-Roll

| Problem | Don't Build | Use Instead | Why |
|---------|-------------|-------------|-----|
| CLI default values | Conditional logic in `main()` | `#[arg(default_value = "Approve")]` | clap handles missing-flag fallback automatically |
| Frontend default labels | Conditional rendering | `useState('Approve')` initialized to defaults | React state naturally holds the fallback |

## Common Pitfalls

### Pitfall 1: Forgetting to update run_review_flow signature and async_main
**What goes wrong:** Labels added to `Commands::Review` but not threaded through `run_review_flow` → `async_main` → `start_server`. Compilation fails or labels silently drop.
**Why it happens:** `run_review_flow` calls `async_main` which calls `start_server`. Each link in the chain must carry the new parameters.
**How to avoid:** Trace the call chain: `main()` → `run_review_flow(no_browser, port, file, approve_label, deny_label)` → `async_main(no_browser, port, plan_md, diff_content)` — note `async_main` does NOT need the labels (it returns a `Decision`). Only `start_server` needs them for `AppState`.
**Correct chain:** `run_review_flow` calls `start_server(plan_md, diff_content, approve_label, deny_label, port)` directly or via `async_main` extended to take labels.

The cleanest approach is to NOT thread labels through `async_main` (which is shared with `run_hook_flow` and `run_opencode_flow`). Instead, pass them only to `start_server`. This requires calling `start_server` with the label parameters only from `run_review_flow`.

**Implication:** `async_main` currently takes `plan_md` and `diff_content` and calls `start_server`. If label parameters are added to `start_server`, `async_main` must also accept them — OR `run_review_flow` must not use `async_main` at all and must call `server::start_server` directly. The latter avoids touching the shared `async_main` that hook flows use. [VERIFIED: main.rs lines 469-551 — run_review_flow already duplicates run_opencode_flow structure]

**Recommended approach:** Since `run_review_flow` already has its own runtime setup that mirrors `run_opencode_flow` (lines 510-551), call `server::start_server` with the label args directly inside `run_review_flow`'s `rt.block_on(async { ... })`, bypassing `async_main`. This keeps hook flows untouched.

### Pitfall 2: Deny toggle button vs. deny submit button label confusion
**What goes wrong:** Custom deny label applied to inner "Submit Denial" button instead of (or in addition to) the toggle button.
**Why it happens:** There are two deny-related buttons: the toggle that opens the form, and the submit inside the form.
**How to avoid:** Only the toggle button (line 1305, text `Deny`) should use `denyLabel`. The submit button ("Submit Denial") stays hardcoded — it represents the confirmation action, not the initial denial intent.

### Pitfall 3: annotate.md test assertions break
**What goes wrong:** `install_creates_annotate_md_with_expected_content` test fails because the Step 2 message text changed.
**Why it happens:** The test explicitly checks for `"Use **Deny** to leave feedback"` phrasing (or similar).
**How to avoid:** After updating `annotate_content`, update the corresponding test assertion strings. Check `claude.rs` lines 848-918 for all content assertions. [VERIFIED: claude.rs test block read]

Current Step 2 text to change (claude.rs line 175-176):
```
> Use **Deny** to leave feedback with comments, **Approve** if you're satisfied.
```
Updated text:
```
> Use **Leave feedback** to add comments, **No issues** if you're satisfied.
```

### Pitfall 4: review_serves_plan_content integration test hard-codes spawn args
**What goes wrong:** `spawn_review_flow` helper in `tests/integration/review_subcommand.rs` passes hardcoded args. New tests that pass labels need a separate spawn helper or an args override.
**How to avoid:** Add a second spawn helper `spawn_review_flow_with_labels(file_path, approve_label, deny_label, port)` for the new label tests, keeping the existing helper unchanged (ACT-03 requirement).

## Open Questions (RESOLVED)

1. **Should the `ConfirmationView` messages also use custom labels?**
   - What we know: `ConfirmationView` shows "Plan approved" / "Plan denied" after decision (App.tsx line 455). These are confirmation messages, not button labels.
   - What's unclear: Whether the phase goal extends to confirmation text or only button text.
   - Recommendation: Scope to button text only per requirements ("renders those labels in the browser UI" — ACT-02 refers to button rendering). Confirmation text can remain hardcoded.

2. **Should `/api/config` be fetched in a Promise.all alongside `/api/plan`?**
   - What we know: Both are non-critical fetches needed before the UI is interactive. App state is `loading` until `/api/plan` resolves.
   - What's unclear: Whether a failed `/api/config` fetch should block the `reviewing` state transition.
   - Recommendation: Fetch independently with silent fallback. Config failure is non-blocking — the UI renders with defaults.
