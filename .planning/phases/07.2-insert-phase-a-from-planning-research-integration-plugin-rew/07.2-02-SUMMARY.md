# Phase 07.2 Plan 02: Rewrite GeminiIntegration — Extension Directory Model Summary

Rewrote GeminiIntegration install/uninstall to use the Gemini CLI extension directory auto-discovery model (`~/.gemini/extensions/plan-reviewer/`) instead of writing bare hook entries to `~/.gemini/settings.json`.

## What Was Built

**New `src/integrations/gemini.rs` implementation:**

- `gemini_extension_dir(home)` — `pub(crate)` helper returning `{home}/.gemini/extensions/plan-reviewer` (for use by future `update.rs`)
- `GeminiIntegration::install()` — writes `gemini-extension.json` manifest (with `CARGO_PKG_VERSION`) and `hooks/hooks.json` (with `BeforeTool`/`exit_plan_mode` hook, `timeout: 300000`); does NOT touch `settings.json`
- `GeminiIntegration::uninstall()` — removes extension directory entirely via `remove_dir_all`; checks `exists()` first (Pitfall 6 mitigation); does NOT touch `settings.json`
- `GeminiIntegration::is_installed()` — checks `gemini-extension.json` presence in extension dir (new idempotency key)
- `gemini_legacy_hook_installed()` — `pub(crate)` with `#[allow(dead_code)]`; kept for Phase 07.3 migration to detect and clean old settings.json entries

**Removed entirely:** `gemini_settings_path`, `gemini_hook_entry`, `gemini_is_installed` (old settings.json helpers)

**Test coverage:** 14 unit tests covering:
- `gemini_extension_dir` path computation
- `gemini_legacy_hook_installed` detection (true/false cases)
- `install` creates extension dir, writes valid manifests, does NOT create `settings.json`
- `install` is idempotent (double-install produces valid JSON files)
- `install` returns Err when `binary_path` is None
- `uninstall` removes directory entirely
- `uninstall` on non-existent directory returns Ok
- `is_installed` returns true when manifest exists, false when dir absent
