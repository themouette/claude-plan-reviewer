## Summary

This phase introduces three new integration backends (Claude Code plugin model, Gemini CLI extension model, OpenCode JS plugin), a refresh-on-update mechanism in `update.rs`, integration tests, and a React-based UI in `App.tsx`. The Rust integration code is generally well-structured, idempotent, and has solid test coverage. The main concern is a security issue in the OpenCode plugin (empty `catch {}` silently swallows errors in the `finally` block), two functional bugs in `update.rs` (silently discarding write errors during refresh and using a fragile binary-name assumption), and minor code quality issues in `App.tsx` and the Gemini manifest writer.

## Critical Issues

### CR-01: OpenCode plugin silently swallows `unlinkSync` errors, masking temp-file cleanup failures

**File:** `src/integrations/opencode_plugin.mjs:56-58`
**Issue:** The `finally` block catches all errors from `unlinkSync` with an empty `catch {}`. While temp-file cleanup failure is non-fatal on its own, an empty `catch` at this level has a separate problem: if `unlinkSync` throws synchronously (e.g. permission error or ENOENT race), the error is completely invisible. More critically, the pattern `try { unlinkSync(...) } catch {}` with no logging makes debugging impossible and silently leaves secrets (the plan markdown) in `/tmp` on certain failure modes (e.g. EPERM on a read-only tmp mount). This is the same pattern the review scope flags as a security/quality risk.

Additionally, because the outer `try/catch` returns a user-facing string on error (`return \`Plan review failed: ${err.message}\``), any exception from `execFileSync` (e.g. non-zero exit code) leaks the raw Node.js error message — including the full command line — back to the AI model. If the binary path contains sensitive information from the environment, this becomes a data-exposure vector.

**Fix:**
```js
} finally {
  try {
    unlinkSync(tmpFile);
  } catch (cleanupErr) {
    // Log to stderr so the hook runner can surface it without polluting stdout
    process.stderr.write(`plan-reviewer: failed to remove temp file ${tmpFile}: ${cleanupErr.message}\n`);
  }
}
```
For the outer error, sanitize before returning:
```js
} catch (err) {
  // Avoid leaking binary path or full stack trace to the AI model
  return `Plan review failed: ${err.code ?? 'UNKNOWN'} — check that plan-reviewer is installed and in PATH`;
}
```

## Warnings

### WR-01: `write_claude_plugin_files` / `write_gemini_extension_files` / `write_opencode_plugin_file` silently discard write errors

**File:** `src/update.rs:281-295` (Claude), `src/update.rs:326-338` (Gemini), `src/update.rs:353-356` (OpenCode)
**Issue:** All three `write_*` helper functions use `let _ = std::fs::write(...)` which discards `Err` values entirely. A failed write during `update` leaves stale integration files on disk (wrong version) with no error reported to the user. The binary has been replaced successfully, but the integration file still has the old version embedded — the next hook invocation will run the old plugin logic.

**Fix:** Return `Result<(), String>` from each helper and propagate errors up to `refresh_integrations_with_home`:
```rust
fn write_claude_plugin_files(home: &str, current_version: &str) -> Result<(), String> {
    // ...
    std::fs::write(manifest_dir.join("plugin.json"), ...)
        .map_err(|e| format!("cannot write plugin.json: {}", e))?;
    std::fs::write(hooks_dir.join("hooks.json"), ...)
        .map_err(|e| format!("cannot write hooks.json: {}", e))?;
    println!("plan-reviewer: Claude plugin files updated to v{}", current_version);
    Ok(())
}
```
Then in `refresh_integrations_with_home`, propagate or `eprintln!` the returned error so the user sees it.

### WR-02: `write_opencode_plugin_file` uses hardcoded binary name `"plan-reviewer"` without path — breaks if binary is installed to a non-`PATH` location

**File:** `src/update.rs:349-351`
**Issue:** During `update`, `write_opencode_plugin_file` replaces `__PLAN_REVIEWER_BIN__` with the literal string `"plan-reviewer"` (bare name). This differs from the `install` path, which uses `ctx.binary_path` (the absolute path provided at install time). If the user originally installed the binary to a custom prefix (e.g. `~/bin/plan-reviewer` which is in `PATH`), the update will rewrite the plugin with the bare name — which works only if the binary's directory is in `PATH` in the shell environment that executes the hook. If it is not, the hook silently fails with a `ENOENT` error from `execFileSync`.

The original install used the absolute path precisely to avoid this ambiguity.

**Fix:** Derive the binary path from `std::env::current_exe()` when rewriting during update, to match the path of the just-replaced binary:
```rust
fn write_opencode_plugin_file(home: &str, current_version: &str) {
    let bin_path = std::env::current_exe()
        .map(|p| p.to_string_lossy().to_string())
        .unwrap_or_else(|_| "plan-reviewer".to_string());

    let source = include_str!("integrations/opencode_plugin.mjs")
        .replace("__PLAN_REVIEWER_BIN__", &bin_path)
        .replace("__PLAN_REVIEWER_VERSION__", current_version);
    // ...
}
```

### WR-03: Gemini manifest written as compact JSON, inconsistent with the pretty-printed format used everywhere else

**File:** `src/integrations/gemini.rs:36-40`
**Issue:** `gemini-extension.json` is written using a hand-crafted `format!()` string that produces compact (single-line) JSON, while every other manifest in the codebase (`claude.rs`, `update.rs`) uses `serde_json::to_string_pretty`. While this is not a correctness bug, compact JSON is harder to diff, harder to inspect manually, and inconsistent with the rest of the code. If the format string ever needs a new field it is easy to forget to quote special characters correctly.

**Fix:** Use `serde_json::to_string_pretty` consistently:
```rust
let manifest = serde_json::json!({
    "name": "plan-reviewer",
    "version": crate_version(),
    "description": "Plan review hook for Gemini CLI"
});
let manifest_str = serde_json::to_string_pretty(&manifest)
    .map_err(|e| format!("cannot serialize gemini-extension.json: {}", e))?;
if let Err(e) = std::fs::write(&manifest_path, &manifest_str) { ... }
```

### WR-04: `App.tsx` — `dangerouslySetInnerHTML` with server-rendered HTML and no explicit sanitization step documented at the call site

**File:** `ui/src/App.tsx:930`
**Issue:** The plan HTML is produced by calling `marked.parse(data.plan_md)` client-side (line 421) and then injected via `dangerouslySetInnerHTML={{ __html: planHtml }}`. `marked` with GFM enabled does NOT sanitize HTML by default — any HTML present in the plan markdown (e.g. `<script>` tags, `<img onerror=...>`) will be passed through and executed in the browser.

In this application the plan comes from the local server, which in turn comes from the AI tool's hook stdin — so in the normal use case the content is trusted. However, if a plan ever includes raw HTML (which is valid GFM), it executes in the browser tab. For a local-only tool this is a low-severity concern, but it is a real XSS path if any annotation text or plan content is reflected back into other contexts.

**Fix:** Either add a sanitization pass before injection using a library like `DOMPurify`, or configure `marked` to strip HTML:
```ts
marked.use({ gfm: true, breaks: false })
// Add a sanitizer at the marked level (marked v5+):
// marked.use({ hooks: { postprocess(html) { return DOMPurify.sanitize(html) } } })
```
At minimum, add a comment at line 930 explaining the trust model so future maintainers understand why `dangerouslySetInnerHTML` is intentional here.

### IN-01: `update.rs` — `test_refresh_skips_*_when_current` tests use `std::thread::sleep` for mtime comparison, which is fragile on fast filesystems

**File:** `src/update.rs:447-461`, `src/update.rs:492-508`, `src/update.rs:540-556`
**Issue:** Three tests sleep for 10ms then check that the file's `modified()` timestamp did not change. On filesystems with 1-second mtime resolution (HFS+ on macOS in some configurations, older ext3) the sleep is too short to detect an unintended write if it happened. On very fast NVMe storage the sleep may not be needed at all. The approach works in practice but is fragile.

A more robust approach would be to check that the file content was not changed (version string is still the same value), rather than relying on mtime.

**Fix:**
```rust
// Instead of mtime comparison:
refresh_integrations_with_home(home, "1.2.3");
let content_after = std::fs::read_to_string(&manifest_path).unwrap();
let json_after: serde_json::Value = serde_json::from_str(&content_after).unwrap();
assert_eq!(json_after["version"].as_str(), Some("1.2.3"),
    "plugin.json should still have version 1.2.3 after refresh (no rewrite)");
```

### IN-02: `opencode_plugin.mjs` — `ctx` parameter accepted but never used

**File:** `src/integrations/opencode_plugin.mjs:12`
**Issue:** The default export function accepts `ctx` as a parameter (`export default (ctx) => { ... }`) but `ctx` is never referenced in the body. This is dead code that could mislead a reader into thinking the plugin uses context from OpenCode. If the OpenCode plugin API passes session context via this argument, ignoring it may be intentional for now, but it should be documented.

**Fix:** Either rename to `_ctx` to signal intentional non-use, or add a brief comment:
```js
// ctx: OpenCode plugin context (available for future use; not needed by this plugin)
export default (_ctx) => {
```

### IN-03: `App.tsx` — `computeAndApplyLayout` and `updateActiveHeading` are defined as plain functions inside the component body, not memoized, causing unnecessary recreation on every render

**File:** `ui/src/App.tsx:676-689`, `ui/src/App.tsx:722-761`
**Issue:** Both functions are recreated on every render because they are plain `function` declarations (not `useCallback`) inside the component. They read from refs (which are stable), so functional correctness is not affected. However, `computeAndApplyLayout` is passed to `window.addEventListener('resize', ...)` inside a `useEffect` with `[]` deps (line 777). Because `useEffect` captures the function at mount time, the listener always holds the initial function reference. Since the function reads from refs this is actually safe, but the pattern is easy to misread as a missing deps bug.

**Fix:** These are low priority given ref usage. If a future change converts any ref to state, `useCallback` would be required. A comment noting the ref-based design is sufficient:
```ts
// computeAndApplyLayout reads only refs (stable), so useCallback is not required.
function computeAndApplyLayout() { ... }
```
