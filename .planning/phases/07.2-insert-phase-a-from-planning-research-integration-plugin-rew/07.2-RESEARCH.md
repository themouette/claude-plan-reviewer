# Phase 07.2: Integration Plugin/Extension Infrastructure — Research

**Researched:** 2026-04-11
**Domain:** Rust filesystem operations, Claude Code plugin system, Gemini CLI extension system, version-aware update logic
**Confidence:** HIGH

---

## Summary

Phase 07.2 is a pure filesystem refactor. The current `claude.rs` and `gemini.rs` integrations write hook entries directly into the user's `settings.json` files. This phase moves to a plugin/extension model where hook config lives in files plan-reviewer owns, enabling a stable settings.json pointer and version-aware rewrites by `update`.

The architecture is fully specified in `.planning/research/INTEGRATION-PLUGIN-REWORK.md`. This research document fills in the implementation-level details the planner needs: exact file structures, verified API formats, idempotency patterns, the gap between the architecture proposal and the Claude Code plugin registration API, and what must change vs. what can stay the same.

**Critical finding:** The architecture doc describes "one minimal entry added once to `~/.claude/settings.json` pointing to the plugin directory." The actual Claude Code API for persistent local-plugin registration requires **two entries** in `settings.json`: an `extraKnownMarketplaces` entry (declaring the local directory as a marketplace source) and an `enabledPlugins` entry (enabling the plugin from that marketplace). plan-reviewer must write both. The idempotency key for future updates remains the `enabledPlugins` entry. See Architecture Patterns section for the exact JSON.

**Primary recommendation:** Implement three independent file groups per integration — Claude plugin dir, Gemini extension dir, opencode version comment — each with its own idempotency check and write logic. The `update` command then reads the version field from the manifest and rewrites stale files.

## Current State: What Exists

### Claude integration (`src/integrations/claude.rs`)

**Current behavior:**
- `install`: reads `~/.claude/settings.json`, upserts a `PermissionRequest` hook entry with `matcher: "ExitPlanMode"`, writes back
- `uninstall`: reads `~/.claude/settings.json`, removes all entries with `matcher: "ExitPlanMode"`, writes back
- `is_installed`: returns `true` if any `hooks.PermissionRequest[]` entry has `matcher: "ExitPlanMode"`
- Idempotency key: presence of `matcher: "ExitPlanMode"` in `hooks.PermissionRequest[]`
- Hook command stored: `binary_path` (full path to plan-reviewer binary)

**What changes in Phase 07.2:**
- `install` writes a plugin directory structure AND writes two entries to `~/.claude/settings.json` (marketplace registration + plugin enablement)
- `uninstall` removes the plugin directory AND removes those two settings.json entries
- Old behavior (bare PermissionRequest entry) must be left in place for Phase B migration

### Gemini integration (`src/integrations/gemini.rs`)

**Current behavior:**
- `install`: reads `~/.gemini/settings.json`, upserts a `BeforeTool` hook entry with `matcher: "exit_plan_mode"`, writes back
- Hook entry includes `name: "plan-reviewer"`, `timeout: 300000`
- Idempotency key: presence of a BeforeTool entry whose `hooks[].name == "plan-reviewer"`
- Hook command stored: `binary_path`

**What changes in Phase 07.2:**
- `install` writes `~/.gemini/extensions/plan-reviewer/` directory
- `uninstall` removes the extension directory
- settings.json is no longer touched by Gemini install/uninstall (auto-discovery replaces registration)

### OpenCode integration (`src/integrations/opencode.rs`)

**Current behavior:**
- Writes `OPENCODE_PLUGIN_SOURCE` (embedded `.mjs` file) to `~/.config/opencode/plugins/plan-reviewer-opencode.mjs` on install
- Adds plugin path to `opencode.json`'s `plugin` array
- Idempotency: plugin file is always rewritten; config entry uses path-string presence check

**What changes in Phase 07.2:**
- Add a version comment `// plan-reviewer-version: X.Y.Z` to the `.mjs` template
- `update` reads this comment to check version; rewrites file if stale
- No other changes to `opencode.rs` itself

### Update command (`src/update.rs`)

**Current behavior:**
- Downloads new binary via `self_update`, replaces in-place
- No integration-aware post-update steps

**What changes in Phase 07.2:**
- After binary replacement, detect installed integrations by manifest presence
- Compare manifest version to current binary version
- Rewrite stale plugin/extension files

### Main CLI (`src/main.rs`)

**No changes in Phase 07.2.** The `Commands` enum gains `Commands::Hook` in Phase B (07.3), not here.

## Architecture Patterns

### Claude Code Plugin Registration (VERIFIED)

[VERIFIED: code.claude.com/docs/en/plugins-reference, code.claude.com/docs/en/settings]

A Claude Code plugin is a directory with this structure:

```
<plugin-dir>/
├── .claude-plugin/
│   └── plugin.json        # manifest (name, version, description)
└── hooks/
    └── hooks.json         # hook configuration
```

`plugin.json` minimum schema:
```json
{
  "name": "plan-reviewer",
  "version": "0.2.0",
  "description": "Plan review hook for Claude Code"
}
```

`hooks/hooks.json` format (PermissionRequest + ExitPlanMode):
```json
{
  "hooks": {
    "PermissionRequest": [
      {
        "matcher": "ExitPlanMode",
        "hooks": [{ "type": "command", "command": "plan-reviewer" }]
      }
    ]
  }
}
```

**Registration in `~/.claude/settings.json` requires TWO entries:**

```json
{
  "extraKnownMarketplaces": {
    "plan-reviewer-local": {
      "source": {
        "source": "directory",
        "path": "/home/user/.local/share/plan-reviewer/claude-plugin"
      }
    }
  },
  "enabledPlugins": {
    "plan-reviewer@plan-reviewer-local": true
  }
}
```

The `directory` source type takes an absolute path to the directory containing `.claude-plugin/plugin.json`. The marketplace name (`plan-reviewer-local`) must match exactly in both `extraKnownMarketplaces` and `enabledPlugins`.

**Idempotency key for install:** presence of `enabledPlugins["plan-reviewer@plan-reviewer-local"]` in `~/.claude/settings.json`.

**Idempotency key for update:** `version` field in `plugin.json`.

**Note on `--plugin-dir` flag:** This is a development/testing flag for the `claude` CLI (`claude --plugin-dir ./my-plugin`). It is session-only and does not persist. plan-reviewer must write the marketplace+enabledPlugins entries for persistence. [VERIFIED: code.claude.com/docs/en/plugins]

**CRITICAL ARCHITECTURE NOTE:** The architecture doc proposes a simpler single-entry approach. The actual Claude Code API requires two entries. The planner must account for this in both `install` (write both entries) and `uninstall` (remove both entries). Idempotency check must test `enabledPlugins` key presence, not `extraKnownMarketplaces`.

### Gemini CLI Extension Registration (VERIFIED)

[VERIFIED: geminicli.com/docs/extensions/reference]

A Gemini CLI extension is a directory written to `~/.gemini/extensions/<name>/`:

```
~/.gemini/extensions/plan-reviewer/
├── gemini-extension.json    # manifest (name, version required)
└── hooks/
    └── hooks.json           # hook configuration
```

`gemini-extension.json` format:
```json
{
  "name": "plan-reviewer",
  "version": "0.2.0",
  "description": "Plan review hook for Gemini CLI"
}
```

`hooks/hooks.json` for Gemini CLI:
- Uses same `hooks.json` format as Claude Code hooks: [ASSUMED] Gemini CLI docs state "hooks are defined in `hooks/hooks.json` within the extension directory" and "note that hooks are not defined in the `gemini-extension.json` manifest," but the exact hook event names for Gemini extensions are not documented in the reference.
- The current `gemini.rs` uses `hooks.BeforeTool` with `matcher: "exit_plan_mode"` via settings.json.
- Whether extension `hooks/hooks.json` uses the same `BeforeTool` key is [ASSUMED].

**Registration:** Gemini CLI **auto-discovers** extensions in `~/.gemini/extensions/`. No separate settings.json entry required. [VERIFIED: geminicli.com/docs/extensions/reference]

**Idempotency key for install:** directory `~/.gemini/extensions/plan-reviewer/` exists with `gemini-extension.json`.

**Idempotency key for update:** `version` field in `gemini-extension.json`.

### OpenCode Version Comment Pattern (VERIFIED from codebase)

Current `opencode_plugin.mjs` starts with:
```javascript
// plan-reviewer-opencode.mjs
// Installed by plan-reviewer. Binary path injected at install time.
```

Phase 07.2 adds a version comment (example):

**Version extraction:** `update` reads the file and parses the `// plan-reviewer-version: X.Y.Z` line using a simple regex or string split.

### Recommended Plugin Directory Location

The architecture doc proposes `~/.local/share/plan-reviewer/claude-plugin/`. This is appropriate on Linux. On macOS the conventional XDG equivalent is `~/.local/share/` which is also fine. The path is derived as:

```rust
PathBuf::from(&ctx.home).join(".local/share/plan-reviewer/claude-plugin")
```

Gemini extension goes to `~/.gemini/extensions/plan-reviewer/` (fixed by the Gemini CLI spec).

### InstallContext Extension

`InstallContext` in `src/integrations/mod.rs` currently has:
```rust
pub struct InstallContext {
    pub home: String,
    pub binary_path: Option<String>,
}
```

No new fields are needed. The plugin directory paths are computed from `ctx.home`. The binary path (`binary_path`) is used for the hook command in the template files.

### Write Pattern (Atomic File Creation)

The existing integrations write files with `std::fs::write`. There is no atomic rename pattern (write to `.tmp` then rename). This is acceptable for this project — `std::fs::write` is idempotent when called twice on the same path and the risk of partial writes on the local filesystem is low for small JSON/MJS files. Follow the existing pattern.

**Directory creation:** `std::fs::create_dir_all(parent)` before writing any file in a new directory. See existing patterns in `claude.rs` (line 115-119) and `opencode.rs` (line 37-41).

### Idempotency Contract

For Claude:
1. Plugin directory files: always rewritten on install (overwrite is idempotent for identical content)
2. settings.json `extraKnownMarketplaces` entry: use `entry().or_insert_with()` pattern
3. settings.json `enabledPlugins` entry: presence check before insert

For Gemini:
1. Extension directory files: always rewritten on install
2. No settings.json involvement

For OpenCode:
1. Plugin file: always rewritten (existing behavior — no change)
2. Config entry: existing path-string presence check (no change)
3. Version comment: add to template (no behavior change to idempotency)

### Uninstall Patterns

**Claude uninstall (Phase 07.2):**
- Remove `~/.local/share/plan-reviewer/claude-plugin/` directory recursively (`std::fs::remove_dir_all`)
- Remove `extraKnownMarketplaces["plan-reviewer-local"]` from `~/.claude/settings.json`
- Remove `enabledPlugins["plan-reviewer@plan-reviewer-local"]` from `~/.claude/settings.json`
- Do NOT touch the old-style `hooks.PermissionRequest` entry (that is Phase B's migration concern)
- Idempotency: each step checks existence before acting

**Gemini uninstall (Phase 07.2):**
- Remove `~/.gemini/extensions/plan-reviewer/` directory recursively
- Idempotency: check `path.exists()` before `remove_dir_all`

### Update Logic

```rust
// Pseudocode for post-update integration refresh
fn refresh_integrations_after_update(home: &str, current_version: &str) {
    // Claude
    let manifest_path = claude_plugin_dir(home).join(".claude-plugin/plugin.json");
    if manifest_path.exists() {
        let installed_version = read_version_from_json(&manifest_path);
        if installed_version != current_version {
            write_claude_plugin_files(home, current_version);
        }
    }

    // Gemini
    let manifest_path = gemini_extension_dir(home).join("gemini-extension.json");
    if manifest_path.exists() {
        let installed_version = read_version_from_json(&manifest_path);
        if installed_version != current_version {
            write_gemini_extension_files(home, current_version);
        }
    }

    // OpenCode
    let plugin_path = opencode_plugin_path(home);
    if plugin_path.exists() {
        let installed_version = read_version_from_mjs_comment(&plugin_path);
        if installed_version != current_version {
            write_opencode_plugin_file(home, binary_path, current_version);
        }
    }
}
```

The current binary version is always available via `self_update::cargo_crate_version!()` (already used in `update.rs`).

## Common Pitfalls

### Pitfall 1: Two-entry settings.json registration for Claude plugins

**What goes wrong:** Writing only `extraKnownMarketplaces` without `enabledPlugins`, or vice versa. Claude Code will show the plugin in the marketplace but not load it, or load it but not persist across restarts.

**Why it happens:** The architecture doc describes "one minimal entry." The actual API needs two.

**How to avoid:** Always write and remove both `extraKnownMarketplaces["plan-reviewer-local"]` and `enabledPlugins["plan-reviewer@plan-reviewer-local"]` together.

**Warning signs:** Integration test shows settings.json was written but Claude Code does not load the hook.

### Pitfall 2: Path value in `extraKnownMarketplaces.source.path` is not the plugin root

**What goes wrong:** Writing the path to `.claude-plugin/` subdirectory instead of the parent plugin directory.

**Why it happens:** The `source.path` for `"source": "directory"` must point to the directory that CONTAINS `.claude-plugin/plugin.json`, not to `.claude-plugin/` itself.

**How to avoid:** The `path` value is `~/.local/share/plan-reviewer/claude-plugin`, not `~/.local/share/plan-reviewer/claude-plugin/.claude-plugin`.

**Warning signs:** Claude Code shows "plugin directory not found" error.

### Pitfall 3: Gemini hooks/hooks.json event name

**What goes wrong:** Using `BeforeTool` (which is what the current `gemini.rs` writes to settings.json) in extension `hooks/hooks.json` without confirming that's the right event name for the extension system.

**Why it happens:** Gemini CLI extension reference does not document hook events explicitly. The current settings.json format uses `BeforeTool`, but extension hooks.json may use different names.

**How to avoid:** This is flagged as ASSUMED in the Assumptions Log. The planner should note that the Gemini extension `hooks.json` format needs confirmation before implementation. A fallback is to use the same `BeforeTool` format that currently works in settings.json.

**Warning signs:** Gemini CLI loads the extension but the hook does not fire on `exit_plan_mode`.

### Pitfall 4: Forgetting the old-style settings.json entry during uninstall

**What goes wrong:** Phase 07.2 uninstall removes the plugin directory but leaves both the new plugin registration entries AND the old bare hook entry (if user was on the old install).

**Why it happens:** Phase 07.2 does not handle migration from old installs (that's Phase B / 07.3). But uninstall should still clean up the new plugin entries.

**How to avoid:** The Phase 07.2 uninstall removes the new entries only. The old entries are left for Phase B to handle. Document this explicitly in code comments.

### Pitfall 5: Not computing binary_path for plugin template

**What goes wrong:** Writing `hooks.json` with a hardcoded `"plan-reviewer"` command that assumes the binary is in PATH, when `install` always has the actual `binary_path` available.

**Why it happens:** The architecture doc shows `"command": "plan-reviewer"` in hooks.json (bare name). This works if plan-reviewer is in PATH but fails otherwise.

**Resolution:** Phase 07.2 should write the bare `"plan-reviewer"` command in hooks.json (consistent with the architecture doc, and because Phase B changes it to `"plan-reviewer hook"`). The `install` command should verify PATH when writing bare names. This aligns with the architecture doc constraint: "install should verify PATH."

### Pitfall 6: remove_dir_all on non-existent path panics

**What goes wrong:** Calling `std::fs::remove_dir_all` on a path that doesn't exist returns an error.

**How to avoid:** Check `path.exists()` before calling `remove_dir_all`. Or handle the `ErrorKind::NotFound` case.

## Open Questions

1. **Gemini extension hooks.json event name**
   - What we know: Current settings.json format uses `hooks.BeforeTool[].matcher: "exit_plan_mode"` and this works
   - What's unclear: Whether extension `hooks/hooks.json` uses `BeforeTool` or a different event name
   - Recommendation: Use the same `BeforeTool` + `exit_plan_mode` format that currently works in settings.json. Gemini CLI's extension reference says "hooks are not defined in `gemini-extension.json`" and points to `hooks/hooks.json`, suggesting the format is the same as user-level hooks.

2. **Timeout in Gemini extension hooks.json**
   - What we know: The current settings.json entry includes `timeout: 300000` (5 minutes, from Phase 6 research)
   - What's unclear: Whether the extension `hooks/hooks.json` format supports `timeout` at the hook entry level
   - Recommendation: Include `timeout: 300000` in the extension hooks.json; it is unlikely to cause harm if unsupported (fields ignored gracefully).

3. **PATH verification in install**
   - What we know: Architecture doc says "`install` should verify PATH"
   - What's unclear: What action to take if `plan-reviewer` is not in PATH — error or warning?
   - Recommendation: Print a warning to stderr but continue install. The user may be setting up in a non-PATH location temporarily, or the binary may not be in the shell PATH at install time.
