Reviewed `src/main.rs`, `src/integrations/claude.rs`, `src/integrations/gemini.rs`, and `src/update.rs`. The core logic is sound and well-tested. ## Warnings

### WR-01: Silent settings.json clobber on serialization failure in `perform_claude_migration`

**File:** `src/update.rs:243`
**Issue:** `serde_json::to_string_pretty(&root).unwrap_or_default()` returns an empty string on serialization failure. The empty string is then passed directly to `std::fs::write`, which silently overwrites `~/.claude/settings.json` with an empty file, destroying all user settings. While `serde_json` serialization of a `serde_json::Value` rarely fails in practice, the defensive path destroys data.

**Fix:**
```rust
let output = match serde_json::to_string_pretty(&root) {
    Ok(s) => s,
    Err(e) => {
        eprintln!("plan-reviewer: cannot serialize settings.json: {} (no changes written)", e);
        return;
    }
};
let _ = std::fs::write(&settings_path, output);
```

### WR-02: Silent settings.json clobber on serialization failure in `perform_gemini_migration`

**File:** `src/update.rs:281`

**Fix:** Same as WR-01 — return early with an `eprintln!` instead of using `unwrap_or_default()`.

### WR-03: Silent write failures in `write_claude_plugin_files`, `write_gemini_extension_files`, `write_opencode_plugin_file`

**File:** `src/update.rs:419-427`, `src/update.rs:463-470`, `src/update.rs:491`
**Issue:** All three file-writing helpers use `let _ = std::fs::write(...)`, silently discarding I/O errors. If the write fails (e.g., disk full, permissions), the integration files remain stale, the caller receives no signal, and the update is reported as successful when the plugin files were not actually refreshed.

**Fix:** Propagate errors to the caller by returning `Result<(), String>`:
```rust
fn write_claude_plugin_files(home: &str, current_version: &str) -> Result<(), String> {
    // ...
    std::fs::write(manifest_dir.join("plugin.json"), ...)
        .map_err(|e| format!("cannot write plugin.json: {}", e))?;
    // ...
    Ok(())
}
```
Callers in `refresh_integrations_with_home` can then log the error with `eprintln!` and continue with the remaining integrations.

### WR-04: Watchdog task in `async_main` never executes — dead code with misleading comment

**File:** `src/main.rs:646-649`
**Issue:** A tokio task is spawned at the end of `async_main` to call `std::process::exit(0)` after 3 seconds. However, the runtime is a `current_thread` runtime and `async_main` is called via `rt.block_on(...)`. Once `block_on` returns, the runtime is never polled again before `rt` is dropped at the end of `run_hook_flow` / `run_opencode_flow`. The spawned task is therefore cancelled when the runtime drops — the `std::process::exit(0)` call never fires. The comment claims this is a clean-exit watchdog ("HOOK-04: after stdout write completes"), but it has no effect.

**Fix:** If the intent is to ensure the process exits after the hook stdout write, either:
1. Remove the watchdog entirely — `main` returns normally and the OS cleans up.
2. Move the watchdog spawn before `block_on`, or drive the runtime until the task completes:
```rust
// In run_hook_flow, after serde_json::to_writer(stdout, &output_json):
// The write is already done; just return and let main() exit naturally.
// Remove the tokio::spawn block from async_main entirely.
```

### IN-01: `_binary_path` guard checks presence but value is never used

**File:** `src/integrations/claude.rs:36-40`, `src/integrations/gemini.rs:22-25`
**Issue:** Both `install` implementations extract `binary_path` into `_binary_path` solely to return an error if it is `None`, but the path value itself is never embedded into any generated file. The hook command uses the bare string `"plan-reviewer review-hook"` (PATH-based lookup). This is intentional by design, but the guard creates a misleading API contract — callers who provide the path expecting it to be recorded will be surprised.

**Fix:** Either document explicitly that `binary_path` is required as a guard only (not embedded), or rename the field/parameter to make the intent clear. A doc comment on `InstallContext.binary_path` would suffice:
```rust
/// Required as a presence guard for install operations.
/// The actual value is not embedded in generated files —
/// hooks use PATH-based binary lookup ("plan-reviewer").
pub binary_path: Option<String>,
```

### IN-02: `gemini-extension.json` written as compact JSON; `plugin.json` written pretty-printed

**File:** `src/integrations/gemini.rs:36-41`
**Issue:** The Gemini manifest is written using a hand-crafted `format!` string (compact, single-line JSON), while the Claude `plugin.json` uses `serde_json::to_string_pretty`. The inconsistency makes the Gemini manifest harder to diff and inspect. Not a functional bug.

**Fix:** Use `serde_json::to_string_pretty` consistently:
```rust
let manifest = serde_json::json!({
    "name": "plan-reviewer",
    "version": crate_version(),
    "description": "Plan review hook for Gemini CLI"
});
let manifest_str = serde_json::to_string_pretty(&manifest)
    .map_err(|e| format!("cannot serialize gemini-extension.json: {}", e))?;
std::fs::write(&manifest_path, manifest_str)
    .map_err(|e| format!("cannot write {}: {}", manifest_path.display(), e))?;
```

### IN-03: Fragile string-match for permission error detection in `perform_update`

**File:** `src/update.rs:105-109`
**Issue:** Permission errors are detected by checking if the error message string contains `"Permission denied"` or `"EACCES"`. This pattern is fragile: it is locale-dependent on some platforms, and the error string format could change between `self_update` crate versions. Non-English OS locales may produce different error text.

**Fix:** Check the error kind where possible. As a minimum improvement, add the Windows equivalent:
```rust
if msg.contains("Permission denied") || msg.contains("EACCES") || msg.contains("Access is denied") {
```
A more robust long-term fix would be to check `std::io::Error::kind() == ErrorKind::PermissionDenied` if the `self_update` crate exposes the underlying IO error.
