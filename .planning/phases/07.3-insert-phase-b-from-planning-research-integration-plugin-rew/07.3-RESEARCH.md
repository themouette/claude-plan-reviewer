# Phase 07.3: Hook Subcommand — Research

**Researched:** 2026-04-11
**Domain:** Rust CLI subcommand dispatch (clap), deprecation warning pattern, backward-compatible migration
**Confidence:** HIGH

---

## Summary

Phase 07.3 is a focused CLI refactor. The bare `plan-reviewer` invocation (reads stdin JSON as a hook) becomes the explicit subcommand `plan-reviewer hook`. The old bare invocation remains functional with a stderr deprecation warning for at least one major version. Plugin templates in `claude.rs` and `gemini.rs` update the hook command from `"plan-reviewer"` to `"plan-reviewer hook"`. The `update` command's post-update integration refresh (delivered in Phase 07.2 Plan 03) is extended with a Case 2 migration path for pre-plugin users.

This phase is entirely internal to the Rust codebase — no new crates, no external APIs, no config file format changes. The change surface is:
1. `src/main.rs` — add `Commands::Hook`, route `None` arm to deprecation-warned flow
2. `src/integrations/claude.rs` — update hooks.json template command string
3. `src/integrations/gemini.rs` — update hooks.json template command string
4. `src/update.rs` — extend `refresh_integrations_with_home` with Case 2 migration (old bare settings.json entry → new plugin model + `hook` subcommand)

**Primary recommendation:** Add `Commands::Hook` to the `Commands` enum in `main.rs`, reuse the existing `run_hook_flow` function body for its implementation, and add a one-line `eprintln!` deprecation warning in the `None` arm before calling the same body. Then update the two template strings in `claude.rs` and `gemini.rs`.

## Architecture Patterns

### Pattern 1: Adding a Subcommand to an Existing `Commands` Enum (clap derive)

**What:** Add `Hook` variant to the existing `Commands` enum in `src/main.rs`.

**Current enum (before Phase 07.3):**
```rust
// Source: src/main.rs lines 274-297 [VERIFIED: codebase read]
#[derive(Subcommand, Debug)]
enum Commands {
    Install { integrations: Vec<String> },
    Uninstall { integrations: Vec<String> },
    Update { check: bool, version: Option<String>, yes: bool },
}
```

**After Phase 07.3:**
```rust
#[derive(Subcommand, Debug)]
enum Commands {
    /// Receive a hook event from stdin and open the browser review UI.
    ///
    /// This is the explicit subcommand form. The bare `plan-reviewer` invocation
    /// still works but emits a deprecation warning on stderr.
    Hook,
    Install { integrations: Vec<String> },
    Uninstall { integrations: Vec<String> },
    Update { check: bool, version: Option<String>, yes: bool },
}
```

**No flags on `Hook`:** The `--no-browser` and `--port` global flags on `Cli` remain global — they already apply to any subcommand. The `Hook` variant carries no fields.

**Why no fields:** The hook flow receives all input from stdin and the global `--no-browser`/`--port` flags. No per-subcommand arguments are needed.

### Pattern 2: Main Dispatch — Hook arm and Deprecated None arm

**Current `None` arm (before Phase 07.3):**
```rust
// Source: src/main.rs lines 466-474 [VERIFIED: codebase read]
None => {
    if let Some(ref plan_file) = cli.plan_file {
        run_opencode_flow(cli.no_browser, cli.port, plan_file);
    } else {
        run_hook_flow(cli.no_browser, cli.port);
    }
}
```

**After Phase 07.3:**
```rust
Some(Commands::Hook) => {
    // Explicit hook subcommand — no deprecation warning
    run_hook_flow(cli.no_browser, cli.port);
}
None => {
    if let Some(ref plan_file) = cli.plan_file {
        // opencode uses --plan-file flag directly (no subcommand)
        run_opencode_flow(cli.no_browser, cli.port, plan_file);
    } else {
        // Deprecated: bare plan-reviewer invocation. Will be removed in v1.0.
        eprintln!(
            "plan-reviewer: deprecation warning: invoking plan-reviewer without \
             the 'hook' subcommand is deprecated. Use 'plan-reviewer hook' instead."
        );
        run_hook_flow(cli.no_browser, cli.port);
    }
}
```

**Key insight:** The `--plan-file` path (opencode flow) does NOT get a deprecation warning. It is a legitimate non-hook invocation. Only the bare stdin-reading hook flow needs the warning.

**`run_hook_flow` function:** No changes needed — it already reads stdin JSON, parses HookInput, and dispatches to Claude/Gemini output formats. `Commands::Hook` simply routes to it without the deprecation message.

### Pattern 3: Updating Hook Command in Plugin Template Files

**Claude hooks.json template (in `claude.rs` after Phase 07.2):**
```json
{"hooks":{"PermissionRequest":[{"matcher":"ExitPlanMode","hooks":[{"type":"command","command":"plan-reviewer"}]}]}}
```

**Change for Phase 07.3:**
```json
{"hooks":{"PermissionRequest":[{"matcher":"ExitPlanMode","hooks":[{"type":"command","command":"plan-reviewer hook"}]}]}}
```

**Gemini hooks.json template (in `gemini.rs` after Phase 07.2):**
```json
{"hooks":{"BeforeTool":[{"matcher":"exit_plan_mode","hooks":[{"name":"plan-reviewer","type":"command","command":"plan-reviewer","timeout":300000}]}]}}
```

**Change for Phase 07.3:**
```json
{"hooks":{"BeforeTool":[{"matcher":"exit_plan_mode","hooks":[{"name":"plan-reviewer","type":"command","command":"plan-reviewer hook","timeout":300000}]}]}}
```

These changes affect BOTH the `install()` method (which writes the hooks.json file) AND the `write_claude_plugin_files` / `write_gemini_extension_files` functions in `update.rs` (which rewrite stale files post-update). Both locations must be updated in sync.

### Pattern 4: Case 2 Migration in `update.rs`

The INTEGRATION-PLUGIN-REWORK.md defines three migration cases for `update`:

- **Case 1 (fresh install):** Normal path — no migration needed.
- **Case 2 (pre-plugin install):** User has OLD-style bare hook entry in settings.json but NO plugin manifest. This user was installed before Phase 07.2 shipped. On `update`, we must:
  1. Write the plugin/extension directory (same as `install`)
  2. Write settings.json registration entries (same as `install`)
  3. Remove the old bare hook entry from settings.json (the entry that `claude_legacy_hook_installed` detects)
- **Case 3 (stale plugin):** Plugin exists, version field is stale — `refresh_integrations` already handles this via `write_*_plugin_files`.

[ASSUMED] Phase 07.2's `refresh_integrations_with_home` only covers Case 3 (version-stale rewrite). Case 2 must be added in Phase 07.3.

**Case 2 detection logic:**
```rust
// Claude Case 2: old bare entry exists, no plugin manifest
let claude_manifest = claude_plugin_dir(&home).join(".claude-plugin/plugin.json");
if !claude_manifest.exists() {
    // Check for old-style hook entry in settings.json
    let settings_path = PathBuf::from(&home).join(".claude/settings.json");
    if settings_path.exists() {
        if let Ok(content) = std::fs::read_to_string(&settings_path) {
            if let Ok(json) = serde_json::from_str::<serde_json::Value>(&content) {
                if claude_legacy_hook_installed(&json) {
                    // Pre-plugin era user — perform migration
                    perform_claude_migration(&home, current_version);
                }
            }
        }
    }
}
```

**`perform_claude_migration`** logic:
1. Call `write_claude_plugin_files(home, current_version)` (already writes plugin dir)
2. Write `extraKnownMarketplaces` + `enabledPlugins` registration to settings.json (same as `install`)
3. Remove old `hooks.PermissionRequest` ExitPlanMode entry from settings.json (same as old `uninstall`)
4. Print migration summary message to stdout

**Gemini Case 2 detection:** Check if extension dir absent AND old-style `~/.gemini/settings.json` has `BeforeTool` entry with `name: "plan-reviewer"`.

## Common Pitfalls

### Pitfall 1: Deprecation Warning Goes to Stdout

**What goes wrong:** Writing `println!` instead of `eprintln!` for the deprecation warning. Plan-reviewer's stdout is the hook JSON output — Claude Code parses it. Any extra text on stdout breaks the hook protocol.

**Why it happens:** Easy to confuse in a binary that prints status to stdout in other commands.

**How to avoid:** The deprecation warning MUST use `eprintln!`. All diagnostic and status text in the hook flow already uses `eprintln!`. The only `println!` in `run_hook_flow` is... none — the final stdout write uses `serde_json::to_writer(std::io::stdout(), ...)`. No change needed there.

**Warning signs:** Claude Code returns a JSON parse error or unexpected output on the next plan approval.

### Pitfall 2: Forgetting to Update `write_claude_plugin_files` and `write_gemini_extension_files` in update.rs

**What goes wrong:** Only updating the `install()` method in `claude.rs` / `gemini.rs` to use `"plan-reviewer hook"`, but NOT updating the equivalent write functions in `update.rs`. Users who update via `plan-reviewer update` would get a stale hooks.json with the old `"plan-reviewer"` bare command.

**Why it happens:** There are two code paths that write hooks.json: `install()` (direct) and `write_*_files()` in update.rs (post-update refresh). They must stay in sync.

**How to avoid:** Update ALL four write locations:
1. `claude.rs` `install()` → writes hooks.json
2. `gemini.rs` `install()` → writes hooks.json
3. `update.rs` `write_claude_plugin_files()` → writes hooks.json
4. `update.rs` `write_gemini_extension_files()` → writes hooks.json

### Pitfall 3: The `--plan-file` None Arm Incorrectly Gets a Deprecation Warning

**What goes wrong:** Adding the deprecation warning to the entire `None` arm, which also handles `--plan-file` (opencode flow). OpenCode's JS plugin legitimately calls `plan-reviewer --plan-file /path/to/plan.md` — this is not a deprecated invocation.

**How to avoid:** The deprecation warning belongs ONLY in the `else` branch when `cli.plan_file.is_none()`. The code structure is `if let Some(plan_file) = cli.plan_file { opencode } else { deprecated + hook_flow }`.

**Warning signs:** OpenCode plugin stops working after Phase 07.3 ships because stderr output confuses the plugin's response parsing (if it checks stderr).

### Pitfall 4: Case 2 Migration Writes Plugin Files With Old Command in hooks.json

**What goes wrong:** The `perform_claude_migration` function calls `write_claude_plugin_files` which writes the hooks.json with `"plan-reviewer hook"` (the new command, as updated in Pitfall 2's fix). But if the `write_*` functions in update.rs are NOT updated to use `"plan-reviewer hook"`, Case 2 migration writes an immediately-stale hooks.json.

**How to avoid:** Fix all four write locations (Pitfall 2) BEFORE implementing Case 2 migration. Phase 07.3 plans should sequence: update command strings → update write functions → implement Case 2.

### Pitfall 5: Case 2 Migration for Gemini Lacks `gemini_legacy_hook_installed` Access

**What goes wrong:** `update.rs` needs to call `gemini_legacy_hook_installed()` to detect old-style Gemini settings.json entries. If Phase 07.2 Plan 02 did not make this function `pub(crate)`, `update.rs` cannot import it.

**How to avoid:** Verify `gemini_legacy_hook_installed` is `pub(crate)` in `gemini.rs` (per Phase 07.2 Plan 02 spec). If not, make it `pub(crate)` as part of this phase's cleanup.

**Similarly for Claude:** `claude_legacy_hook_installed` must be `pub(crate)` in `claude.rs`.

### Pitfall 6: `Cli::plan_file` Flag Position Conflict With `hook` Subcommand

**What goes wrong:** clap may reject `--plan-file` when used together with a subcommand like `hook` because `--plan-file` is a top-level `Cli` flag and the `hook` variant takes no args. This is actually FINE — `--plan-file` is a global `Cli` flag, not a subcommand-specific flag, so clap allows it with any subcommand.

**Verification:** `plan-reviewer --plan-file /tmp/plan.md hook` should NOT be a valid invocation (opencode doesn't use the `hook` subcommand). The `None` arm with `plan_file` remains the opencode path. The `Some(Commands::Hook)` arm deliberately ignores `cli.plan_file`.

**How to avoid:** Do NOT forward `cli.plan_file` when routing to `Commands::Hook`. The `hook` subcommand always reads from stdin only.

### `perform_claude_migration` Helper

```rust
/// Migrate a pre-plugin Claude install to the plugin model.
///
/// Called when settings.json has an old bare ExitPlanMode entry but no plugin manifest.
/// After this: plugin dir exists, settings.json has two registration entries, old bare
/// entry is removed.
fn perform_claude_migration(home: &str, current_version: &str) {
    use crate::integrations::claude::claude_plugin_dir;

    // 1. Write plugin directory files (with "plan-reviewer hook" command)
    write_claude_plugin_files(home, current_version);

    // 2. Write settings.json registration entries
    let settings_path = std::path::PathBuf::from(home).join(".claude/settings.json");
    if let Ok(content) = std::fs::read_to_string(&settings_path) {
        if let Ok(mut root) = serde_json::from_str::<serde_json::Value>(&content) {
            let plugin_dir = claude_plugin_dir(home);

            // Add extraKnownMarketplaces entry
            root.as_object_mut()
                .unwrap()
                .entry("extraKnownMarketplaces")
                .or_insert_with(|| serde_json::json!({}));
            root["extraKnownMarketplaces"]
                .as_object_mut()
                .unwrap()
                .entry("plan-reviewer-local")
                .or_insert_with(|| serde_json::json!({
                    "source": {
                        "source": "directory",
                        "path": plugin_dir.to_string_lossy()
                    }
                }));

            // Add enabledPlugins entry
            root.as_object_mut()
                .unwrap()
                .entry("enabledPlugins")
                .or_insert_with(|| serde_json::json!({}));
            root["enabledPlugins"]
                .as_object_mut()
                .unwrap()
                .entry("plan-reviewer@plan-reviewer-local")
                .or_insert(serde_json::Value::Bool(true));

            // Remove old bare ExitPlanMode entry
            if let Some(arr) = root["hooks"]["PermissionRequest"].as_array_mut() {
                arr.retain(|e| e["matcher"].as_str() != Some("ExitPlanMode"));
            }

            let _ = std::fs::write(
                &settings_path,
                serde_json::to_string_pretty(&root).unwrap_or_default(),
            );
        }
    }
    println!("plan-reviewer: Claude integration migrated to plugin model");
}
```
