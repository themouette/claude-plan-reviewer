## Spacing Scale

Declared values (all multiples of 4):

| Token | Value | Usage in this phase |
|-------|-------|---------------------|
| xs | 4px | (not used) |
| sm | 8px | Vertical gap between the two banner copy lines |
| md | 16px | Banner vertical padding (top + bottom) |
| lg | 24px | (not used) |
| xl | 32px | Banner horizontal padding (matches `PageHeader` `padding: '0 32px'` at App.tsx:47) |
| 2xl | 48px | (not used; PageHeader height baseline only) |
| 3xl | 64px | (not used) |

**Spacing decisions for this phase:**

- Banner internal padding: `16px 32px` (vertical 16px, horizontal 32px). Horizontal value matches `PageHeader` so the banner copy left-edge aligns with the page header label "Plan Review".
- Banner sits in **normal flow**, NOT `position: sticky`. Rationale: `PageHeader` already owns sticky-top (`position: 'sticky', top: 0, zIndex: 10` at App.tsx:48-50). Stacking a second sticky element either requires nested z-index management or a sticky `top: 48px` calculation that breaks if the header height ever changes. Normal flow is correct because (a) the banner only appears while offline (a low-frequency state), (b) success criterion 5 requires the annotation UI remain fully interactive — a banner that scrolls away with the page does not occlude any sidebar/comment affordance.
- Margin between banner and `<PageHeader>`: **0px** (banner sits flush against the header bottom border). The header already has `borderBottom: '1px solid var(--color-border)'` (App.tsx:43), which provides the visual separator.
- Margin between banner and content beneath: **0px** (banner has no bottom margin; the content area beneath — `LoadingSpinner` / `ErrorView` / two-column review / `ConfirmationView` — already provides its own top padding, e.g. `padding: '32px'` at App.tsx:122 for ErrorView).

Exceptions: none.

## Typography

The banner is the only new typographic surface in this phase. The button labels reuse the existing button typography unchanged (only the displayed string changes).

| Role | Size | Weight | Line Height | Phase 14 usage |
|------|------|--------|-------------|----------------|
| Body | 14px | 400 (regular) | 1.5 | Banner copy (both lines) |
| Heading | 16px | 600 (semibold) | 1.5 | (not used in this phase) |
| Display | 20px | 600 (semibold) | 1.2 | (not used in this phase) |

**Typography decisions:**

- Banner copy is rendered at **14px / weight 400 / line-height 1.5** — the project's secondary body scale (matches `App.tsx:1142` review-content-secondary copy and `index.css:108` `.plan-prose code`/table secondary copy). Rationale for not using 16px body: the banner is a status surface, not the primary reading layer, and 14px keeps it visually subordinate to the review prose while remaining well above the 12px floor.
- Two lines of copy at `line-height: 1.5` produce 21px line boxes; with the 8px gap between lines (rendered as `<div>` + `<div>` with `marginTop: 8px` on the second line, OR a single `<p>` with `<br />` separator and a containing flex column with `gap: 8px`), the total banner copy block height is `21 + 8 + 21 = 50px`. With 16px top + 16px bottom padding the banner is 82px tall — acceptable; no taller than the page header plus one more standard row.
- Font family is inherited from `body` (system stack); no banner-specific font override.
- Text color: white-on-amber (dark theme) and dark-on-amber (light theme) — see Color section. NO underline, NO italic, NO uppercase transformation. Plain sentence-case copy as authored.
- Both lines share the same size and weight. The banner does NOT use a heading-then-subhead hierarchy; it is two equally-weighted status sentences.

**Total declared sizes:** 3 (14, 16, 20). **Total declared weights:** 2 (400, 600). Within the 3-4 sizes / 2 weights budget.

The banner introduces a new amber surface tier (semantic: connectivity-warning). The 60/30/10 split below describes the **app surface** before this phase's banner is added; the banner is treated as a **temporary semantic overlay**, not a permanent accent.

### Existing Surface Palette (carried over, not modified)

| Role | Dark theme | Light theme | Usage |
|------|------------|-------------|-------|
| Dominant (60%) | `#0f1117` (`--color-bg`) | `#f8fafc` (`--color-bg`) | Page background, content area |
| Secondary (30%) | `#1a1d27` (`--color-surface`) | `#f1f5f9` (`--color-surface`) | Page header, sidebar cards, surface containers |
| Accent — approve (10%) | `#22c55e` (`--color-accent-approve`) | `#16a34a` | Primary CTA — Approve button only |

**Accent reserved for:** Approve button background, Deny button background, focus rings. The new banner color does NOT join this 10% accent budget — it is a transient semantic surface, present only while offline, and on average covers ~0% of viewport time.

### New Banner Surface Tokens (Phase 14)

| Role | Dark theme | Light theme | Usage |
|------|------------|-------------|-------|
| Connectivity warning bg | `#f59e0b` (amber-500) | `#d97706` (amber-600) | Banner background only |
| Connectivity warning text | `#0f1117` (matches dark `--color-bg`) | **`#0f172a`** (matches light `--color-text-primary`) | Banner copy only |

**CSS variable additions** (in `ui/src/index.css`):

```css
:root {
  /* ...existing dark vars... */
  --color-banner-bg: #f59e0b;
  --color-banner-text: #0f1117;
}

[data-theme="light"] {
  /* ...existing light vars... */
  --color-banner-bg: #d97706;
  --color-banner-text: #0f172a;
}
```

The banner element references these as `background: var(--color-banner-bg); color: var(--color-banner-text);`.

### Contrast Verification (WCAG)

Computed via the WCAG relative-luminance formula on the chosen pairs:

| Theme | Background | Text | Ratio | AA normal (4.5) | AA large (3.0) | AAA normal (7.0) |
|-------|-----------|------|-------|-----------------|----------------|------------------|
| Dark  | `#f59e0b` | `#0f1117` | **8.79:1** | PASS | PASS | PASS |
| Light | `#d97706` | `#0f172a` | **5.60:1** | PASS | PASS | FAIL (5.60 < 7.0) |

**Override rationale (light-theme text color):** CONTEXT.md D-08 originally specified light-theme text as `#f8fafc` (matching `--color-bg`). Direct verification of that pair against `#d97706` yields **3.04:1**, which fails WCAG AA for normal text (4.5:1 required) and only passes AA for "large text" (3.0:1) — banner copy at 14px is NOT large text under WCAG (large = 18px regular OR 14px bold). The author intent stated in D-08 itself ("matches `--color-bg` for inversion / AA contrast on amber") is contradicted by the math; this UI-SPEC honors the **intent** (AA contrast) over the **literal value**, replacing `#f8fafc` with the project's existing `--color-text-primary` light value `#0f172a`. This change does not introduce a new color token (the value already exists in the codebase at `index.css:32`), and it produces the inverse-contrast pattern the user described — light-theme banner gets dark text on amber, dark-theme banner gets dark text on brighter amber. The dark-theme pair (`#f59e0b` / `#0f1117`) is unchanged from D-08 and passes AAA.

**If the planner wants strict adherence to D-08's literal `#f8fafc` text:** flag this as an accessibility BLOCKER and either (a) darken the light-theme bg to `#92400e` (amber-800), which produces 6.78:1 against `#f8fafc` and passes AA easily, or (b) accept the failure with explicit user sign-off. The recommended path is the override above — minimum-surprise, AA-compliant, no new tokens.

**Border / outline:** none. The banner is solid-fill, no border. Visual separation from `<PageHeader>` is provided by the header's existing `borderBottom: '1px solid var(--color-border)'` (App.tsx:43).

## Layout & Interaction Contract

This section is Phase 14-specific (not in the standard template) because the design contract is dominated by layout placement and the no-blocking interaction rule.

### Banner Placement (DOM order)

```
<App>
  <PageHeader />                                     ← App.tsx line 1075 today
  {connectivity === 'offline' && <OfflineBanner />}  ← NEW: insert here, line 1076-ish
  {appState === 'loading' && <LoadingSpinner />}     ← App.tsx line 1078 today
  {appState === 'error' && <ErrorView />}            ← App.tsx line 1080 today
  {appState === 'reviewing' && <Two-column layout>}  ← App.tsx line 1083 today
  {appState === 'confirmed' && <ConfirmationView />} ← App.tsx line ~1175 today
</App>
```

The banner is rendered **between** `PageHeader` and the appState-conditional block. It does NOT live inside the appState branches. Its visibility is purely a function of `connectivity === 'offline'` (D-10).

### Banner Layout

- Element: `<div role="status">`. NOT `<aside>`, NOT `<section>`, NOT `<header>`. A plain div with the ARIA live-region role is the correct semantic for a polite, non-interrupting status (D-09).
- Width: full bleed (`width: 100%`). The banner spans the entire viewport width, edge to edge.
- Display: `flex; flex-direction: column; gap: 8px` — produces the 8px vertical gap between the two copy lines without needing margins on individual children.
- Alignment: `align-items: flex-start` (text left-aligned). The leading edge sits at `padding-left: 32px`, aligning with the `PageHeader` "Plan Review" label.
- Height: intrinsic (driven by content). Approximately 82px at 14px / 1.5 / two lines / 16px+16px vertical padding.
- No `position: sticky`. No `position: fixed`. Normal flow only.
- No `box-shadow`, no `border`, no `border-radius` (full-bleed solid bar).
- No icon, no close affordance, no link, no button (D-07, OFX-01).

### Tab Order & Focus

- The banner contains **zero interactive elements**. It is text-only. It MUST NOT participate in keyboard tab order. Since `<div role="status">` is non-focusable by default and contains no `<a>`, `<button>`, or `[tabindex]`, this is automatic. Do NOT add `tabindex="0"` to the banner.
- The existing tab order through the page (theme toggle → tab bar → annotation textarea → approve → deny → etc.) is unchanged in either online or offline state.
- Focus styles on the relabeled buttons are unchanged — they continue to use `outline: '2px solid var(--color-focus)'` from existing handlers (App.tsx pattern at line 79).

### Live-Region Behavior

- `role="status"` implies `aria-live="polite"` and `aria-atomic="true"` by default. Both lines of copy are inside the same banner element, so when the banner mounts (offline transition), the screen reader announces both lines as a single utterance. When the banner unmounts (online recovery), the screen reader does NOT announce a removal — silent recovery is correct (no requirement for online-recovery announcement; OFX-04 is deferred).
- Do NOT use `role="alert"` (which implies `aria-live="assertive"`). An offline state is non-destructive; assertive interruption would be over-eager and contradicts the success criterion 5 spirit ("never represented as a blocking error").

### Animation, Transitions, Reduced Motion

- **No animation.** No fade-in, no slide-down, no transition on the banner appearance, dismissal, or button label flip. CONTEXT.md `<deferred>` explicitly defers animation.
- **`prefers-reduced-motion`:** moot — there is no animation to gate. The banner appears and disappears synchronously with React re-render. No `transition` properties on the banner element. No `transform` properties.
- Button label transitions are also instantaneous — the displayed string switches between online and offline labels without any cross-fade.

### Click & Keyboard Interaction

- The banner is non-interactive. Clicking it does nothing. Hovering does nothing. It has no `:hover` style.
- The relabeled buttons retain identical click handlers and keyboard activation (Space / Enter) as their online counterparts. THIS PHASE MUST NOT modify the button click handlers' bodies (CONTEXT.md `<domain>` "explicitly out of scope"). The handlers continue to POST to `/api/decide`. The clipboard write path is Phase 15.

### Annotation UI Coexistence

- Banner is shown above (in DOM order) the two-column review layout but does NOT prevent any interaction with that layout. Text selection, comment textarea, deny textarea, theme toggle, tab switching, sidebar card open/close, annotation creation — all remain fully functional while the banner is shown (success criterion 5).
- No backdrop, no overlay, no `pointer-events: none` on adjacent elements. The banner is purely visual.
- The two-column layout's vertical scroll behavior is unchanged. Scrolling the review content does NOT scroll the banner with it (banner is in the page-level flex column above the content scroll container; this is a property of the existing layout, not added in this phase).

### Online Recovery

- The banner unmounts via React conditional render the moment `useHeartbeat()` returns `'online'` after recovery. No transition, no toast, no announcement.
- Button labels revert to their online defaults (`approveLabel`, `denyLabel`, `Submit Denial`) on the same render.
- The deny form's open/closed state is preserved across the connectivity transition (the toggle state lives in `App.tsx` local state and is not coupled to connectivity).

## Testability Contract

CONTEXT.md `<decisions>` "Claude's Discretion" requires Phase 14 to include a test task per `CLAUDE.md` Test Coverage Requirements (modifying `App.tsx` with new business logic — label-switch + banner-render branches). The test approach below satisfies that requirement while honoring the project's no-`@testing-library/react` policy.

**Recommended test approach:** Extract the label-selection logic into a pure helper function and unit-test the helper. This aligns with Phase 13 RESEARCH.md A1 and the existing project pattern (`serializeAnnotations.ts` + `serializeAnnotations.test.ts`).

**Suggested helper signature** (planner has discretion to refine):

```ts
// ui/src/utils/offlineLabels.ts
import type { ConnectivityStatus } from '../hooks/useHeartbeat'

export const OFFLINE_APPROVE_LABEL = 'Copy to clipboard — approve'
export const OFFLINE_DENY_LABEL = 'Copy to clipboard — deny'
export const OFFLINE_SUBMIT_DENIAL_LABEL = 'Copy to clipboard'

export function approveButtonLabel(status: ConnectivityStatus, defaultLabel: string): string {
  return status === 'offline' ? OFFLINE_APPROVE_LABEL : defaultLabel
}

export function denyButtonLabel(status: ConnectivityStatus, defaultLabel: string): string {
  return status === 'offline' ? OFFLINE_DENY_LABEL : defaultLabel
}

export function submitDenialButtonLabel(status: ConnectivityStatus): string {
  return status === 'offline' ? OFFLINE_SUBMIT_DENIAL_LABEL : 'Submit Denial'
}
```

**Test file:** `ui/src/utils/offlineLabels.test.ts`. Cases:
1. `approveButtonLabel('online', 'Approve')` → `'Approve'`
2. `approveButtonLabel('online', 'No issues')` → `'No issues'` (custom label from Phase 11.1 preserved online)
3. `approveButtonLabel('offline', 'Approve')` → `'Copy to clipboard — approve'`
4. `approveButtonLabel('offline', 'No issues')` → `'Copy to clipboard — approve'` (offline string overrides custom labels)
5. Same matrix for `denyButtonLabel`
6. `submitDenialButtonLabel('online')` → `'Submit Denial'`
7. `submitDenialButtonLabel('offline')` → `'Copy to clipboard'`

The banner-render branch is too thin to require its own test (a single conditional in JSX); the constant-string assertions above plus a manual smoke against `useHeartbeat` (already tested in Phase 13) cover the contract.

**Banned test approaches:**

- `@testing-library/react` (`renderHook`, `render`, `screen.getByRole`) — not in `package.json`, would add a dependency, contradicts project convention.
- DOM mounting via `jsdom` to assert banner presence/absence — overkill; the conditional render is one line.

**Required test gates before commit:**

- `npm test` (vitest run) in `ui/` passes with the new tests.
- `npm run lint` (eslint) in `ui/` passes.
