## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| user textarea -> Annotation.comment (edit mode) | Free-form user input replaces existing annotation text |
| × button click -> annotation removal | Immediate destructive action with no confirmation (D-09) |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-21-10 | Repudiation | × button deletes annotation without confirmation | accept | Documented behavior per D-09 ("delete removes immediately with no confirmation"). UX intent is explicit; annotations live only in local React state until Phase 22 submit. |
| T-21-11 | Elevation of Privilege | Icon button clicks trigger CommentBubble.onClick (line 91) | mitigate | All new icon buttons MUST call `e.stopPropagation()` in their onClick handlers. Acceptance criterion asserts this is present. Without it, clicking pencil/× would also toggle `focusedCommentId` via the article's onClick. |
| T-21-12 | Denial of Service | position: sticky on editing bubble may fail in scroll container | mitigate | RESEARCH.md Pitfall 4 warns this. The shared scroller (`mainRef` in ReviewerV2Shell line 73-75) has `overflowY: 'auto'` — sticky positioning should work within that ancestor. Wave 3's human-verification step in Plan 04 will catch any actual failure; the fallback (computing `position: 'fixed'` from `aside.getBoundingClientRect()`) can be added then if needed. |
| T-21-13 | Tampering | textarea defaultValue divergence on rapid edit | accept | `defaultValue` is uncontrolled — React only seeds the initial value. If the user clicks pencil twice in quick succession (without canceling), Shell's `setEditingId` is a no-op the second time (already same id), so the textarea won't re-mount and the user's edits are preserved. |
| T-21-SC | Tampering | npm/cargo installs | accept | No new package installs in this plan. |
</threat_model>
