<key_implementation_notes>
1. **`async_main` URL hardcoding** — current code at `src/main.rs:567` is `let url = format!("http://127.0.0.1:{}/", port);`. Change this to accept a `path: &str` parameter (e.g., `"/"` for review flows, `"/code-review"` for code-review flow). All callers (`run_review_flow`, `run_hook_flow`, `run_opencode_flow`) pass `"/"`; the new `run_code_review_flow` passes `"/code-review"`.

2. **`code-review` empty plan content** — pass `String::new()` for `plan_md`. The `/code-review` SPA route never calls `/api/plan`. Verified safe in RESEARCH.md Pitfall 4.

3. **PreToolUse filter strategy (D-01)** — Filtering happens inside `run_pre_pr_hook_flow`, NOT via the `if` field in hooks.json (RESEARCH.md Recommendation, Open Question 1). The function reads stdin JSON, extracts `tool_input.command` from the existing `ToolInput.extra` map (or a new typed field), checks against the regex `^(gh\s+pr\s+create|git\s+push)\b`, and either exits 0 silently (with NO stdout) or invokes the code-review flow.

4. **Subcommand naming (D-02)** — Use `pre-pr-hook` (kebab-case, matches `review-hook`). Clap renames `Commands::PreP rHook` → `pre-p-r-hook` by default if named `PreP rHook`; use explicit `#[command(name = "pre-pr-hook")]` on the variant OR name the variant `PreP rHook` (clap default kebab-case of `PrePrHook` is `pre-pr-hook`). Verify with a parse test.

5. **stdout discipline** — `run_code_review_flow` writes exactly ONE JSON document to stdout via `serde_json::to_writer(std::io::stdout(), &output)` at the end. `run_pre_pr_hook_flow` writes ZERO bytes to stdout in the non-matching branch (early `return`); in the matching branch it delegates to the same single stdout write as `run_code_review_flow`. NO `println!` to stdout — eprintln for diagnostics only.

6. **Output JSON shape** — Per RESEARCH.md Open Question 2, the code-review subcommand emits `build_opencode_output(&decision)` (`{"behavior":"allow"}` / `{"behavior":"deny","message":"..."}`). This is identical to `run_review_flow`'s output and matches what the existing `Decision` deserializer in `plan_review.rs` produces. The richer `{message?,comments?}` payload is a pre-existing phase-28 gap — out of scope here.

7. **`extract_command_from_tool_input` helper** — Add a private helper that reads `tool_input.extra.get("command").and_then(Value::as_str)` and returns `Option<&str>`. This isolates the JSON shape from the filtering logic and is unit-testable.
</key_implementation_notes>
