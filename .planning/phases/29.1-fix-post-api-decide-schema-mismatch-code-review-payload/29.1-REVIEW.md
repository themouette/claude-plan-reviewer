## Summary

The two files implement the schema-mismatch fix for `POST /api/decide` in the
code-review flow: `src/plan_review.rs` makes `behavior` optional (defaulting to
`"allow"` via `unwrap_or`) so the code-review frontend's `{ message, comments }`
payload no longer returns 422, and `tests/integration/server_cycle.rs` adds the
integration test `server_cycle_code_review_submit` that covers the full binary
cycle.

The fix itself is structurally sound, but there are four correctness or quality
issues worth calling out.

## Critical Issues

### CR-01: `find_free_port` introduces a TOCTOU race that can cause spurious test failures

**File:** `tests/integration/server_cycle.rs:13-16`

**Issue:**
`find_free_port` binds a `TcpListener` to port 0, reads the OS-assigned port,
then *drops* the listener before the port number is passed to the subprocess.
Between the `drop` (when the listener releases the port) and the binary's own
`bind`, another process — or a parallel test run — can steal the port, causing
the `wait_for_port` assertion to fail non-deterministically.

The comment on line 11 acknowledges the window ("small TOCTOU window … acceptable
in tests"), but that assessment is wrong when tests are run in parallel
(`cargo test` defaults to multi-threaded). Five integration tests in this file
all call `find_free_port()` independently; they can legitimately collide with
each other. The failure mode is a flaky assertion:

```
assertion `left == right` failed: server did not start within 10 seconds on port 51234
```

This is a test-reliability blocker: it can mask real regressions by causing
random CI failures, and it can suppress failures by making a broken binary look
like a port collision.

**Fix:**
Pass the already-bound listener (or its fd) into `spawn_hook_flow` and have the
binary inherit the file descriptor, so the OS port is never released before
use. If that is not viable with the current binary API, at minimum pass `--port 0`
directly (OS-assigned binding inside the binary) and read the actual port back
from the binary's stderr line `"Review UI: http://127.0.0.1:<port>"` before
proceeding — that is the only race-free portable approach.

```rust
// Pattern: let the binary choose the port, parse it from stderr
// instead of pre-allocating and releasing.
fn spawn_hook_flow_zero_port(fixture_json: &str) -> (std::process::Child, u16) {
    // spawn with --port 0 (OS-assigned inside the binary)
    // read stderr until the "Review UI: http://127.0.0.1:<port>" line
    // parse port from that line
    // return (child, port)
}
```

## Warnings

### WR-01: `post_decide` silently treats any non-`"behavior"` payload as `"allow"`

**File:** `src/plan_review.rs:42-65`

**Issue:**
`post_decide` uses `unwrap_or("allow")` as the fallback when `behavior` is
absent from the request body (line 48-50). This is the intentional fix for the
code-review schema mismatch (`{ message, comments }` has no `behavior` key), but
the fallback is **unconditional**: any malformed or empty payload — including a
completely empty JSON object `{}`, or a payload where `behavior` is `null` — is
silently treated as `"allow"`.

A user who sends a typo (`{ "behvior": "deny" }`) will silently get an approval
instead of an error, with no feedback that the request was misread. This is an
accidental-approval vector that is hard to debug.

**Fix:**
The correct approach is a discriminated union in the deserialized type. Introduce
a `DecidePayload` enum that deserializes to either the plan-review shape
(`behavior` required) or the code-review shape (`message` + optional `comments`,
no `behavior`). Route on the active mode (stored in `AppState`) and reject
payloads that do not match the expected shape with `StatusCode::UNPROCESSABLE_ENTITY`.

Alternatively, if keeping the permissive `serde_json::Value` approach, at least
log a warning to stderr when `behavior` is absent and the route is in plan-review
mode.

```rust
// Minimal defensive patch inside post_decide:
let behavior = body
    .get("behavior")
    .and_then(|v| v.as_str())
    .unwrap_or_else(|| {
        // Only default to "allow" when this is code-review mode.
        // In plan-review mode, emit a warning.
        "allow"
    })
    .to_string();
```

### WR-02: `spawn_code_review_flow` leaks a `TempDir` on every test invocation

**File:** `tests/integration/server_cycle.rs:266-280`

**Issue:**
`spawn_code_review_flow` (line 267-280) calls `Box::leak(Box::new(home))` to
prevent the `TempDir` from being dropped before the child process exits. The
comment says "the OS cleans /tmp on reboot", but:

1. The same pattern in `spawn_hook_flow` (line 44-47) leaks one dir per test
   per invocation; `spawn_code_review_flow` adds another. On a typical test run
   (five tests × two spawners) that is multiple leaked temp directories per run.
2. The justification "OS cleans on reboot" is only valid for `/tmp`-backed
   tmpfs. On macOS, `tempfile::TempDir` uses a subdirectory under
   `$TMPDIR` (e.g. `/var/folders/…`), which is cleaned on logout, not reboot —
   but it accumulates across test runs within a session.
3. Most importantly, the leak prevents the test harness from ever cleaning up,
   which can fill `/tmp` on resource-constrained CI runners.

**Fix:**
Store the `TempDir` alongside the child and drop it explicitly after
`wait_with_output()` completes:

```rust
fn spawn_code_review_flow(port: u16) -> (std::process::Child, tempfile::TempDir) {
    let home = tempfile::TempDir::new().unwrap();
    let child = Command::new(binary_path())
        .env("HOME", home.path())
        // ...
        .spawn()
        .expect("failed to spawn plan-reviewer code-review");
    (child, home)  // caller holds TempDir until after wait_with_output
}
```

### WR-03: `server_cycle_code_review_submit` asserts on `output.stdout` without checking for stderr noise

**File:** `tests/integration/server_cycle.rs:307-316`

**Issue:**
The test asserts:

```rust
assert_eq!(
    json["behavior"].as_str(),
    Some("allow"),
    "code-review submit: expected behavior == \"allow\" in stdout"
);
```

`run_code_review_flow` writes `build_opencode_output(&decision)` to stdout.
However, `async_main` fires a 3-second watchdog (`tokio::spawn`) that calls
`std::process::exit(0)` after 3 seconds (see `main.rs:854-857`). If the
watchdog races with `wait_with_output` and stdout is not fully flushed before
`exit(0)` is called, `output.stdout` may be empty or truncated.

In practice `serde_json::to_writer` + `write_all` completes synchronously before
the `tokio::spawn` future starts, so the data is almost always flushed.
But the test relies on this ordering implicitly, with no assertion that stdout
is non-empty before `from_slice`. If `output.stdout` is empty,
`serde_json::from_slice` returns an error, the test panics with:
`stdout is not valid JSON`, which is misleading.

**Fix:**
Add an explicit assertion before the JSON parse to produce a clearer failure
message:

```rust
assert!(
    !output.stdout.is_empty(),
    "stdout was empty — binary may have exited before writing JSON"
);
let json: serde_json::Value =
    serde_json::from_slice(&output.stdout).expect("stdout is not valid JSON");
```

This pattern is already missing in all five `server_cycle_*` tests (not just the
new one), but it is most likely to matter for `server_cycle_code_review_submit`
because the code-review path has an additional early-exit risk via the watchdog
spawned immediately after the decision is consumed.

## Info

### IN-01: `Decision` struct in `plan_review.rs` is no longer the canonical shape for the code-review flow

**File:** `src/plan_review.rs:14-18`

**Issue:**
`Decision { behavior: String, message: Option<String> }` was designed for the
plan-review flow where `behavior` is always populated. After the phase-29.1 fix,
`post_decide` accepts a `serde_json::Value` and synthesizes a `Decision` with
`behavior` defaulted to `"allow"` when absent. This means `Decision` is now
used as an internal pass-through type whose `behavior` field can be populated
with a fabricated default rather than a real user choice.

The mismatch between the struct's apparent contract ("behavior is the user's
explicit choice") and its actual runtime semantics ("behavior may be an inferred
default") will cause confusion for future maintainers.

Consider splitting the type: keep `Decision` for plan-review (with `behavior`
required) and introduce a `CodeReviewSubmission { message: String, comments: Vec<...> }`
for the code-review flow, converting to `Decision` explicitly at the handler
boundary.
