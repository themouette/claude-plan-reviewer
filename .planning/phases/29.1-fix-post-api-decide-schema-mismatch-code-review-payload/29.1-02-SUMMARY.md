## Accomplishments

- Changed `post_decide` extractor from `Json<Decision>` to `Json<serde_json::Value>` — axum accepts any valid JSON object
- Added key-presence dispatch: if `behavior` key present use its value; if absent synthesize `"allow"` (code-review flow)
- Extracted optional `message` field from body in both paths
- `server_cycle_code_review_submit` transitions from RED (422) to GREEN (200)
- All 5 pre-existing `server_cycle` tests continue to pass (plan-review allow/deny flows unchanged)

## Files Created/Modified

- `src/plan_review.rs` - Modified `post_decide` function signature and body only; Decision struct, AppState, and router() unchanged (13 lines added, 2 lines removed)

## Decisions Made

- Used `Json<serde_json::Value>` (not a new enum variant or wrapper struct) — this is the minimal change to unblock the code-review submit path with no structural impact on the rest of the module
- `unwrap_or("allow")` for the absent `behavior` key — the code-review frontend never sends this key, so synthesizing `"allow"` is the correct semantic for the code-review accept path
- `Decision` struct left intact — it is still used by `AppState.decision_tx` channel and the callers that process the decision downstream; only the HTTP entry point changes
