# Phase 29.1: Fix POST /api/decide Schema Mismatch — Code-Review Payload - Research

**Researched:** 2026-05-26
**Domain:** Rust axum route handler + serde deserialization + frontend-backend contract
**Confidence:** HIGH

## Summary

Phase 28 and Phase 29 introduced an unresolved contract gap: the code-review frontend
(`AppToolbar.tsx`) POSTs `{"message?":"...", "comments?:[...]}` to `/api/decide`, but the
Rust `Decision` struct at `src/plan_review.rs` requires `{"behavior":"allow"|"deny",
"message?":"..."}`. The `behavior` field is a required non-optional `String` in serde —
axum returns `422 Unprocessable Entity` with message "Failed to deserialize the JSON body
into the target type: missing field `behavior`" when the code-review payload hits the
endpoint.

This was confirmed empirically: starting the binary with `plan-reviewer --no-browser
code-review` and POSTing `{"message":"looks good"}` returns `422`. POSTing
`{"behavior":"allow"}` returns `200`. The existing `server_cycle` integration tests all
pass because they use the old `{"behavior":"..."}` shape — they never test the code-review
payload.

The fix requires adding a second deserializable struct for the code-review decision
(`CodeReviewDecision`) and a second route handler in `plan_review.rs`, OR making the
`Decision` struct flexible enough to accept both shapes. The most maintainable approach is
a separate enum-based input type that models both possible payloads, converting them
internally to the existing `Decision` struct used by downstream callers in `main.rs`.

**Primary recommendation:** Add a `CodeReviewBody` struct (`message`, `comments`
optional fields) that serde-deserializes the code-review frontend's payload. Add a second
axum route handler `post_code_review_decide` in `plan_review.rs` that accepts this struct,
maps it to a `Decision` (behavior = "allow" always for code-review submissions — the agent
uses the payload content, not a deny/allow signal), and fires the existing
`decision_tx.send()` channel. The route is exposed at the same `/api/decide` path via the
same `plan_review::router`, meaning `server.rs` needs no changes.

## Architecture Patterns

### The Exact Mismatch (VERIFIED by reading source + empirical test)

**Frontend sends** (`ui/src/code-review/buildCodeReviewPayload.ts`, `buildReviewPayload()`):

```typescript
// Shape: { message?: string, comments?: CommentOutput[] }
// Example: {"message":"looks good","comments":[{"file":"src/foo.rs","line":42,"side":"additions","text":"add test"}]}
// Minimal: {}
// No "behavior" field ever emitted
```

**Server expects** (`src/plan_review.rs`, `Decision` struct):

```rust
#[derive(serde::Deserialize, Debug, Clone)]
pub struct Decision {
    pub behavior: String,        // REQUIRED — no default, no Option<>
    pub message: Option<String>,
}
```

**Result:** axum's `Json<Decision>` extractor returns:
```
HTTP 422 Unprocessable Entity
Failed to deserialize the JSON body into the target type: missing field `behavior` at line 1 column 24
```

The frontend's `handleSend` in `AppToolbar.tsx` checks `if (res.ok || res.status === 409)`
and falls through to the clipboard fallback when `/api/decide` returns 422. So the user
sees "Copied to clipboard" instead of "Review sent", and the binary never receives the
decision and times out after 540 seconds.

### Why the Mismatch Exists

Phase 28 Plan 03 (28-03-SUMMARY.md) explicitly changed the payload schema from the D-01
design (`{decision, global_instruction?, comments?}`) to `{message?, comments?}` based on
user feedback during browser verification:

> "The original plan specified separate Approve / Request Changes buttons with gate logic
> (D-06). During browser verification, the user redirected to a simpler single 'Send
> Review' button — the agent decides what to do from the payload."
> — 28-03-SUMMARY.md, "Design change during verification"

Phase 29 was then planned and executed without noticing that the backend `/api/decide`
endpoint still expected the old `{behavior, message?}` shape. The Phase 29 Research
(29-RESEARCH.md Assumption A3) flagged this as uncertain but did not resolve it:

> "A3: The code-review subcommand output format (what it writes to stdout) should mirror
> run_review_flow's build_opencode_output"

Phase 29 Plans were executed assuming the POST would work — no integration test exercised
the code-review payload shape against a live `/api/decide` endpoint.

### Fix Options

**Option A — Separate route handler (RECOMMENDED):** Add a `CodeReviewBody` struct with
`message` and optional `comments`. Add `post_code_review_decide` handler that deserializes
`CodeReviewBody` and synthesizes a `Decision { behavior: "allow", message }` before
sending to the oneshot channel. Register the handler at the same `/api/decide` path by
distinguishing it via a query parameter or by replacing the existing handler.

The simplest variant: **replace the existing `post_decide` handler with one that accepts
a `serde_json::Value`** and maps it to a `Decision` based on which keys are present.

**Option B — Two distinct endpoints:** Add `/api/code-review/decide` used only by the
code-review frontend. Requires frontend change.

**Option C — Make `behavior` optional with a serde default:** Add
`#[serde(default = "default_allow")]` to `behavior`. Simple. But this silently accepts any
JSON that doesn't include `behavior` as an "allow" — including malformed or incorrect
payloads from the existing reviewer-v2 frontend, which always sends `behavior`.

Option A is recommended because it is explicit about the shape contract for each flow
without changing existing behavior for the plan-review flow. The two flows have genuinely
different contracts and should be modelled distinctly.

### Recommended Fix Architecture (Option A detail)

The cleanest approach that minimizes changes to existing code:

**Step 1 — Add `CodeReviewDecision` struct in `plan_review.rs`:**

```rust
/// Payload sent by the code-review frontend (/code-review SPA).
/// Shape: { message?: string, comments?: [...] }
/// The behavior is implicitly "allow" (the user submitted a review —
/// the agent reads the payload to determine next steps, not a deny/allow signal).
#[derive(serde::Deserialize, Debug, Clone)]
pub struct CodeReviewDecision {
    pub message: Option<String>,
    #[serde(default)]
    pub comments: Vec<serde_json::Value>,   // pass-through; server does not inspect
}
```

**Step 2 — Add `post_code_review_decide` handler in `plan_review.rs`:**

```rust
async fn post_code_review_decide(
    State(state): State<Arc<AppState>>,
    Json(body): Json<CodeReviewDecision>,
) -> impl IntoResponse {
    // Synthesize a Decision from the code-review payload.
    // behavior is always "allow" — the frontend never sends a deny.
    // The payload is forwarded to the agent via the stdout write in run_code_review_flow.
    let decision = Decision {
        behavior: "allow".to_string(),
        message: body.message,
    };
    let tx = state.decision_tx.lock().unwrap().take();
    match tx {
        Some(tx) => {
            let _ = tx.send(decision);
            StatusCode::OK
        }
        None => StatusCode::CONFLICT,
    }
}
```

**Step 3 — Register the new handler in `router()`:**

Replace the existing single route with two routes on the same path using a content-based
approach, OR replace `/api/decide` entirely with the new handler that handles both shapes.

Actually, the cleanest minimal fix: **replace `post_decide` with a handler that accepts
`serde_json::Value` (untyped) and dispatches based on key presence.** This avoids two
competing routes on the same path:

```rust
async fn post_decide(
    State(state): State<Arc<AppState>>,
    Json(body): Json<serde_json::Value>,
) -> impl IntoResponse {
    // Determine payload shape:
    // Plan-review frontend sends: { behavior: "allow"|"deny", message?: "..." }
    // Code-review frontend sends: { message?: "...", comments?: [...] }
    let decision = if let Some(behavior) = body.get("behavior").and_then(|v| v.as_str()) {
        // Plan-review shape
        Decision {
            behavior: behavior.to_string(),
            message: body.get("message").and_then(|v| v.as_str()).map(str::to_string),
        }
    } else {
        // Code-review shape — behavior is implicitly "allow"
        Decision {
            behavior: "allow".to_string(),
            message: body.get("message").and_then(|v| v.as_str()).map(str::to_string),
        }
    };
    let tx = state.decision_tx.lock().unwrap().take();
    match tx {
        Some(tx) => {
            let _ = tx.send(decision);
            StatusCode::OK
        }
        None => StatusCode::CONFLICT,
    }
}
```

This approach:
- Preserves the existing `Decision` type unchanged
- Preserves the existing route path and method
- Handles both payload shapes in one handler
- No frontend changes required
- All existing `server_cycle` tests continue to pass (they post `{"behavior":"allow"}` which still dispatches correctly)
- Adds no new struct types visible outside `plan_review.rs`

**Step 4 — Add integration test to `server_cycle.rs`:**

Add `server_cycle_code_review_submit` test that starts the binary in `code-review` mode
(`--no-browser code-review`), waits for the port, POSTs `{"message":"looks good",
"comments":[]}`, and asserts the response is `200` and the binary exits successfully.

This test does NOT exist today. Its absence is why the regression shipped.

### Anti-Patterns to Avoid

- **Changing `Decision.behavior` to `Option<String>` with a default:** Would silently
  accept an empty `{}` body and treat it as "allow". The empty-body case should fail, not
  default to "allow". Use the explicit presence check instead.
- **Adding a second route `/api/code-review/decide`:** Requires a frontend change
  (`AppToolbar.tsx` hardcodes `/api/decide`). This is unnecessary churn.
- **Forwarding the full code-review payload (comments array) through the `Decision` struct
  to `build_opencode_output`:** The `comments` array is currently dropped. That is correct
  for Phase 29.1. Full payload forwarding is a future concern (Phase 29.1 scope is to
  unblock the POST — the agent feedback handling is separate).

## Don't Hand-Roll

| Problem | Don't Build | Use Instead | Why |
|---------|-------------|-------------|-----|
| Untyped JSON dispatch | Custom JSON parser | `serde_json::Value` + `.get().and_then()` | Already in scope; zero new deps |
| Shape discrimination | Content-type headers or separate routes | Key-presence check in one handler | Simple, no routing complexity |

## Common Pitfalls

### Pitfall 1: Existing Tests Pass But They Test the Wrong Shape
**What goes wrong:** The existing `server_cycle` tests POST `{"behavior":"allow"}` — they
never exercise the code-review payload. The bug ships and existing CI stays green.
**Why it happens:** The code-review payload was designed in Phase 28 but never has an
integration test that exercises it against a live server.
**How to avoid:** Add `server_cycle_code_review_submit` test that POSTs
`{"message":"looks good"}` (no `behavior`) and asserts `200`.
**Warning signs:** The only existing "decide" tests use `behavior` — if you see that, the
code-review shape is untested.

### Pitfall 2: Breaking the Plan-Review Flow
**What goes wrong:** If the fix unconditionally synthesizes `behavior: "allow"` for all
bodies (e.g., using `serde(default)`), then a reviewer-v2 POST with
`{"behavior":"deny","message":"Needs tests"}` would be treated as "allow" if the handler
reads `behavior` wrong.
**Why it happens:** Subtle logic error in the key-presence check.
**How to avoid:** The presence check MUST check for the `behavior` key explicitly.
`body.get("behavior").is_some()` should be the discriminator. When `behavior` is present,
use its value. When absent, default to "allow".
**Warning signs:** `server_cycle_deny_claude` test fails after the fix — that test sends
`{"behavior":"deny"}` and expects the binary to emit a deny output.

### Pitfall 3: `comments` Field Silently Dropped
**What goes wrong:** The code-review payload includes a `comments` array that is parsed
but not forwarded anywhere. The binary outputs `{"behavior":"allow"}` without the
comments.
**Why it happens:** The `Decision` struct has no `comments` field.
**How to avoid:** This is intentional for Phase 29.1 scope — the `code-review` subcommand
writes `build_opencode_output(&decision)` to stdout, which only uses `behavior` and
`message`. For Phase 29.1, the agent sees `{"behavior":"allow"}` plus its console output.
The full structured feedback path (forwarding `comments` to stdout) is deferred.
**Warning signs:** None — this is by design for Phase 29.1.

### Pitfall 4: `post_decide` Deserialization Fails for Empty Body
**What goes wrong:** POSTing `{}` (empty body, no `behavior`, no `message`) could be
treated as a code-review submission with `behavior: "allow"` and no message.
**Why it happens:** The key-presence check treats any body without `behavior` as
code-review.
**How to avoid:** This is actually acceptable — the user clicked "Send Review" with no
message and no comments. `{}` is a valid code-review submission (the popover requires
`commentsCount > 0 || message.trim().length > 0` client-side, but the server should not
enforce this). The server accepting `{}` as "allow" is correct.

```rust
// Source: Derived from server_cycle::server_cycle_approve_claude pattern
// NOTE: code-review subcommand reads no stdin — spawn without stdin write
fn spawn_code_review_flow(port: u16) -> std::process::Child {
    let home = Box::leak(Box::new(tempfile::TempDir::new().unwrap()));
    Command::new(binary_path())
        .env("HOME", home.path())
        .args(["--no-browser", "--port", &port.to_string(), "code-review"])
        .stdin(Stdio::null())   // code-review reads no stdin
        .stdout(Stdio::piped())
        .stderr(Stdio::piped())
        .spawn()
        .expect("failed to spawn plan-reviewer code-review")
}

#[test]
fn server_cycle_code_review_submit() {
    let port = find_free_port();
    let child = spawn_code_review_flow(port);

    assert!(
        wait_for_port(port, Duration::from_secs(10)),
        "server did not start on port {port}"
    );

    // POST code-review payload (no "behavior" field)
    let response = ureq::post(&format!("http://127.0.0.1:{port}/api/decide"))
        .send_json(serde_json::json!({ "message": "looks good", "comments": [] }))
        .expect("POST /api/decide failed");
    assert_eq!(response.status(), 200, "expected 200 OK — schema mismatch would return 422");

    let output = child.wait_with_output().expect("failed to wait for child");
    assert!(output.status.success(), "expected exit code 0");

    let json: serde_json::Value =
        serde_json::from_slice(&output.stdout).expect("stdout is not valid JSON");

    // code-review flow uses build_opencode_output — flat {behavior: "allow"}
    assert_eq!(
        json["behavior"].as_str(),
        Some("allow"),
        "code-review submit: expected behavior == \"allow\" in stdout"
    );
}
```
