<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Browser → embedded SPA | The `ReviewerV2` component now executes in the same browser context as `App.tsx` when `/v2` is visited; both run in the same origin. No cross-origin trust boundary. |
| Browser → `/api/ping` | The v2 `useHeartbeat` issues GET requests to the same endpoint v1 uses. No new endpoint, no new attack surface. |
| `main.tsx` routing decision → component tree | A malicious URL like `/v2<script>` cannot inject because `window.location.pathname` is the parsed URL pathname (already URI-decoded by the browser); the only test is `.startsWith('/v2')` which is a pure string operation. No HTML/DOM injection vector. |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-17-03-01 | Tampering | URL path injection in `main.tsx` routing branch | accept | `window.location.pathname.startsWith('/v2')` is a pure string operation on a browser-parsed URL; no string interpolation into HTML, no `eval`, no `innerHTML`. ASVS L1 V5.3.1 (output encoding) is moot — there is no output here, only a boolean. |
| T-17-03-02 | Information Disclosure | Both v1 and v2 mounted simultaneously across two tabs share the same backend session and could race on /api/decide | accept | Only one tab can submit because v1's POST /api/decide returns 409 on second submit (Phase 1 SC). The user-facing impact of opening v2 in a second tab during an active v1 review is at worst a confusing UX — no privilege escalation, no data corruption. v2 has no submit path in Phase 17 anyway. |
