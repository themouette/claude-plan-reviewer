# Phase 26.1: Code Review Report

**Reviewed:** 2026-05-25
**Depth:** standard
**Files Reviewed:** 7
**Status:** issues_found

## Summary

Reviewed the commit-navigation bug-fix phase covering `useDiff` hook logic, `CodeReviewApp` selector computation, `CommitDrawer`, and `DiffPane`. The core hook logic and SHA validation are sound. One critical data-flow bug exists: deselecting all commit checkboxes sends an empty `shas[]` array to `fetchFilteredBranchDiff`, which triggers a vacuously-true `allFailed` guard and surfaces an error panel to the user instead of an appropriate "no commits selected" empty state. Three warnings cover ARIA violations in `CommitDrawer` and dead state in `DiffPane`. Three info items flag test quality issues.

## Critical Issues

### CR-01: All-commits-unchecked produces an error state instead of an empty state

**File:** `ui/src/code-review/hooks/useDiff.ts:99` / `ui/src/code-review/CodeReviewApp.tsx:28-29`

**Issue:** When the user unchecks every commit in `CommitDrawer`, `checkedCommitShas` becomes `[]`. The selector condition in `CodeReviewApp` (line 28) fires for any `checkedCommitShas.length < commits.length`, including `0 < N`, and produces `{ mode: 'branch-union', shas: [] }`.

`fetchFilteredBranchDiff` then calls `Promise.allSettled([])` which resolves to `[]`. The `allFailed` guard at line 99 uses `[].every(...)` which returns `true` vacuously in JavaScript — so the function returns `{ files: [], error: 'all fetches failed' }`. `DiffPane` renders this as the generic error panel with "Could not load diff — server returned an error", which is wrong and confusing.

**Fix:** Add an early-return guard in `fetchFilteredBranchDiff` for the empty-array case, and/or add a guard in the `CodeReviewApp` selector:

Option A — guard in `fetchFilteredBranchDiff` (safest, fixes it at the source):
```typescript
export async function fetchFilteredBranchDiff(
  shas: string[],
  doFetch: DoFetch,
  contextLines?: number,
): Promise<FetchDiffResult> {
  if (shas.length === 0) {
    return { files: [], error: null }  // empty selection = no files, not an error
  }
  // ... rest of function unchanged
}
```

Option B — guard in `CodeReviewApp` selector (line 28-29):
```typescript
const selector: DiffFetchSelector =
  viewMode === 'commit' && activeCommitSha !== null
    ? { mode: 'commit', sha: activeCommitSha }
    : viewMode === 'branch' &&
        commits.length > 0 &&
        checkedCommitShas.length > 0 &&          // <-- add: must have at least one checked
        checkedCommitShas.length < commits.length
      ? { mode: 'branch-union', shas: checkedCommitShas }
      : { mode: 'branch' }
```

Option B alone is insufficient because it would fall through to `mode: 'branch'` (showing all commits' diff) when the user intends to see nothing. Option A is the correct semantic fix. Both together are ideal.

**Note:** This scenario is also untested — `useDiff.test.ts` has no test for `fetchFilteredBranchDiff` with an empty `shas` array.

## Warnings

### WR-01: Interactive `input[checkbox]` nested inside `role="button"` — ARIA violation

**File:** `ui/src/code-review/CommitDrawer.tsx:109-170`

**Issue:** Each commit row is an `<li role="button">` (line 111) that contains an `<input type="checkbox">` (line 130). The ARIA spec prohibits interactive descendants inside `role="button"`. Screen readers that implement this rule will either ignore the checkbox entirely or produce conflicting announcements. The `e.stopPropagation()` calls reduce click-event conflicts but do not repair the broken accessibility tree.

**Fix:** Remove `role="button"` from the `<li>`. Instead, give the `<li>` a semantic role that allows interactive children (e.g., `role="listitem"` — the default — is sufficient), and make the commit-click area a real `<button>` element that does not contain the checkbox:

```tsx
<li key={commit.sha} style={{ /* row styles */ }}>
  <input
    type="checkbox"
    aria-label={`Include commit ${commit.short_sha} in diff`}
    checked={isChecked}
    onChange={(e) => onCheckChange(commit.sha, e.target.checked)}
  />
  <button
    type="button"
    onClick={() => onCommitClick(commit.sha)}
    onKeyDown={(e) => { if (e.key === 'Enter') onCommitClick(commit.sha) }}
    style={{ /* existing row content styles */ }}
  >
    {/* SHA chip + message + author/date */}
  </button>
</li>
```

### WR-02: Checkboxes have no accessible name

**File:** `ui/src/code-review/CommitDrawer.tsx:129-137`

**Issue:** The `<input type="checkbox">` elements have no `aria-label`, no `aria-labelledby`, and no associated `<label>` element. Screen readers announce them as "checkbox" with no context about which commit they control.

**Fix:** Add an `aria-label` derived from the commit's short SHA and/or message:

```tsx
<input
  type="checkbox"
  aria-label={`Include ${commit.short_sha} in diff`}
  checked={isChecked}
  onChange={(e) => {
    e.stopPropagation()
    onCheckChange(commit.sha, e.target.checked)
  }}
  onClick={(e) => e.stopPropagation()}
/>
```

### WR-03: Dead `reloadFocused` state in `DiffPane`

**File:** `ui/src/code-review/DiffPane.tsx:73` and `82`

**Issue:** `reloadFocused` is declared as React state (line 73) and set by `onFocus`/`onBlur` handlers (lines 160-166), but it never drives any render output. Line 82 suppresses the lint warning with `void reloadFocused`, which confirms the value is not actually used. The focus ring is applied by directly mutating `e.currentTarget.style.outline` in event handlers — bypassing React state entirely. This pattern leaks a state slot with zero benefit and leaves the code open to the confusing impression that state controls the focus ring when it does not.

**Fix:** Remove the state declaration and the `void reloadFocused` suppression. The DOM-mutation focus ring already works without it:

```tsx
// Remove these two lines entirely:
// const [reloadFocused, setReloadFocused] = useState(false)
// void reloadFocused

// In onFocus/onBlur, also remove the setReloadFocused calls:
onFocus={(e) => {
  e.currentTarget.style.outline = '2px solid var(--color-focus)'
  e.currentTarget.style.outlineOffset = '2px'
}}
onBlur={(e) => {
  e.currentTarget.style.outline = 'none'
}}
```

### IN-02: `CommitDrawer.test.ts` and `DiffPane.test.ts` use source-text matching — not behavior tests

**File:** `ui/src/code-review/CommitDrawer.test.ts:7`, `ui/src/code-review/DiffPane.test.ts:7`

**Issue:** Both test files read the source file from disk as a raw string and assert on substring presence (e.g., `expect(source).toContain("role=\"navigation\"")`). This approach:
- Does not verify that the component renders correctly with given props.
- Does not catch regressions where a string is present but unreachable (behind a dead code path).
- Is brittle to formatting changes (though less so for semantic strings).
- Gives a false confidence that render states are exercised.

The `CommitDrawer` is imported at line 5 of each test file but never rendered. The `exports a function` test is the only one that exercises the runtime module.

**Fix:** Replace source-string assertions with `@testing-library/react` render tests that exercise the three distinct render states (loading, error, list) with real props. Given the project's stated preference for avoiding `@testing-library/react` (see `useDiff.ts` comment), a minimal `renderToStaticMarkup` check would at least confirm the HTML output, not just the source text.

### IN-03: No test for `fetchFilteredBranchDiff` with empty `shas` array

**File:** `ui/src/code-review/hooks/useDiff.test.ts`

**Issue:** The test suite covers all-success, partial-failure, all-failure, and context-URL cases for `fetchFilteredBranchDiff`. It does not cover the `shas = []` case, which is the root of CR-01. Had this test existed, CR-01 would have been caught before shipping.

**Fix:** Add a test:

```typescript
it("returns { files: [], error: null } when shas is empty (no commits selected)", async () => {
  let fetchCalled = false
  const doFetch = () => {
    fetchCalled = true
    return Promise.resolve(new Response(JSON.stringify([]), { status: 200 }))
  }
  const result = await fetchFilteredBranchDiff([], doFetch)
  expect(fetchCalled).toBe(false)
  expect(result.files).toHaveLength(0)
  expect(result.error).toBeNull()
})
```

This test will fail until CR-01 is fixed, at which point it documents the expected contract.
