## Summary

This phase adds the OutlinePane component, wires it into ReviewerV2Shell, extends ContentPane with a heading-walk effect, and introduces `markdownRenderer.ts` for GFM rendering. The core logic is largely sound. One critical issue was found: `marked.parse()` returns `string | Promise<string>` and the `as string` cast suppresses the type error rather than guaranteeing synchronous execution — if `marked` is ever configured with an async `walkTokens` this silently produces `[object Promise]` in the UI. Four warnings cover: empty-slug generation for all-special-character headings (produces `id=""`, breaking IntersectionObserver observation and `document.getElementById` lookups), non-unique React keys when multiple headings slug to the same empty string, inline style mutation bypassing React's rendering model, and error swallowing in the fetch pipeline. Two info items note the source-reading test pattern and the unconstrained `paddingLeft` depth calculation.

## Critical Issues

### CR-01: `marked.parse()` cast `as string` hides async-Promise risk

**File:** `ui/src/reviewer-v2/utils/markdownRenderer.ts:77`

**Issue:** `marked.parse(md)` is typed `string | Promise<string>` in the installed marked v18 type declarations (`marked.d.ts:704`). The implementation casts with `as string` to appease TypeScript, but this is a type assertion, not a runtime guarantee. The return type is synchronous today because neither `markedHighlight` nor `{ gfm: true }` installs an async `walkTokens`. However, if any future `marked.use()` call registers an async `walkTokens`, `marked.parse()` will silently return a `Promise` at runtime. `setPlanHtml` would then receive a `Promise` object, and React would render the string `"[object Promise]"` inside the plan pane with no error surfaced to the user or developer.

The cast also means the TypeScript compiler cannot flag call-sites if the function signature drifts.

**Fix:** Validate the return value at runtime and throw an explicit error if it is a Promise, so any future async regression surfaces immediately rather than silently corrupting the UI:

```typescript
export function renderMarkdown(md: string): string {
  // ... configuration block unchanged ...
  headingSlugCounts = new Map()
  const result = marked.parse(md)
  if (result instanceof Promise) {
    throw new Error(
      'renderMarkdown: marked.parse() returned a Promise — an async walkTokens was registered. ' +
      'Use the synchronous highlight path or await and change the return type.'
    )
  }
  return result
}
```

## Warnings

### WR-01: Headings with only special characters produce `id=""` breaking outline and observer

**File:** `ui/src/reviewer-v2/utils/markdownRenderer.ts:30-36`

**Issue:** `slugify()` strips all non-alphanumeric characters. A markdown heading whose text contains only special characters (e.g., `# ---`, `# !!!`, `# 💡`) produces `baseSlug = ""`. The heading renderer then sets `id=""` on the element. Downstream effects:

1. `document.getElementById("")` returns `null` in all browsers — the IntersectionObserver in `OutlinePane` skips the guard (`if (el) observer.observe(el)`) so this heading is never observed and never becomes the active item.
2. The click handler in `OutlinePane` silently no-ops (`?.scrollIntoView` on `null`).
3. Multiple such headings receive the same key (`""`, `"-2"`, `"-3"`) in the React `<ol>` — see WR-02.

This is a realistic edge case: plan markdown generated by Claude Code frequently uses decorative divider headings.

**Fix:** Fall back to a positional id when the slug is empty:

```typescript
heading({ tokens, depth }: Tokens.Heading) {
  const rawText = extractRawText(tokens)
  const baseSlug = slugify(rawText) || `section-${depth}`
  // rest unchanged
}
```

A more robust fix also passes a `headingIndex` counter so truly unique ids are generated: `section-${headingIndex}`.

### WR-02: Non-unique React `key` when multiple headings produce empty slugs

**File:** `ui/src/reviewer-v2/OutlinePane.tsx:51`

**Issue:** `<li key={section.id}>` uses `section.id` as the React list key. When two or more headings produce `id=""` (see WR-01), the duplicate counter logic in `markdownRenderer` produces `""`, `"-2"`, `"-3"` as ids. The id `""` appears once; the suffixed ids (`"-2"`) are distinct, so only the first empty slug is a duplicate key. React will warn in development and may mis-reconcile the list.

**Fix:** Use the index as a fallback key when `section.id` is empty, or ensure `markdownRenderer` never emits an empty id (WR-01 fix eliminates this entirely):

```tsx
{sections.map((section, i) => (
  <li
    key={section.id || `__section-${i}`}
    ref={section.id === activeId ? activeItemRef : undefined}
  >
```

### WR-03: Inline style mutation in `onMouseEnter`/`onMouseLeave` bypasses React rendering

**File:** `ui/src/reviewer-v2/OutlinePane.tsx:63-81`

**Issue:** The hover handlers directly mutate `e.currentTarget.style.background` and `e.currentTarget.style.color`. This bypasses React's virtual DOM and can produce state desynchronisation: if a re-render occurs between `onMouseEnter` and `onMouseLeave` (e.g., `sections` prop changes), React will overwrite the imperative style mutation from the `style=` prop, but only at the next commit — leaving a transient visual glitch. More critically, the `if (section.id !== activeId)` guard reads a stale closure value when the active id changes during hover.

**Fix:** Replace with CSS `:hover` pseudo-class via a stylesheet or CSS module, or use a `hoveredId` state variable and derive styles from it declaratively. The `onFocus`/`onBlur` handlers have the same imperative pattern — these should also be replaced with `:focus-visible` CSS.

### WR-04: `fetch` error silently swallowed — no diagnostic information retained

**File:** `ui/src/reviewer-v2/ContentPane.tsx:28`

**Issue:** `.catch(() => setStatus('error'))` discards the error object entirely. If the fetch fails due to a network error, a non-JSON response body, or a JSON parse error, there is no way to distinguish the failure mode from the error UI. The error message shown to the user ("Could not render plan — The markdown content is unavailable") is the same regardless of cause. In a local tool that is expected to always have the server running, silent error swallowing makes debugging significantly harder.

**Fix:** Log the error to console at minimum, and optionally store the message for display:

```typescript
.catch((err: unknown) => {
  console.error('[ContentPane] Failed to load plan:', err)
  setStatus('error')
})
```

## Info

### IN-01: Source-reading test pattern provides no runtime behavior coverage

**Files:** `ui/src/reviewer-v2/ContentPane.test.ts`, `ui/src/reviewer-v2/OutlinePane.test.ts`

**Issue:** Both test files use `readFileSync` to load the `.tsx` source file and then assert that specific string literals are present (e.g., `expect(source).toContain("fetch('/api/plan')")`). These tests verify that the source text was written a certain way, not that the component behaves correctly at runtime. A refactoring that preserves semantics but changes syntax (e.g., template literals, variable extraction, renamed identifiers) would break the tests without breaking the component. Conversely, a logic bug that does not change the textual structure would pass the tests.

Per CLAUDE.md Test Coverage Requirements, TypeScript modules with business logic require tests that would fail if the logic were removed.

**Fix:** Replace source-reading assertions with rendered component tests using `@testing-library/react`. For `OutlinePane`, assert that clicking a list item calls `scrollIntoView`; for `ContentPane`, mock `fetch` and assert that `renderMarkdown` output appears in the DOM. The `markdownRenderer.test.ts` file in the same PR is a good model — it tests actual return values.

### IN-02: `paddingLeft` depth calculation has no upper bound

**File:** `ui/src/reviewer-v2/OutlinePane.tsx:95`

**Issue:** `paddingLeft: 16 + (section.depth - 1) * 8` produces `16px` for h1 through `88px` for h6. This is within the 200px outline panel width (leaving 112px for text at h6). However, `Section.depth` is typed as `number` and populated with `parseInt(el.tagName[1], 10)`. If a heading element with a non-standard tag (e.g., depth 0 from a custom renderer) were ever present, `paddingLeft` would be `8px` (negative offset from 16 - 8). HTML headings are capped at h6 so in practice this cannot occur, but the type does not enforce it.

**Fix:** Clamp the depth value at the computation site:

```typescript
paddingLeft: 16 + (Math.max(1, Math.min(6, section.depth)) - 1) * 8,
```

Or add `depth: 1 | 2 | 3 | 4 | 5 | 6` to the `Section` type.
