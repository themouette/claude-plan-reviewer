# Phase 06.1: Code Review Report

**Reviewed:** 2026-04-10T00:00:00Z
**Depth:** standard
**Files Reviewed:** 8
**Status:** issues_found

## Summary

Reviewed the integration test harness files plus the two production source files they exercise (`src/main.rs`, `src/server.rs`). The tests are well-structured and cover the critical approval/denial round-trips for both Claude and Gemini formats. The install/uninstall tests correctly isolate filesystem side-effects via a temporary `HOME`. Several real bugs and reliability risks were found:

- A TOCTOU race in port selection will cause flaky tests under parallel execution.
- The spawned child process and its `TempDir` both leak unconditionally, even on test failure — this is a correctness issue in the test helper, not just a style matter.
- The Gemini fixture uses `"BeforeTool"` as `hook_event_name`, but the production Gemini format detection (`is_gemini()`) keyed on that field means the fixture correctly identifies as Gemini. However the Gemini deny test does not assert the absence of `hookSpecificOutput`, leaving a potential regression undetected.
- `server.rs` drops the `CancellationToken` immediately after spawn — the graceful shutdown future is cancelled before it can ever fire, so the server never shuts down cleanly.

## Warnings

### WR-01: TOCTOU race in `find_free_port` causes intermittent test failures

**File:** `tests/integration/server_cycle.rs:13-16`
**Issue:** `find_free_port` binds a `TcpListener` to obtain a free port, then immediately drops it and passes the port number to the binary. Between the `drop` and the binary calling `bind`, another process (or another parallel test) can steal that port. This is a real race condition, not just theoretical — `cargo test` runs integration tests in parallel threads by default, and all four `server_cycle_*` tests call `find_free_port` independently.
**Fix:** Pass `--port 0` to the binary and have it print the assigned port to stderr. Parse stderr for the `Review UI: http://127.0.0.1:<port>` line before polling. Alternatively, use `--test-threads=1` as a stopgap, but that slows the suite. The production binary already supports `--port 0`, so the preferred fix is:

```rust
// In spawn_hook_flow, always use port 0 and extract assigned port from stderr
fn spawn_and_get_port(fixture_json: &str) -> (std::process::Child, u16) {
    // ... spawn with --port 0 ...
    // Read stderr line-by-line until "Review UI: http://127.0.0.1:" is seen
    // Parse the port from that line
}
```

### WR-02: `TempDir` is unconditionally leaked in `spawn_hook_flow`

**File:** `tests/integration/server_cycle.rs:44-47`
**Issue:** `Box::leak(Box::new(home))` permanently leaks the `TempDir` allocation and prevents `Drop` from ever running, even when the test panics or the child exits early. This means temporary directories accumulate in `/tmp` across the test suite lifetime. The comment says "the OS cleans /tmp on reboot" — acceptable for CI, but on a long-running developer machine this silently grows unbounded. More importantly, leaking unconditionally means a failed spawn (line 49–56) still leaks the allocation with no cleanup path.

**Fix:** Keep `TempDir` alive for the duration of the test by storing it alongside the child handle, rather than leaking it. Return a tuple:

```rust
fn spawn_hook_flow(fixture_json: &str, port: u16) -> (std::process::Child, tempfile::TempDir) {
    let home = tempfile::TempDir::new().unwrap();
    let mut child = Command::new(binary_path())
        .env("HOME", home.path())
        // ...
        .spawn()
        .expect("failed to spawn plan-reviewer");
    // write stdin, close it...
    (child, home) // caller drops home after wait_with_output()
}
```

### WR-03: `CancellationToken` dropped immediately — server graceful shutdown is broken

**File:** `src/server.rs:117`
**Issue:** `drop(token)` is called at line 117 immediately after spawning the server task. The spawned task holds `token_clone` and waits on `token_clone.cancelled()`. Because `token` is the only other handle, dropping it cancels the token instantly. In practice this means `token_clone.cancelled()` resolves as soon as the event loop first polls it — the graceful shutdown fires at server startup, not when a decision is posted. The server may or may not continue serving requests depending on how `axum::serve`'s internal state machine handles a concurrently-resolved shutdown future. Under load or slow event loops this is a subtle race.

**Fix:** Keep `token` alive for as long as the server should run. Return it from `start_server` alongside the `decision_rx`, or hold it in `AppState` and cancel it from inside `post_decide`:

```rust
// Option A: return token to caller
pub async fn start_server(
    plan_md: String,
    diff_content: String,
    port: u16,
) -> Result<(u16, oneshot::Receiver<Decision>, CancellationToken), ...> {
    // ... (don't drop token)
    Ok((port, decision_rx, token))
}

// Option B: cancel inside post_decide when decision is sent
// Store token in AppState and call token.cancel() after tx.send(body)
```

### WR-04: Gemini deny test does not assert absence of `hookSpecificOutput`

**File:** `tests/integration/server_cycle.rs:177-215`
**Issue:** `server_cycle_deny_gemini` checks that `json["decision"] == "deny"` and `json["reason"] == "Needs tests"`, but does not assert that `json.get("hookSpecificOutput").is_none()`. The approve variant (`server_cycle_approve_gemini`, line 171-174) does make this assertion. If a future regression causes the Gemini deny path to emit Claude-format output, the deny test would not catch it — only the approve test would.
**Fix:** Add the missing assertion:

```rust
assert!(
    json.get("hookSpecificOutput").is_none(),
    "Gemini deny: must NOT have hookSpecificOutput key"
);
```

### IN-01: Fixture `hook_input_gemini.json` uses inline `plan` field, not `plan_path`

**File:** `tests/fixtures/hook_input_gemini.json:7-9`
**Issue:** The Gemini fixture exercises the `plan` (inline text) code path, not the `plan_path` (filesystem path) code path described in `extract_plan_content` comments. According to the code comments, Gemini CLI sends a filesystem path via `tool_input.plan_path`. If the fixture does not match the real Gemini wire format, the integration test gives false confidence. The `plan_path` code path (including the `std::process::exit(1)` on read error) has no integration test coverage at all.
**Fix:** Either add a second Gemini fixture that uses `plan_path` pointing to a real file, or add an explanatory comment in the fixture noting that inline `plan` is accepted by both integrations and this is intentional. If the real Gemini CLI always sends `plan_path`, the fixture should be updated to match.

### IN-02: `wait_for_port` busy-polls with `std::thread::sleep` — no timeout feedback on failure

**File:** `tests/integration/server_cycle.rs:20-29`
**Issue:** When the binary fails to start (e.g., build artifact missing, early exit on bad stdin), `wait_for_port` spins for the full 10-second timeout before failing. The assert message only says the server "did not start" — no stderr from the child is surfaced. This makes test failure diagnostics slow and unhelpful.
**Fix:** After `wait_for_port` returns `false`, collect child stderr before panicking:

```rust
if !wait_for_port(port, Duration::from_secs(10)) {
    // Kill child to unblock wait_with_output
    let _ = child.kill();
    let output = child.wait_with_output().unwrap_or_default();
    panic!(
        "server did not start within 10 seconds on port {port}\nstderr: {}",
        String::from_utf8_lossy(&output.stderr)
    );
}
```

### IN-03: `comrak` version `0.52` is a pre-release / unusual semver in `Cargo.toml`

**File:** `Cargo.toml:19`
**Issue:** `comrak = { version = "0.52", ... }` — comrak uses a `0.x` versioning scheme where each minor increment is a breaking change. Version `0.52` is unusually high; the project's own CLAUDE.md mentions "check latest on crates.io" without pinning. If `0.52` is not yet published (as of April 2026 the latest known release was `0.31.x`), Cargo will silently resolve to the highest available `0.x` below `1.0`, which may not be `0.52`. This could cause a surprising version mismatch between what the developer intended and what Cargo resolves.
**Fix:** Verify `comrak 0.52` is actually published (`cargo update --dry-run`). If the intent is "latest stable", use `"0"` as the version string and let Cargo pick. If a specific minimum API is required, pin the exact version.
