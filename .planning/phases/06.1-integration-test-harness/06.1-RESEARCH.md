# Phase 6.1: Integration Test Harness - Research

**Researched:** 2026-04-10
**Domain:** Rust CLI integration testing — assert_cmd, subprocess HTTP testing, HOME isolation
**Confidence:** HIGH

## Summary

The binary already has `--no-browser` wired in `Cli` and `run_hook_flow` (gated with `if !no_browser`). The only production code change needed for TEST-01 and TEST-03 is adding a `--port` flag so tests can pass a fixed port instead of relying on OS assignment. The server currently binds to `127.0.0.1:0` and returns the OS-assigned port through its return value; the test-facing change is surfacing that port number so a subprocess test can discover it without parsing stderr lines.

`assert_cmd 2.x` is the canonical Rust library for CLI integration tests. It uses the `CARGO_BIN_EXE_<name>` env var set by Cargo to locate the built binary — no pre-build step needed when running under `cargo test`. Each integration test file in `tests/` is a separate crate compiled only during `cargo test`. For the server cycle test (TEST-03) the right shape is: spawn the binary as a `std::process::Child` (not `assert_cmd::Command` which blocks until exit), poll TCP readiness, POST to the server with `ureq` (sync, tokio-free), then kill the child and assert on its captured stdout.

**Primary recommendation:** Add `--port <u16>` as an optional flag to `Cli`; when set, pass it to `server::start_server` (or bind directly to it). The binary already prints `"Review UI: http://127.0.0.1:{port}"` to stderr — tests can parse that line OR use a fixed port. Fixed port is simpler. Structure tests as `tests/integration.rs` (single file) or `tests/integration/` (module dir with `mod.rs`) depending on number of test functions; start with a single file.

### Pattern 1: Hook Flow Test (TEST-01) — assert_cmd with write_stdin + HOME isolation

`assert_cmd::Command` blocks until the subprocess exits. For hook flow this is correct — the binary reads stdin, starts the server, waits for the decision channel, then exits. With `--no-browser` the server still starts but the test needs to POST to it before it times out (540 s). For a pure stdin→stdout test where we only want to verify the JSON output format, inject a plan directly and POST via a separate thread.

**Simpler alternative for TEST-01 pure stdout test:** Use a forked approach — spawn `std::process::Child`, POST the decision, then call `child.wait_with_output()`.

```rust
// Source: assert_cmd docs + cargo book (CARGO_BIN_EXE_* pattern)
use assert_cmd::prelude::*;
use std::process::Command;
use predicates::prelude::*;

#[test]
fn hook_install_no_system_files() {
    // HOME points to a clean tmpdir — real ~/.claude is never touched
    let home = tempfile::TempDir::new().unwrap();

    Command::cargo_bin("plan-reviewer").unwrap()
        .env("HOME", home.path())
        .args(["install", "claude"])
        .assert()
        .success()
        .stdout(predicate::str::contains("ExitPlanMode hook installed"));

    // Verify the file was written inside the isolated HOME
    let settings = home.path().join(".claude/settings.json");
    assert!(settings.exists(), "settings.json should be written in tmpdir HOME");
}
```

### Pattern 3: Server Approve/Deny Cycle (TEST-03) — background Child + ureq POST

`assert_cmd::Command` does not expose async spawn — it blocks on `.assert()`. For the server cycle test, use `std::process::Command` (std library) to spawn the process non-blocking, then poll for port readiness, POST, and assert on the child's output after `wait_with_output()`.

**Critical design decision: how does the test know which port to connect to?**

Two options:
1. **Fixed `--port` flag** — test passes `--port 0` and the binary prints the actual port to stderr; test parses stderr. Fragile.
2. **Fixed `--port <N>` flag** — test passes a known port (e.g., `19876`). Binary binds to that exact port. Test connects to that port. Simple, but port collisions possible in CI with parallel tests.
3. **Best: `--port 0` + parse stderr** — binary prints `"Review UI: http://127.0.0.1:<port>"` to stderr already; test reads stderr line before posting. Requires reading from a pipe before the process exits.

**Recommended:** Fixed `--port <N>` per test (use different values per test function), or find a free port in the test and pass it explicitly. This avoids the pipe-reading complexity.

```rust
// Source: pattern derived from async-book testing chapter + ureq docs
use std::process::{Command, Stdio};
use std::time::Duration;
use std::net::TcpStream;

const HOOK_JSON_CLAUDE: &str = include_str!("fixtures/hook_input_claude.json");

fn wait_for_port(port: u16, timeout: Duration) -> bool {
    let deadline = std::time::Instant::now() + timeout;
    while std::time::Instant::now() < deadline {
        if TcpStream::connect(format!("127.0.0.1:{}", port)).is_ok() {
            return true;
        }
        std::thread::sleep(Duration::from_millis(50));
    }
    false
}

#[test]
fn server_cycle_approve() {
    let port: u16 = 19876; // unique per test; or find free port dynamically
    let home = tempfile::TempDir::new().unwrap();

    let mut child = Command::new(assert_cmd::cargo::cargo_bin("plan-reviewer"))
        .env("HOME", home.path())
        .args(["--no-browser", "--port", &port.to_string()])
        .stdin(Stdio::piped())
        .stdout(Stdio::piped())
        .stderr(Stdio::piped())
        .spawn()
        .expect("failed to spawn plan-reviewer");

    // Write hook JSON to stdin
    use std::io::Write;
    child.stdin.as_mut().unwrap().write_all(HOOK_JSON_CLAUDE.as_bytes()).unwrap();
    drop(child.stdin.take()); // close stdin so the binary proceeds

    // Wait for server to be ready (up to 5 seconds)
    assert!(wait_for_port(port, Duration::from_secs(5)), "server did not start");

    // POST the approve decision
    let resp = ureq::post(&format!("http://127.0.0.1:{}/api/decide", port))
        .send_json(serde_json::json!({"behavior": "allow"}))
        .expect("POST failed");
    assert_eq!(resp.status(), 200);

    // Wait for child to exit and collect output
    let output = child.wait_with_output().expect("wait failed");
    assert!(output.status.success(), "binary did not exit 0");

    let stdout: serde_json::Value = serde_json::from_slice(&output.stdout).unwrap();
    assert_eq!(
        stdout["hookSpecificOutput"]["decision"]["behavior"].as_str(),
        Some("allow")
    );
}
```

### Anti-Patterns to Avoid

- **Don't use `assert_cmd::Command::assert()` for the server cycle test** — it blocks until the process exits; the binary will not exit until the decision is posted, creating a deadlock.
- **Don't parse stderr in a separate thread to discover the port** — reading from a `Stdio::piped()` stderr while waiting for the process to read stdin is a deadlock risk. Use a fixed `--port` instead.
- **Don't use a single fixed port across tests** — if tests run in parallel (Cargo default), two tests binding the same port will race. Use `#[serial]` attribute or unique ports per test.
- **Don't share `TempDir` across tests** — `TempDir` drops at end of scope; keeping it in a `lazy_static` causes cross-test contamination.
- **Don't skip `drop(child.stdin.take())`** — if stdin is not closed, the binary's `read_to_string(stdin)` will block waiting for EOF.

## Don't Hand-Roll

| Problem | Don't Build | Use Instead | Why |
|---------|-------------|-------------|-----|
| Binary location | `std::path::PathBuf::from("target/debug/plan-reviewer")` | `assert_cmd::cargo::cargo_bin("plan-reviewer")` | Cargo sets `CARGO_BIN_EXE_plan_reviewer`; hardcoded paths break cross-compilation and `--release` builds |
| Stdout assertions | String equality on `output.stdout` | `predicates::str::contains` via assert_cmd | Better error messages; `normalize()` handles line endings |
| Port availability probe | `nc -z 127.0.0.1 PORT` subprocess | `TcpStream::connect` retry loop | No external process dependency; works on all platforms |
| JSON fixture | Inline string in test | `include_str!("fixtures/hook_input_claude.json")` | Fixtures are visible/editable without recompiling; Cargo includes them at compile time |

**Key insight:** `cargo test` already builds the binary as a pre-requisite for integration tests (via `CARGO_BIN_EXE_*`). Never invoke `cargo build` from within a test.

## Common Pitfalls

### Pitfall 1: Deadlock from assert_cmd blocking + stdin not closed
**What goes wrong:** Test spawns via `assert_cmd::Command`, calls `write_stdin()` and then `.assert()`. The binary's `read_to_string(stdin)` waits for EOF. `assert_cmd` only closes stdin after writing, so this works for `write_stdin` — BUT if you then need to POST HTTP before the binary exits, `.assert()` will block indefinitely.
**Why it happens:** `assert_cmd::Command` is a synchronous wrapper that runs the entire subprocess to completion before returning. There is no `spawn_background()` in assert_cmd.
**How to avoid:** Use `std::process::Command` with `Stdio::piped()` for the server cycle test. Use `assert_cmd` only for tests where the binary exits without waiting for an HTTP decision (hook stdout tests with mock decisions not tested, or install/uninstall which exit immediately).
**Warning signs:** Test hangs indefinitely without any output.

### Pitfall 2: Port collision in parallel tests
**What goes wrong:** Two test functions both bind to port `19876`. One fails with "Address already in use".
**Why it happens:** `cargo test` runs tests in parallel by default (multiple threads in the same process or OS-level).
**How to avoid:** Use distinct ports per test function. Or find a free port dynamically: bind a `TcpListener` to port 0, get the assigned port, drop the listener, pass the port to the child. (There is a small TOCTOU race but acceptable for test use.)
**Warning signs:** Flaky test failures with `EADDRINUSE`.

### Pitfall 3: TempDir dropped too early
**What goes wrong:** `let home = tempfile::TempDir::new().unwrap();` is created, then `home.path()` is passed to `.env()`. If `home` is dropped (moved or explicit drop) before the subprocess exits, the directory is deleted while the process is still running.
**Why it happens:** Rust's RAII drop semantics; `TempDir` deletes the directory on drop.
**How to avoid:** Ensure `home` is held in the same scope as `child.wait_with_output()`. Do not call `drop(home)` prematurely.
**Warning signs:** File-not-found errors inside the subprocess when it tries to write settings.json.

### Pitfall 4: stdin pipe not closed before waiting
**What goes wrong:** `child.wait_with_output()` blocks forever because the binary is blocked in `read_to_string(stdin)`.
**Why it happens:** `std::process::Child` with `Stdio::piped()` keeps the stdin pipe open until explicitly closed.
**How to avoid:** Always `drop(child.stdin.take())` after writing fixture bytes.
**Warning signs:** Test hangs indefinitely.

### Pitfall 5: macOS XDG / HOME path shadowing
**What goes wrong:** On macOS, some tools resolve `~` using the password database (`getpwuid`) rather than `$HOME`. Claude integration reads `$HOME` via `std::env::var("HOME")` [VERIFIED: src/install.rs line 24], so our tests work correctly. Gemini integration (once implemented in Phase 6) will also need to use `$HOME`.
**Why it happens:** macOS apps sometimes bypass `$HOME` env var.
**How to avoid:** Verify each new integration uses `std::env::var("HOME")` not `dirs::home_dir()` (which uses `getpwuid` as fallback on macOS). This is a concern for Phase 6/7 implementations, not Phase 6.1.
**Warning signs:** Settings written to real `~/.gemini/` instead of tmpdir.

### Pitfall 6: The 540-second server timeout
**What goes wrong:** The server has a 540-second timeout [VERIFIED: src/main.rs line 440]. If a test never POSTs to `/api/decide`, the test will run for 9 minutes before the binary exits with a deny decision.
**Why it happens:** The binary is designed for human-interactive use.
**How to avoid:** Always POST a decision in server cycle tests. For tests that only check hook stdout format, the server cycle path is never reached (test pipes stdin and the binary exits after writing stdout — except it doesn't: the hook flow always starts the server). Consider a `--timeout-secs` flag for test use, or simply always include the POST step.
**Warning signs:** Test runs for >10 seconds and doesn't print anything.
