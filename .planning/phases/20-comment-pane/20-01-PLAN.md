## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Plan markdown character offsets → `Annotation.anchorStart`/`anchorEnd` | Offset integers are captured from `getOffsets()` and stored in client-side React state; they are not user-typed strings |
| CSS Highlights API `::highlight(comment-hover)` rule | Browser-rendered style applied to ranges constructed in JS from stored offsets |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-20-01 | Tampering | `anchorStart`/`anchorEnd` integer fields on `Annotation` | accept | The fields are produced by `getOffsets()` which walks live DOM text nodes (untyped by user); even an out-of-range offset only causes `rangeFromOffsets` to return null and the bubble to skip rendering. No XSS or injection surface exists — these are numbers, not strings |
| T-20-02 | Information Disclosure | `computeCommentLayout` output | accept | The function operates on already-visible annotation data (anchorY positions of comments the user can already see in the sidebar). No new information is leaked |
| T-20-03 | Denial of Service | `computeCommentLayout` runtime on large `items[]` | accept | A1 from RESEARCH.md: annotation count is bounded by human review capacity (tens, not thousands); the O(n log n) sort + O(n) sweep runs in well under one millisecond at expected sizes |
| T-20-SC | Tampering | npm/pip/cargo installs | accept | No new packages installed in this plan; `sidenotes@2.0.1` was evaluated and REJECTED in 20-RESEARCH.md (architectural incompatibility, not a security finding). All code changes are additive to existing files |
</threat_model>
