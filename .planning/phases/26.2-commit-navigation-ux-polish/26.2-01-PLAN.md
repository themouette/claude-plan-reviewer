<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| filesystem→Rust process | git2 reads refs from the .git directory; ref names come from the local repo (not network input) |
| Rust→JSON wire format | New fields are serialized to the HTTP response consumed by the React frontend |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-26.2-01 | Information Disclosure | Branch/tag name leak in API response | accept | Branch and tag names from the user's own local repo are returned to the user's own browser. No multi-tenant model, no remote attacker — the data already crosses no untrusted boundary. The wire format is consumed by JSON.parse in the frontend; no string interpolation. |
| T-26.2-02 | Tampering | Malicious ref name causing JSON injection | mitigate | serde_json escapes ref names automatically when serializing Vec<String>. Frontend reads via res.json() (structured parse), not innerHTML — no XSS surface even if a ref name contained `<script>`. |
| T-26.2-03 | Denial of Service | Repo with thousands of refs slowing /api/commits | accept | COMMIT_LIMIT = 500 commits already caps the outer loop. Inner ref iteration is O(refs) per commit, bounded by the user's own repo size. For Phase 26.2 this is acceptable; revisit only if profiling shows >500ms on a real repo. |
| T-26.2-SC | Tampering | npm/cargo installs | mitigate | No new dependencies — only existing git2, serde, axum already audited in prior phases. |
</threat_model>
