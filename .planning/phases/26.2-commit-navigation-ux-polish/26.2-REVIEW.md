# Phase 26.2: Code Review Report

**Reviewed:** 2026-05-25T00:00:00Z
**Depth:** standard
**Files Reviewed:** 11

## Summary

This phase delivers: per-file collapse/expand (D-07), a global Expand Files / Collapse Files toolbar button (D-08), a global stats strip in the diff pane (D-09), branch/tag pills in the commit drawer (D-11), and the backing Rust API endpoints for commit listing and per-commit diffs.

Two critical bugs were found. First, `contextExpanded` state is never reset when the diff selector changes (commit switch, branch-union, back to branch), so the Expand All button shows "Collapse" and the spinner can linger in the wrong state. Second, the `onKeyDown` handler in `CommitDrawer` passes a `KeyboardEvent` typed as `React.MouseEvent` through an unsafe `as unknown as` cast, causing `handleCommitClick`'s `event.metaKey`, `event.ctrlKey`, and `event.shiftKey` to silently read the correct underlying values only because `KeyboardEvent` also carries those modifiers — but `event.shiftKey` triggers range-select in `handleCommitClick`, so Enter-key on any row performs a range-select rather than a plain single-select, which is wrong behaviour.

Several warnings follow: stale closure over `selector` in `useDiff`'s `dispatch` function, `handleToggleAllFiles` collapsing to the current `files` snapshot (stale when the diff is being refetched), the `Fragment` key including `index` alongside `filename` (unnecessary and can cause remount flicker on rename), and the `focusedButton` state in `AppToolbar` being immediately suppressed via `void` with no functional use.

## Critical Issues

### CR-01: `contextExpanded` not reset on selector change — spinner / label stuck

**File:** `ui/src/code-review/CodeReviewApp.tsx:24-31`

**Issue:** `contextExpanded` is never cleared when `selectedCommitShas` changes and a new selector is computed. When the user clicks a commit (selector becomes `{ mode: 'commit', sha }`) the `useDiff` hook re-fetches immediately via the `selectorKey` `useEffect` — but `contextExpanded` remains `true` from the previous "Expand All" press. This causes:
1. The AppToolbar button to show "Collapse" instead of "Expand All" for the new commit.
2. `contextLoading={loading && contextExpanded}` to flash the "Loading…" spinner on every commit switch while `contextExpanded` is `true`, even though the user never asked for expanded context on this commit.
3. `handleReload` to pass `refetch(999)` when the user clicks Reload while on a commit view they never expanded.

The selector changes on lines 24-30 but there is no `useEffect` or derived reset for `contextExpanded`.

**Fix:**
```tsx
// Add a useEffect that clears contextExpanded whenever the selector changes.
// Place this near the existing selector derivation, before useDiff.
useEffect(() => {
  setContextExpanded(false)
}, [selectorKey]) // selectorKey can be passed from useDiff, or recomputed here
```

Because `selectorKey` is computed inside `useDiff`, the cleanest approach is to expose it from the hook or re-derive the same key in `CodeReviewApp`:

```tsx
const selectorKey =
  selector.mode === 'commit'
    ? `commit:${selector.sha}`
    : selector.mode === 'branch-union'
      ? `branch-union:${[...selectedCommitShas].sort().join(',')}`
      : 'branch:'

useEffect(() => {
  setContextExpanded(false)
}, [selectorKey])
```

### CR-02: `Enter` key on commit row triggers range-select instead of plain single-select

**File:** `ui/src/code-review/CommitDrawer.tsx:109`

**Issue:** The `onKeyDown` handler casts a `React.KeyboardEvent` to `React.MouseEvent` via `as unknown as React.MouseEvent` and passes it directly to `onCommitClick`. Inside `handleCommitClick` (CodeReviewApp.tsx:54), the first branch checks `event.shiftKey && selectedCommitShas.length > 0`. When the user presses Enter on a row while another commit is already selected, `event.shiftKey` is `false` for a plain Enter key press, so the immediate result is "plain click" — **however**, if the user presses `Shift+Enter` on a row, `event.shiftKey` is `true` (because the `KeyboardEvent` carries `shiftKey`) and the range-select path fires. This is an intentional half-fix that does not fully isolate keyboard from mouse — the `metaKey` and `ctrlKey` paths also fire on `Meta+Enter` / `Ctrl+Enter`, meaning multi-select can be triggered from the keyboard against the intent.

More critically, passing a `KeyboardEvent` masquerading as `MouseEvent` is an unsafe type lie: `event.clientX`, `event.clientY`, `event.button`, etc. are all `0` or `undefined` on a keyboard event. Any future caller of `handleCommitClick` that reads those MouseEvent fields will silently get garbage.

The correct fix is a dedicated keyboard handler path that always performs plain single-select:

**Fix:**
```tsx
// CommitDrawer.tsx — replace the onKeyDown handler
onKeyDown={(e) => {
  if (e.key === 'Enter') {
    e.preventDefault()
    // Keyboard activation always means plain single-select; construct a
    // minimal synthetic object rather than casting a KeyboardEvent.
    onCommitClick(commit.sha, {
      shiftKey: false,
      metaKey: false,
      ctrlKey: false,
    } as React.MouseEvent)
  }
}}
```

## Warnings

### WR-01: Stale closure over `selector` in `useDiff` `dispatch`

**File:** `ui/src/code-review/hooks/useDiff.ts:145-155`

**Issue:** `dispatch` is a plain `async function` declared inside the `useDiff` render body. It closes over `selector`, which is the current render's value. However, `dispatch` is also called inside the `refetch` callback (line 163), and `refetch` is memoised with `useCallback([selectorKey])`. When `selectorKey` changes, `useCallback` rebuilds `refetch` with the new `dispatch` closure — this is correct.

The subtler problem is that `dispatch` is **not** memoised and is re-created on every render, while `refetch` holds the `dispatch` from the render in which `refetch` was last rebuilt. If `selectorKey` is stable but `selector` object identity changes (which it does on every render of `CodeReviewApp` because `{ mode: 'branch' }` is constructed inline), `dispatch` inside the stable `refetch` closure will reference the **stale** `selector` from the render where `refetch` was memoised, not the latest render.

In practice the object contents are the same (`mode: 'branch'` never changes when `selectorKey === 'branch:'`) so there is no observable bug today. But the pattern will silently break if `selector` fields are ever mutated or if a more complex selector is added. The fix is to capture `selector` inside `refetch` via a `useRef`:

**Fix:**
```tsx
const selectorRef = useRef(selector)
selectorRef.current = selector

const refetch = useCallback((contextLines?: number) => {
  const sel = selectorRef.current
  async function go() { /* dispatch using sel */ }
  setLoading(true)
  void go().then(...)
}, [selectorKey])
```

### WR-02: `handleToggleAllFiles` collapses to a stale `files` snapshot

**File:** `ui/src/code-review/CodeReviewApp.tsx:112-120`

**Issue:** `handleToggleAllFiles` reads `files` directly from the render closure:

```tsx
function handleToggleAllFiles() {
  if (collapsedFiles.size === 0) {
    setCollapsedFiles(new Set(files.map((f) => f.filename)))
  } else {
    setCollapsedFiles(new Set())
  }
}
```

`files` is the value at the time `handleToggleAllFiles` was created (each render recreates it, so it is usually current). The real problem is that `allFilesExpanded` is derived as `collapsedFiles.size === 0`, not as `collapsedFiles.size === 0 || files.length === 0`. When `files` is an empty array (during loading, or when branch has no changes), `collapsedFiles.size` is also `0`, so `allFilesExpanded` is `true` and the toolbar shows "Collapse Files" even though there is nothing to collapse. Clicking it sets `collapsedFiles` to `new Set([])` (empty set from an empty map) — this is a no-op — but the label flickers incorrectly.

**Fix:**
```tsx
const allFilesExpanded = files.length === 0 || collapsedFiles.size === 0
// or more precisely:
const allFilesExpanded = files.length > 0 && collapsedFiles.size === 0
```

Use the second form and ensure the button is disabled when `files.length === 0`:

```tsx
// In AppToolbar: disable the toggle-all-files button when there are no files
// (pass a `filesCount` prop, or derive it)
```

### WR-03: `Fragment` key includes array `index` — unnecessary remount on rename/reorder

**File:** `ui/src/code-review/DiffPane.tsx:245`

**Issue:** The `Fragment` key is constructed as `` `${file.filename}-${index}` ``. Appending `-${index}` is pointless when `file.filename` is already unique (filenames in a diff are unique per entry), and it is harmful when the diff result is re-fetched after a commit switch: if the file list reorders or a rename occurs, React will see a different key for the same file and will unmount/remount the `FileDiffRenderer`, discarding any in-progress expansion state and causing a visible flash.

The anchor `div` already uses `id={`file-${index}`}` (line 247) as a scroll target, but that ID is independent of the Fragment key.

**Fix:**
```tsx
<Fragment key={file.filename}>
```

If two entries could theoretically have the same filename (e.g., a rename producing both old and new name), use `${file.status}-${file.filename}`.

### WR-04: `focusedButton` state in `AppToolbar` is dead code

**File:** `ui/src/code-review/AppToolbar.tsx:27-44`

**Issue:** `focusedButton` is set by all `makeFocusHandlers` calls but is never read to affect rendering — the focus ring is applied imperatively via `e.currentTarget.style.outline`, not reactively via `focusedButton`. The `void focusedButton` on line 44 is an explicit suppression of the `noUnusedLocals` TypeScript error. This means every button focus/blur event triggers a state update (`setFocusedButton`) that causes a full component re-render of `AppToolbar` with no visible effect.

This is a real performance degradation in addition to being dead code: the re-render runs on every focus and blur event across all 5+ toolbar buttons.

**Fix:** Remove the `focusedButton` state entirely. The imperative `e.currentTarget.style.outline` approach already works without state. If focus tracking for styling purposes is ever needed, use CSS `:focus-visible` instead.

```tsx
// Remove these lines:
// const [focusedButton, setFocusedButton] = useState<string | null>(null)
// void focusedButton

// Simplify makeFocusHandlers to not call setFocusedButton:
function makeFocusHandlers(id: string) {
  void id  // kept for potential future labelling
  return {
    onFocus: (e: React.FocusEvent<HTMLButtonElement>) => {
      e.currentTarget.style.outline = '2px solid var(--color-focus)'
      e.currentTarget.style.outlineOffset = '2px'
    },
    onBlur: (e: React.FocusEvent<HTMLButtonElement>) => {
      e.currentTarget.style.outline = 'none'
    },
  }
}
```

### WR-05: `commit_to_dto` iterates all repository references for every commit — O(refs × commits)

**File:** `src/diff_api.rs:115-128`

**Issue:** `commit_to_dto` opens `repo.references()` and iterates the entire ref list to find which refs point at the commit being mapped. This is called once per commit in the revwalk loop (line 312). For a repository with `N` commits and `R` refs, `try_list_commits` runs O(N × R) ref iterations. In a repository with 500 commits (the limit) and 100 branches, that is 50,000 ref iterations for a single `/api/commits` request.

The function is also called during testing in a tiny repo so the tests pass, but the production use case (a live feature branch on a large monorepo) will be noticeably slow.

The correct approach is to build a ref → OID map once before the revwalk, then look up each OID in O(1):

**Fix:**
```rust
// Build once: OID → Vec<(name, kind)>
let mut oid_to_refs: std::collections::HashMap<git2::Oid, (Vec<String>, Vec<String>)> =
    std::collections::HashMap::new();
if let Ok(refs) = repo.references() {
    for r in refs.flatten() {
        if let Ok(peeled) = r.peel_to_commit() {
            let entry = oid_to_refs.entry(peeled.id()).or_default();
            if let Some(name) = r.shorthand() {
                if r.is_branch() { entry.0.push(name.to_string()); }
                else if r.is_tag() { entry.1.push(name.to_string()); }
            }
        }
    }
}

// Then in the revwalk, map each commit using oid_to_refs lookup instead.
```

### IN-02: `allSelected` computed from length equality — fragile for multi-select

**File:** `ui/src/code-review/CodeReviewApp.tsx:126`

**Issue:** `allSelected` is computed as:

```tsx
const allSelected = commits.length > 0 && selectedCommitShas.length === commits.length
```

This checks that the _count_ of selected SHAs equals the total commit count but does not verify that the selected SHAs are a subset of the actual `commits` array. If `selectedCommitShas` somehow contains stale SHAs from a previous fetch (e.g., after a refetch that returns a different commit set), the condition is true when lengths match even though the SHAs are wrong. The diff selector for the `branch` endpoint would then not be used.

This is a low-probability edge case given the current code, but the count equality check is a fragile proxy for "all commits are selected."

**Fix:**
```tsx
const allSelectedShaSet = new Set(selectedCommitShas)
const allSelected =
  commits.length > 0 &&
  selectedCommitShas.length === commits.length &&
  commits.every((c) => allSelectedShaSet.has(c.sha))
```
