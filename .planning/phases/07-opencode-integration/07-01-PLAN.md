**Step A: Create the JS plugin file at `src/integrations/opencode_plugin.mjs`.**

This file is the opencode plugin that registers a `submit_plan` tool. It will be embedded in the Rust binary at compile time via `include_str!("opencode_plugin.mjs")` in opencode.rs.

The opencode plugin API shape is confirmed in FEATURES.md lines 139-156: the plugin export is a function that returns `{tool: {...}}` (singular `tool`, not `tools`). The executor should use the pattern below -- no live research needed.

The plugin file MUST:
1. Export a default function that receives opencode's plugin context
2. Register a `submit_plan` tool with a `plan` string parameter (description: "The complete plan text to submit for human review")
3. On `submit_plan` invocation: write the plan text to a temp file, spawn the plan-reviewer binary as a subprocess with `--plan-file <tempfile>` argument, capture stdout, parse the JSON decision, clean up the temp file, return the decision to opencode
4. Use `child_process.execFileSync` (NOT `spawn` with piped stdin -- unreliable per Pitfall from PITFALLS.md issue #21293)
5. The binary path is injected at install time as a string constant at the top of the file: `const PLAN_REVIEWER_BIN = "__PLAN_REVIEWER_BIN__";` -- the Rust install code replaces this placeholder with the actual absolute path
6. Handle subprocess errors gracefully: if spawn fails, return an error message to opencode rather than crashing
7. Write the temp file to `os.tmpdir()` with a unique filename using `crypto.randomUUID()` or `Date.now()`
8. The plan content in the tool result should indicate allow/deny and include any reviewer message

The plugin structure should follow this pattern:
```javascript
// plan-reviewer-opencode.mjs
// Installed by plan-reviewer. Binary path injected at install time.
import { execFileSync } from "node:child_process";
import { writeFileSync, unlinkSync } from "node:fs";
import { tmpdir } from "node:os";
import { join } from "node:path";
import { randomUUID } from "node:crypto";

const PLAN_REVIEWER_BIN = "__PLAN_REVIEWER_BIN__";

export default (ctx) => {
  return {
    tool: {
      submit_plan: {
        description: "Submit a plan for human review before implementation. The reviewer opens a browser UI where the user can approve, deny, or annotate the plan.",
        parameters: {
          type: "object",
          properties: {
            plan: {
              type: "string",
              description: "The complete plan markdown text to submit for review"
            }
          },
          required: ["plan"]
        },
        execute: async ({ plan }) => {
          const tmpFile = join(tmpdir(), `plan-reviewer-${randomUUID()}.md`);
          try {
            writeFileSync(tmpFile, plan, "utf-8");
            const stdout = execFileSync(PLAN_REVIEWER_BIN, ["--plan-file", tmpFile], {
              encoding: "utf-8",
              timeout: 600000,
              cwd: process.cwd(),
            });
            const result = JSON.parse(stdout.trim());
            if (result.behavior === "allow") {
              return "Plan APPROVED by reviewer.";
            } else {
              return `Plan DENIED by reviewer. Feedback:\n${result.message || "No message provided."}`;
            }
          } catch (err) {
            return `Plan review failed: ${err.message}`;
          } finally {
            try { unlinkSync(tmpFile); } catch {}
          }
        }
      }
    }
  };
};
```

**Step B: Replace the stub in `src/integrations/opencode.rs` with the full implementation.**

Follow the gemini.rs pattern exactly. The implementation needs:

NOTE: The execFileSync+tempfile approach used by the JS plugin is architecturally valid and avoids GitHub issue #21293. That issue is specifically about `child_process.spawn()` with piped stdin delivering data to the subprocess unreliably. Our approach: (a) uses execFileSync (not spawn with piped stdio), (b) passes plan content via a temp file instead of stdin, and (c) `run_opencode_flow()` in the binary (implemented in Plan 07-02) NEVER reads stdin -- it reads the plan from the file path passed via `--plan-file`. No stdin piping occurs, so issue #21293 does not apply. The HTTP-fetch model referenced in FEATURES.md research is for persistent server scenarios -- our single-review invocation model is simpler and equally reliable.

**Private helper functions:**

1. `opencode_config_path(home: &str) -> PathBuf` -- returns `PathBuf::from(home).join(".config/opencode/opencode.json")`

2. `opencode_plugin_path(home: &str) -> PathBuf` -- returns `PathBuf::from(home).join(".config/opencode/plugins/plan-reviewer-opencode.mjs")`

3. `opencode_is_installed(config: &serde_json::Value, plugin_path_str: &str) -> bool` -- checks if `config["plugin"]` is an array containing `plugin_path_str`. The idempotency key is the absolute path string in the plugin array.

4. `OPENCODE_PLUGIN_SOURCE: &str = include_str!("opencode_plugin.mjs");` -- compile-time embedded plugin source.

**`install()` implementation:**

1. Require `ctx.binary_path` -- return Err if None (same as gemini.rs line 18-21)
2. Compute `plugin_path = opencode_plugin_path(&ctx.home)` and `config_path = opencode_config_path(&ctx.home)`
3. Create plugin directory `~/.config/opencode/plugins/` if it doesn't exist (`fs::create_dir_all`)
4. Write the plugin file: take `OPENCODE_PLUGIN_SOURCE` and replace the placeholder `__PLAN_REVIEWER_BIN__` with the actual `binary_path` value, then write to `plugin_path`
5. Read existing opencode.json or start with `{}` -- same error handling pattern as gemini.rs (refuse to overwrite if parse fails, return Err with "refusing to overwrite" message)
6. Ensure root is an object
7. Ensure `root["plugin"]` exists as an array (use `entry().or_insert_with(|| json!([]))`)
8. Compute `plugin_path_str = plugin_path.to_string_lossy().to_string()`
9. Idempotency check: if `opencode_is_installed(&root, &plugin_path_str)` -> print "already configured" and return Ok
10. Push `plugin_path_str` as a JSON string into the `plugin` array
11. Pretty-print and write config back to `config_path`
12. Create `~/.config/opencode/` directory if it doesn't exist
13. Print success messages: "plugin file written to {plugin_path}" and "plugin entry added to {config_path}"

**`uninstall()` implementation:**

1. Compute `plugin_path` and `config_path`
2. Remove plugin file if it exists (`fs::remove_file`), print message. If not exists, print "no plugin file found" and continue (do NOT return early -- still need to clean config)
3. If config file exists: read and parse JSON (return Ok on invalid JSON with warning, same as gemini.rs). Remove the plugin path string from `config["plugin"]` array using `retain()`. Write back.
4. If config file doesn't exist: print "no config file found" and return Ok
5. Print success message

**`is_installed()` implementation:**

1. Check BOTH conditions: plugin file exists on disk AND config entry is present
2. If plugin file doesn't exist -> return false
3. Read and parse config. If missing or invalid -> return false
4. Compute `plugin_path_str` and call `opencode_is_installed(&config, &plugin_path_str)`

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Rust binary -> filesystem | Install writes plugin file and modifies opencode.json |
| JS plugin -> Rust binary | Plugin spawns binary as subprocess with --plan-file argument |
| opencode.json parse | Untrusted JSON from user's existing config |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-07-01 | Tampering | opencode.json | mitigate | Refuse to overwrite if JSON parse fails (return Err with "refusing to overwrite" message); never silently replace corrupted config |
| T-07-02 | Tampering | plugin file | accept | Install always overwrites plugin file with current embedded source (idempotent behavior); user modifications are replaced on reinstall -- this is intentional and documented in install output |
| T-07-03 | Information Disclosure | __PLAN_REVIEWER_BIN__ | accept | Binary path is written into plugin file on disk; this is the user's own binary path in their home directory -- no sensitive data |
| T-07-04 | Denial of Service | plugin subprocess | mitigate | JS plugin sets execFileSync timeout to 600000ms (10 min); prevents infinite hang if binary never responds |
| T-07-05 | Elevation of Privilege | plugin file path | mitigate | Plugin destination path is computed from HOME env var + hardcoded relative path; no user-supplied path fragments accepted (same model as claude.rs T-04-02) |
</threat_model>
