<objective>
Add a `--plan-file <path>` CLI argument to `plan-reviewer` so the opencode JS plugin can invoke the binary without using stdin. When `--plan-file` is provided, the binary reads the plan from the specified file, runs the standard browser review flow, and outputs a flat JSON decision (`{"behavior":"allow"}` or `{"behavior":"deny","message":"..."}`) to stdout.

Purpose: Completes the runtime side of the opencode integration. The JS plugin (from Plan 07-01) invokes `plan-reviewer --plan-file /tmp/plan.md` as a subprocess. This plan wires that invocation mode into the binary.

Output: Working end-to-end opencode flow: JS plugin writes plan to temp file, invokes binary with --plan-file, binary opens browser, user decides, binary outputs JSON, plugin reads it.

ARCHITECTURE NOTE: This plan uses execFileSync+tempfile (not HTTP-fetch-to-server). The plan-reviewer binary enters `run_opencode_flow()` when `--plan-file` is provided. This function NEVER reads stdin. Therefore, execFileSync's default `stdio:'pipe'` for stdin is harmless -- the binary's stdin pipe is simply never used. GitHub issue #21293 (unreliable stdin delivery) does not apply because we don't pipe plan content through stdin at all. The HTTP-fetch pattern from FEATURES.md research applies to persistent-server scenarios; our single-invocation model is simpler and equivalent.

**New function `run_opencode_flow(no_browser: bool, plan_file: &str)`:**

This function handles the opencode invocation mode. It does NOT read stdin. It:

1. Reads plan content from the file at `plan_file` path:
   ```rust
   let plan_md = match std::fs::read_to_string(plan_file) {
       Ok(content) => content,
       Err(e) => {
           eprintln!("Failed to read plan file at {}: {}", plan_file, e);
           std::process::exit(1);
       }
   };
   eprintln!("Plan loaded from file ({} bytes): {}", plan_md.len(), plan_file);
   ```

2. Extracts diff from current working directory (not from a hook-provided cwd, since there is no HookInput):
   ```rust
   let cwd = std::env::current_dir()
       .map(|p| p.to_string_lossy().into_owned())
       .unwrap_or_default();
   let diff_content = extract_diff(&cwd);
   eprintln!("Diff extracted ({} bytes)", diff_content.len());
   ```

3. Starts the tokio runtime and runs `async_main` (same as existing hook flow):
   ```rust
   let rt = tokio::runtime::Builder::new_current_thread()
       .enable_all()
       .build()
       .unwrap();

   let decision = rt.block_on(async_main(no_browser, plan_md, diff_content));
   ```

4. Builds the opencode output format -- a flat JSON object:
   ```rust
   let output = build_opencode_output(&decision);
   serde_json::to_writer(std::io::stdout(), &output).expect("failed to write hook output");
   ```

**New function `build_opencode_output(decision: &Decision) -> serde_json::Value`:**

The opencode plugin expects stdout to be parseable JSON with `behavior` and optional `message` fields:

```rust
/// Build the opencode response JSON from a browser decision.
///
/// Format: {"behavior":"allow"} or {"behavior":"deny","message":"..."}
/// This is the format the JS plugin parses in opencode_plugin.mjs.
fn build_opencode_output(decision: &Decision) -> serde_json::Value {
    match decision.behavior.as_str() {
        "allow" => serde_json::json!({ "behavior": "allow" }),
        _ => {
            let mut obj = serde_json::json!({
                "behavior": "deny"
            });
            if let Some(ref msg) = decision.message {
                obj["message"] = serde_json::Value::String(msg.clone());
            }
            obj
        }
    }
}
```

**CRITICAL: Do NOT modify `run_hook_flow()` or `async_main()`** -- the existing Claude Code and Gemini flows must remain exactly as they are. The opencode flow is a completely separate code path that shares only `async_main`, `extract_diff`, and `build.rs`-built assets.

**CRITICAL: Do NOT read stdin in `run_opencode_flow()`** -- the JS plugin passes content via the file, not stdin. Reading stdin would cause the binary to hang because the plugin uses `execFileSync` which does not pipe stdin.

## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| --plan-file path -> fs::read_to_string | Binary reads arbitrary file path provided by JS plugin |
| stdout | Must contain exactly one JSON line -- no pollution |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-07-06 | Information Disclosure | --plan-file path | accept | The --plan-file argument is provided by the JS plugin which writes a temp file. The binary reads this file with read_to_string -- read-only, never executed, never shell-passed. An attacker controlling the --plan-file arg could read any file the user can read, but this is a local-only tool where the attacker would need local access to the user's shell. The path is not persisted or logged to stdout. Risk: LOW, accepted. |
| T-07-07 | Tampering | stdout pollution | mitigate | The opencode flow has exactly one `serde_json::to_writer(stdout(), ...)` call, same discipline as existing Claude/Gemini flows. All diagnostics use `eprintln!`. Validated by unit tests checking output format. |
| T-07-08 | Denial of Service | --plan-file with large file | accept | fs::read_to_string loads entire file into memory. A multi-GB file could OOM, but this is user-invoked on their own machine with their own files. No mitigation needed for local-only tool. |
</threat_model>
